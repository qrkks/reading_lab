# 接口相当于是方法的类型

可以这么理解，但更准确一点：

> **接口是“方法集合的类型”，也就是一种行为类型。**

例如：

```go
type Speaker interface {
	Speak() string
}
```

它规定：

> 任何拥有 `Speak() string` 方法的类型，都可以当作 `Speaker` 使用。

```go
type Duck string

func (d Duck) Speak() string {
	return "呱呱"
}

type Cat string

func (c Cat) Speak() string {
	return "喵喵"
}
```

于是：

```go
var s Speaker

s = Duck("Donald")
fmt.Println(s.Speak())

s = Cat("Tom")
fmt.Println(s.Speak())
```

`Duck` 和 `Cat` 并没有声明“我要实现 Speaker”。Go 会自动检查：只要方法签名匹配，就实现了接口。

接口也可以包含多个方法：

```go
type Animal interface {
	Speak() string
	Eat()
}
```

某个类型必须同时拥有这两个方法，才能成为 `Animal`。

需要与“函数类型”区分：

```go
type Handler func(string) error
```

这是一个具体函数签名的类型。

而：

```go
type Handler interface {
	Handle(string) error
}
```

这是要求对象具有 `Handle` 方法的接口类型。

所以可以记成：

```text
函数类型：规定一个函数长什么样
接口类型：规定一个对象必须有哪些方法
```

接口不关心结构体有哪些字段，也不关心具体类型是什么，只关心：

> 你有没有我要求的这些能力。

# 接口怎么定义？

Go 接口这样定义：

```go
type 接口名 interface {
	方法名(参数) 返回值
}
```

例如：

```go
type Speaker interface {
	Speak() string
}
```

它表示：任何拥有下面这个方法的类型，都实现了 `Speaker`：

```go
Speak() string
```

例如：

```go
type Duck string

func (d Duck) Speak() string {
	return "呱呱"
}
```

因为 `Duck` 有 `Speak() string`，所以它自动实现了 `Speaker`，不需要写 `implements`：

```go
var s Speaker = Duck("Donald")

fmt.Println(s.Speak())
```

接口可以要求多个方法：

```go
type Animal interface {
	Speak() string
	Eat(food string)
}
```

类型必须同时拥有这两个方法，才实现 `Animal`：

```go
type Duck string

func (d Duck) Speak() string {
	return "呱呱"
}

func (d Duck) Eat(food string) {
	fmt.Println(d, "吃了", food)
}
```

接口里的方法只写**签名**，不写函数体，而且不写 `func`：

```go
type Animal interface {
	Speak() string // 正确
}
```

不是：

```go
type Animal interface {
	func Speak() string // 错误
}
```

接口也能嵌入其他接口：

```go
type Speaker interface {
	Speak() string
}

type Eater interface {
	Eat(food string)
}

type Animal interface {
	Speaker
	Eater
}
```

这等价于：

```go
type Animal interface {
	Speak() string
	Eat(food string)
}
```

最核心的理解是：

> 接口定义“需要具备哪些方法”，具体类型只要方法签名完全匹配，就自动实现接口。

# 说白了，接口就是定义的有这个方法的所有的类型的实例呗？哪怕你一个是 Struct，一个是具名类型，只要你有这个方法，都属于这个接口

对，说白了就是：

> **接口表示“所有具备这些方法的类型”的共同类型。**

不管底层是什么：

```go
type Chicken struct {
	Name string
}

type Duck string

type Robot int
```

只要它们都有同样的方法：

```go
func (c Chicken) Speak() string {
	return "咯咯哒"
}

func (d Duck) Speak() string {
	return "呱呱"
}

func (r Robot) Speak() string {
	return "机器人说话"
}
```

它们就都实现了：

```go
type Speaker interface {
	Speak() string
}
```

于是可以统一放进接口变量或切片：

```go
speakers := []Speaker{
	Chicken{Name: "小黄"},
	Duck("Donald"),
	Robot(1),
}

for _, s := range speakers {
	fmt.Println(s.Speak())
}
```

接口不关心它们：

- 是不是结构体
- 底层是不是字符串
- 有没有相同字段
- 属不属于同一个“继承体系”

接口只检查：

```text
有没有这些方法？
方法签名是否完全一致？
```

不过有一个重要细节：要看方法是挂在值上还是指针上。

```go
func (c *Chicken) Speak() string {
	return "咯咯哒"
}
```

这时实现 `Speaker` 的是：

```go
*Chicken
```

不是：

```go
Chicken
```

所以：

```go
var s Speaker = &Chicken{} // 正确
var s Speaker = Chicken{}  // 错误
```

因此更准确的一句话是：

> 接口定义了一个方法集合；任何类型，只要它的方法集包含这些方法，就实现了这个接口。

# 接口是定义了一个类型集合和它的行为

对，这个理解已经很准确了：

> **接口用一组行为要求，定义了一个类型集合。**

例如：

```go
type Speaker interface {
	Speak() string
}
```

它写出来的是行为要求：

```go
Speak() string
```

由此形成的类型集合是：

```text
所有拥有 Speak() string 方法的类型
```

例如：

```go
type Chicken struct{}
type Duck string
type Robot int

func (Chicken) Speak() string { return "咯咯哒" }
func (Duck) Speak() string    { return "呱呱" }
func (Robot) Speak() string   { return "你好" }
```

那么：

```text
Chicken
Duck
Robot
```

都属于 `Speaker` 所表示的类型集合：

```go
var s Speaker

s = Chicken{}
s = Duck("Donald")
s = Robot(1)
```

所以关系可以概括成：

```text
接口中的方法集合
        ↓
规定必须具备的行为
        ↓
筛选出满足条件的所有类型
```

不过接口通常不是列出具体类型：

```go
// 不是显式写 Chicken、Duck、Robot
```

而是通过行为间接确定它们：

```go
type Speaker interface {
	Speak() string
}
```

因此 Go 接口是一种**结构化、隐式实现**：

> 不问你声明自己是什么，只看你实际上会做什么。

再严谨一点说，是“类型的方法集满足接口”，而不是某个实例临时满足接口。例如方法挂在 `*Chicken` 上时：

```go
func (*Chicken) Speak() string {
	return "咯咯哒"
}
```

此时属于 `Speaker` 类型集合的是 `*Chicken`，而不是 `Chicken`：

```go
var s Speaker = &Chicken{} // 可以
var s Speaker = Chicken{}  // 不可以
```

所以你这句话可以整理成：

> **接口定义行为要求，并由这些行为要求确定所有满足条件的类型集合。**

# 主要还是方便需要在这个方法上扩展很多工作时，可以定义成一个统一的函数，而不需要每个类型都定义一个函数

对，核心就是这个意思，不过要修正一点：

> 每个类型仍然要实现自己的那个方法；接口让“围绕这个方法展开的公共工作”只写一次。

例如每种动物都要定义自己的 `Speak`：

```go
type Speaker interface {
	Speak() string
}

type Chicken struct{}
type Duck struct{}

func (Chicken) Speak() string {
	return "咯咯哒"
}

func (Duck) Speak() string {
	return "呱呱"
}
```

因为鸡和鸭具体怎么叫不一样，这部分无法统一。

但围绕“说话”做的工作可以只写一次：

```go
func ProcessSpeech(s Speaker) {
	text := s.Speak()

	fmt.Println("说话内容：", text)
	fmt.Println("记录日志")
	fmt.Println("保存到数据库")
	fmt.Println("发送通知")
}
```

然后所有实现 `Speaker` 的类型都可以使用：

```go
ProcessSpeech(Chicken{})
ProcessSpeech(Duck{})
```

如果没有接口，可能会写成：

```go
func ProcessChicken(c Chicken) {
	text := c.Speak()
	// 日志、保存、通知……
}

func ProcessDuck(d Duck) {
	text := d.Speak()
	// 再重复一遍日志、保存、通知……
}
```

所以职责分成了两部分：

```text
具体类型的方法
→ 决定“具体怎么做”

接收接口的统一函数
→ 决定“做完以后还要执行哪些公共流程”
```

这在后端里非常常见。例如不同通知渠道各自实现：

```go
type Sender interface {
	Send(message string) error
}
```

```go
func (EmailSender) Send(message string) error { /* 发邮件 */ }
func (SMSSender) Send(message string) error   { /* 发短信 */ }
```

而重试、日志、监控等公共逻辑只写一次：

```go
func SendWithRetry(sender Sender, message string) error {
	for i := 0; i < 3; i++ {
		err := sender.Send(message)
		if err == nil {
			return nil
		}
	}

	return errors.New("发送失败")
}
```

因此你可以这样概括：

> **接口让不同类型保留各自的实现，同时让依赖这些行为的公共流程只编写一次。**

# 有点像泛型的感觉

对，**确实很像泛型**，因为它们都在解决：

> 不把代码写死在某一个具体类型上，让多种类型共用同一套逻辑。

例如接口：

```go
type Speaker interface {
	Speak() string
}

func Process(s Speaker) {
	fmt.Println(s.Speak())
}
```

`Process` 可以接收 `Chicken`、`Duck`、`Robot`，只要它们都有 `Speak()`。

泛型也是类似的：

```go
func First[T any](items []T) T {
	return items[0]
}
```

它可以处理：

```go
First([]int{1, 2, 3})
First([]string{"a", "b"})
```

不过二者关注点不同。

## 接口关注“你会什么”

```go
type Speaker interface {
	Speak() string
}
```

它不关心你的底层类型，只关心你有没有这个方法：

```text
Chicken 会 Speak
Duck 会 Speak
Robot 会 Speak
```

所以接口特别适合表达**行为和替换关系**。

## 泛型关注“你的类型可以变化”

```go
func First[T any](items []T) T
```

它不一定要求类型有什么方法，只是让同一份算法适用于不同类型，并且保留具体类型：

```go
n := First([]int{1, 2})   // n 是 int
s := First([]string{"a"}) // s 是 string
```

而接口变量只保留接口允许你使用的行为：

```go
var s Speaker = Chicken{}
```

通过 `s`，你主要只能调用：

```go
s.Speak()
```

不能随便使用 `Chicken` 独有的字段和方法。

可以先这样区分：

```text
接口：不同类型拥有相同行为
泛型：相同算法适用于不同类型
```

例如“让所有动物说话”适合接口：

```go
func MakeSpeak(s Speaker) {
	fmt.Println(s.Speak())
}
```

而“返回任意切片的第一个元素”适合泛型：

```go
func First[T any](items []T) T {
	return items[0]
}
```

有意思的是，Go 的泛型约束也使用 `interface` 语法：

```go
type Number interface {
	~int | ~float64
}

func Add[T Number](a, b T) T {
	return a + b
}
```

所以你的感觉非常准：**接口和泛型都在抽象类型，只是接口主要抽象行为，泛型主要抽象算法中的具体类型。**
