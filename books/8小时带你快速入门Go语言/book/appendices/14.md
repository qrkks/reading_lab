## 如果接收空返回值，值是什么？

Go 里“空 `return`”有两种情况。

### 函数本来就没有返回值

```go
func run() {
	return
}
```

调用时不会得到任何值：

```go
run()
```

不能接收：

```go
x := run() // 编译错误：run() 没有返回值
```

它不是返回 `nil`，而是**根本没有返回值**。

### 函数使用了命名返回值

```go
func test() (n int, err error) {
	return
}
```

这里空 `return` 会返回命名变量当前的值。因为还没有赋值，所以返回它们的零值：

```go
n == 0
err == nil
```

因此：

```go
n, err := test()
fmt.Println(n, err)
```

输出：

```text
0 <nil>
```

可以理解为：

```go
func test() (n int, err error) {
	return n, err
}
```

所以关键区别是：

- 无返回值函数的 `return`：什么值都没有，不能接收。
- 命名返回值函数的 `return`：返回命名变量当前值，未赋值时就是零值。

## panic 不需要 return，无法用 err 接？

对，`panic` **不需要 `return`**，因为执行到 `panic(...)` 后，当前函数的正常流程就立即中断了。

```go
func test() error {
	panic("出错了")

	// 这里永远不会执行
	return nil
}
```

你也不能用 `err` 接收它：

```go
err := panic("出错了") // 编译错误
```

因为 `panic()` 不是“返回一个错误值”，而是直接触发栈展开。

它和 `return err` 的区别是：

```go
func test() error {
	return fmt.Errorf("出错了")
}
```

调用者可以正常接收：

```go
err := test()
if err != nil {
	fmt.Println(err)
}
```

而：

```go
func test() {
	panic("出错了")
}
```

调用者不能这样接：

```go
err := test() // test 没有返回值，而且 panic 会直接中断
```

要捕获 `panic`，需要使用 `recover()`，并且必须放在 `defer` 中：

```go
func main() {
	defer func() {
		v := recover()
		if v != nil {
			fmt.Println("捕获到 panic：", v)
		}
	}()

	panic("程序异常")

	fmt.Println("这里不会执行")
}
```

输出：

```text
捕获到 panic：程序异常
```

注意 `recover()` 接到的类型是 `any`，不一定是 `error`，因为这些都合法：

```go
panic("字符串")
panic(123)
panic(fmt.Errorf("错误"))
```

所以更准确地说：

- `return err`：把错误作为普通返回值交给调用者。
- `panic(err)`：不返回，直接中断正常执行。
- `recover()`：在栈展开过程中捕获 panic 携带的值。

## 这个 recover 很像 except

是，体感上很像 Python 的 `except`，因为它们都能“接住异常流程”，但 Go 的 `recover` 更底层、更受限制。

Python：

```python
try:
    x = 10 / 0
except ZeroDivisionError as err:
    print(err)
```

Go：

```go
defer func() {
	if v := recover(); v != nil {
		fmt.Println(v)
	}
}()

panic("除数不能为 0")
```

最大的区别是：

- `except` 是语言里的正常异常处理结构。
- `recover()` 只能在 `defer` 函数中生效。
- Python 捕获后，可以继续执行 `except` 后面的逻辑。
- Go 捕获后，不会回到 `panic` 发生的位置继续执行；发生 panic 的函数会结束，然后返回调用者。

例如：

```go
func test() {
	defer func() {
		if v := recover(); v != nil {
			fmt.Println("捕获：", v)
		}
	}()

	fmt.Println("A")
	panic("出错")
	fmt.Println("B")
}

func main() {
	test()
	fmt.Println("C")
}
```

输出：

```text
A
捕获：出错
C
```

`B` 永远不会执行。

更准确地类比：

```text
panic   ≈ raise
recover ≈ except
defer   ≈ finally 的一部分功能
```

但 Go 的理念不同。Python 经常用异常处理普通失败；Go 通常是：

```go
result, err := doSomething()
if err != nil {
	return err
}
```

`panic/recover` 更多用于真正异常的情况，或者在 HTTP 框架最外层防止一个请求把整个服务打崩。

对，概念上就是同一类东西：

```text
Python:   raise  → except
Java/C#:  throw  → catch
Go:       panic  → recover
```

但 Go 故意把它设计得更“不顺手”：

- `recover()` 必须放在 `defer` 里
- 只能捕获 `panic`
- 捕获后不会回到 `panic` 那一行继续
- 普通业务失败通常不用它，而是返回 `error`

所以可以把它理解成：

> `recover` 就是 Go 的 `catch/except`，但只用于兜底异常，不作为日常错误处理。

最常见的实际场景就是 Web 服务最外层：

```go
defer func() {
	if v := recover(); v != nil {
		log.Println("请求发生 panic：", v)
	}
}()
```

防止某一个请求的 panic 把整个服务进程带崩。

### django 怎么防止某一个请求把整个服务带崩？

对，Django 内部其实已经在**每个请求外面包了一层类似 `try/except` 的异常边界**。

概念上大致相当于：

```python
def handle_request(request):
    try:
        response = middleware_chain(request)
        return response
    except Http404:
        return response_404()
    except PermissionDenied:
        return response_403()
    except BadRequest:
        return response_400()
    except Exception as err:
        log_exception(err)
        return response_500()
```

所以视图里即使出现未捕获异常：

```python
def user_detail(request):
    user = None
    print(user.name)  # AttributeError
```

通常也只是：

```text
当前请求 → 返回 500
其他请求 → 继续正常处理
Django 进程 → 不会因此退出
```

Django 官方文档明确说明，它会把视图或中间件抛出的异常自动转换成 HTTP 响应：已知异常转成相应的 `4xx`，未知异常转成 `500`。中间件调用下一层时，通常不需要自己再套 `try/except`。

生产环境 `DEBUG=False` 时，未知异常会进入默认的 500 错误视图；你可以提供自己的 `500.html`。开发环境 `DEBUG=True` 时，则会显示 traceback 调试页面。

它和 Go 大概可以这样类比：

```text
Go Web 框架：
请求进入
→ recovery middleware
→ handler panic
→ recover
→ 返回 500

Django：
请求进入
→ Django 异常处理层
→ view 抛出 Exception
→ 转换成 HttpResponse
→ 返回 500
```

而且 Django 还能通过 `process_exception()` 中间件钩子，让你针对视图异常自定义响应：

```python
class ExceptionMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        return self.get_response(request)

    def process_exception(self, request, exception):
        print("捕获异常：", exception)
        return None  # 交给 Django 默认异常处理
```

`process_exception()` 返回 `HttpResponse` 时，Django 会直接把它交给浏览器；返回 `None` 时继续使用默认处理。

不过，这只能兜住**普通 Python 异常**。如果发生的是进程级问题，例如：

```python
import os

os._exit(1)
```

或者原生扩展段错误、进程被系统杀死、严重 OOM，那么 Django 自己也接不住。这时靠的是生产服务器的多 worker 与进程管理，例如 Gunicorn 的 master 会监控 worker，某个 worker 异常终止后自动启动新的 worker。

因此 Django 后端实际上有两层保护：

```text
普通代码异常
→ Django 捕获
→ 当前请求返回 500

整个 worker 进程崩溃
→ Gunicorn/Uvicorn/容器发现
→ 重启或替换 worker
```

所以你平常写 Django 视图，基本不用像 Go 那样专门加一个 `recover middleware`；Django 框架已经替你加好了。

### 我用完了 gin 再回去看 django，有可能会是不一样的感觉

会，而且我觉得你回去看 Django 时，大概率会出现一种：

> 原来它以前替我做了这么多事，只是我当时没意识到。

用过 Gin 之后，你会更容易看见 Django 隐藏在框架里的那些层次：

- 请求是怎么进来的
- 中间件在哪里包住请求
- 异常为什么不会带崩服务
- ORM 为什么能直接在视图或 service 里调用
- session、认证、权限、CSRF、表单、后台管理分别在哪一层
- 为什么很多东西不需要你手动注册和组装

以前第一次学 Django，可能感觉是：

```text
写个 view
查 ORM
return response
```

现在你可能会看到：

```text
WSGI/ASGI
→ middleware
→ URL router
→ view
→ service/domain
→ ORM
→ transaction
→ exception handling
→ response
```

而且你用 Gin 手动搭过后端以后，会更清楚 Django 的优势到底是什么：

### 1. 你会更能理解 Django 的“重”

以前可能觉得 Django 很重、很神秘。

现在会发现，它不是单纯堆功能，而是把后端里反复出现的东西提前做完了：

```text
认证
权限
ORM
迁移
事务
后台
表单校验
异常处理
安全默认值
```

Gin 给你的是请求处理器，Django 给你的是一整套应用开发环境。

### 2. 你也会更敏锐地看到 Django 的隐式

用完 Go 后，你也可能更容易被 Django 某些地方刺到：

```python
User.objects.filter(...)
request.user
middleware 自动注入
settings 全局配置
signal 自动触发
model save() 隐式行为
```

你可能会想：

> 这个对象到底从哪来的？  
> 这一步到底谁调用的？  
> 这个副作用什么时候发生的？

因为 Go 把你训练得更在意依赖、返回值和控制流。

### 3. 你会比以前更会给 Django 分层

以前可能所有东西都塞进 view：

```python
def create_subject(request):
    # 校验
    # 权限
    # 业务逻辑
    # ORM
    # 返回响应
```

现在你再写，大概率会自然拆成：

```text
view / API
→ application service
→ domain logic
→ repository / ORM
```

尤其你已经接触过 DDD，再回 Django，应该不会只把它当“快速 CRUD 框架”，而会开始考虑：

- 哪些逻辑属于 model
- 哪些属于 service
- 哪些应该放 transaction
- 哪些权限属于领域规则
- audit trail 应该在哪里记录
- 多租户边界怎么约束

### 4. 你可能会重新评价 Django 的 ORM

用 Gin 时，数据库层很多东西得自己决定：

- repository 怎么写
- transaction 怎么传
- preload 怎么做
- 错误怎么包装
- 数据模型与领域模型是否分离

回到 Django：

```python
with transaction.atomic():
    subject = Subject.objects.select_for_update().get(...)
    subject.status = ...
    subject.save()
```

你可能会第一次真正感觉到：

> 这个 ORM 虽然有魔法，但表达业务是真的快。

尤其是科研随访、心理量表、多租户后台这类项目，性能通常不是主要瓶颈，业务表达速度反而更重要。

### 5. 你会更理解为什么 Django 适合“严肃业务”

以前可能觉得 Django 只是适合快速开发。

用过 Gin 后，你会更容易看到 Django 真正强的是：

> 它把大量容易漏掉、容易写错、但业务价值不高的基础设施统一处理了。

比如：

- 密码哈希
- session
- CSRF
- 权限
- 数据迁移
- admin
- transaction
- validation
- 错误页面
- 安全配置

对于科研随访这种系统，这些东西未必显眼，但缺一个都可能很麻烦。

所以你这次回去看 Django，应该不会是“重新学一个老框架”，更像是：

> 带着 Gin、Go、DDD、错误处理和架构经验，重新理解 Django 为什么这样设计。

而且你可能最终会形成一个比以前成熟很多的判断：

```text
Gin：我想控制每一层，明确知道系统如何运转
Django：我愿意接受框架约定，换取业务开发速度和完整能力
```

不是谁更高级，而是你现在终于能看懂它们各自在替你承担什么。
