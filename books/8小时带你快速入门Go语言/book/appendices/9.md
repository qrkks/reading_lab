# 9

## go 的结构体方法一般用什么命名格式？

Go 的结构体方法命名和普通函数一样，通常使用**驼峰命名法**：

```go
type User struct {
	name string
}

func (u User) Name() string {
	return u.name
}

func (u *User) SetName(name string) {
	u.name = name
}
```

首字母大小写决定可见性：

```go
func (u User) Name() string     {} // 导出，其他包可以调用
func (u User) fullName() string {} // 不导出，只能当前包使用
```

### 接收者名称

接收者通常使用类型名的一两个字母，不使用 `this` 或 `self`：

```go
func (u User) Name() string  {}
func (us UserService) Find() {}
func (c Client) Close()      {}
```

一般不要写：

```go
func (this User) Name() string {}
func (self User) Name() string {}
```

同一个类型的接收者名称最好保持一致：

```go
func (u User) Name() string         {}
func (u User) Age() int             {}
func (u *User) SetName(name string) {}
```

### Getter 通常不加 `Get`

Go 通常写：

```go
func (u User) Name() string {
	return u.name
}
```

而不是：

```go
func (u User) GetName() string {
	return u.name
}
```

修改值的方法可以使用 `Set`：

```go
func (u *User) SetName(name string) {
	u.name = name
}
```

### 常见的方法命名

```go
user.Name()
user.Validate()
user.Save()
user.Delete()

service.FindUser()
service.CreateUser()

file.Close()
buffer.Reset()
list.Add(item)
list.Remove(id)
```

布尔方法一般让名字读起来像一个判断：

```go
func (u User) IsActive() bool
func (u User) HasPermission() bool
func (u User) CanEdit() bool
```

### 是否使用指针接收者不影响命名

```go
func (u User) Name() string {
	return u.name
}

func (u *User) Rename(name string) {
	u.name = name
}
```

通常：

- 只读取、小结构体，可以用值接收者 `User`
- 要修改结构体，使用指针接收者 `*User`
- 同一个类型的方法一般尽量统一接收者形式

可以先记成：

> 方法名用 `MixedCaps`；导出方法首字母大写；接收者用类型名的简短缩写；Getter 通常直接叫字段名，不叫 `GetX`。

## 包级函数

### 就是有没有挂在结构体上的，不挂在实例上的方法

对，你问的是类似：

```python
User.create()
```

或者：

```java
User.create()
```

这种“挂在类型上、不挂在具体实例上”的静态方法或类方法。

**Go 没有这种方法。**

Go 的方法必须带接收者：

```go
func (u User) Name() string {
	return u.name
}
```

因此通常要通过实例调用：

```go
u.Name()
```

不能定义成：

```go
func User.Create() {} // Go 没有这种语法
```

Go 的替代方式就是包级函数：

```go
package user

type User struct {
	Name string
}

func New(name string) *User {
	return &User{Name: name}
}

func Parse(data string) (User, error) {
	// ...
}
```

调用时：

```go
u := user.New("Tom")
u2, err := user.Parse(data)
```

看起来其实也很像“挂在 User 上”：

```text
user.New()
user.Parse()
```

只不过这里的 `user` 是**包名**，不是结构体类型。

还有这种语法：

```go
f := User.Name
```

它叫方法表达式，但不是类方法。它只是把实例方法拿出来变成普通函数：

```go
name := f(u)
```

相当于：

```go
name := u.Name()
```

所以结论是：

> Go 没有挂在结构体类型本身上的静态方法；不依赖实例的方法通常写成包级函数。

### 这种方式常用吗？

非常常用，而且这是 Go 的惯用写法。

不依赖某个实例的逻辑，通常直接写成包级函数：

```go
package user

type User struct {
	Name string
}

func New(name string) *User {
	return &User{Name: name}
}

func Parse(data string) (User, error) {
	// ...
}
```

调用时：

```go
u := user.New("Tom")
u2, err := user.Parse(data)
```

它在使用体验上其实很像静态方法：

```text
Java/Python: User.Parse(...)
Go:        user.Parse(...)
```

只是 Go 把函数挂在**包命名空间**上，而不是挂在结构体类型上。

Go 标准库大量使用这种风格：

```go
time.Parse(...)
url.Parse(...)
regexp.Compile(...)
errors.New(...)
http.ListenAndServe(...)
```

构造函数尤其常见：

```go
bytes.NewBuffer(...)
http.NewRequest(...)
json.NewEncoder(...)
```

常见命名模式包括：

```go
New(...)
NewClient(...)
Parse(...)
Compile(...)
Open(...)
Load(...)
MustCompile(...)
```

如果一个包主要围绕一个类型设计，通常可以省略类型名：

```go
package user

func New(...) *User
func Parse(...) User
```

调用已经很明确：

```go
user.New(...)
user.Parse(...)
```

如果同一个包里有多个核心类型，就可能把类型写进函数名：

```go
func NewUser(...)
func NewAdmin(...)
func ParseUser(...)
```

所以可以把 Go 的习惯理解成：

> 有实例状态的逻辑挂在结构体方法上；不需要实例的逻辑挂在包上。

### 它实际结构上就是普通函数啊，只是返回值是结构体指针？

对，**从语法和结构上看，它就是一个普通的包级函数**。

例如：

```go
type User struct {
	Name string
}

func New(name string) *User {
	return &User{
		Name: name,
	}
}
```

`New` 没有任何特殊身份：

```go
func New(name string) *User
```

它只是：

- 名字叫 `New`
- 接收一个 `string`
- 返回一个 `*User`

调用：

```go
u := user.New("Tom")
```

看起来像“构造函数”，只是因为：

1. 它放在 `user` 包里；
2. 它返回 `User`；
3. Go 社区约定把这类函数命名为 `New`、`NewClient` 等。

Go 编译器并不会把 `New` 当成特殊关键字。你完全可以改名：

```go
func Create(name string) *User {
	return &User{Name: name}
}
```

甚至可以返回结构体值：

```go
func New(name string) User {
	return User{Name: name}
}
```

或者返回结构体和错误：

```go
func New(name string) (*User, error) {
	if name == "" {
		return nil, errors.New("name cannot be empty")
	}

	return &User{Name: name}, nil
}
```

甚至它未必返回指针：

```go
func Default() User {
	return User{Name: "Guest"}
}
```

所以你理解得对：

> Go 的“构造函数”本质上就是普通函数，只是按照约定返回某种结构体值或结构体指针。

而真正的方法，区别只在于它多了一个接收者：

```go
func (u *User) Rename(name string) {
	u.Name = name
}
```

可以粗略对比：

```go
func New(name string) *User        // 普通包级函数
func (u *User) Rename(name string) // User 的方法
```

`New` 不属于 `User` 类型本身，它属于当前包；`Rename` 才属于 `User` 的方法集。

## 一般实例化的时候是实例化成一个指针？

不一定。Go 里“创建结构体”默认得到的是**结构体值**：

```go
u := User{
	Name: "Tom",
}
```

此时 `u` 的类型是：

```go
User
```

加上 `&` 才得到结构体指针：

```go
u := &User{
	Name: "Tom",
}
```

此时 `u` 的类型是：

```go
*User
```

### 实际项目里哪种更常见

对于 `User`、`Service`、`Client`、`Repository` 这种会变化、需要共享同一个实例的对象，确实经常创建成指针：

```go
user := &User{Name: "Tom"}
service := &UserService{}
client := &http.Client{}
```

构造函数也经常返回指针：

```go
func NewUser(name string) *User {
	return &User{Name: name}
}
```

而对于较小、适合复制的“值对象”，通常直接创建值：

```go
point := Point{X: 1, Y: 2}
color := Color{R: 255, G: 0, B: 0}
config := Config{Port: 8080}
```

### 一个容易误解的地方

即使变量本身是结构体值：

```go
u := User{Name: "Tom"}
```

也能直接调用指针接收者方法：

```go
func (u *User) Rename(name string) {
	u.Name = name
}

u.Rename("Bob")
```

Go 会在 `u` 可取地址时，自动理解成：

```go
(&u).Rename("Bob")
```

所以看到：

```go
u.Rename("Bob")
```

不能仅凭调用形式判断 `u` 是 `User` 还是 `*User`。

可以先这样理解：

> Go 创建结构体时既可以创建值，也可以创建指针；业务对象常用指针，小型值对象常用值。

另外这两个基本等价：

```go
u1 := &User{}
u2 := new(User)
```

它们的类型都是 `*User`，不过需要填写字段时，`&User{...}` 更常见。

### 指针也可以直接操作啊，这是不是 go 的语法糖？

对，这是 Go 提供的**自动解引用语法糖**。

假设：

```go
type User struct {
	Name string
}

u := &User{Name: "Tom"}
```

`u` 的类型是 `*User`，按底层逻辑访问字段应该写：

```go
(*u).Name
```

但 Go 允许直接写：

```go
u.Name
```

它会自动理解成：

```go
(*u).Name
```

修改字段也一样：

```go
u.Name = "Bob"
```

等价于：

```go
(*u).Name = "Bob"
```

所以 Go 里通常几乎看不到这种写法：

```go
(*u).Name
```

### 方法调用也会自动转换

如果方法使用指针接收者：

```go
func (u *User) Rename(name string) {
	u.Name = name
}
```

即使你手中是结构体值：

```go
u := User{Name: "Tom"}
u.Rename("Bob")
```

Go 会自动理解成：

```go
(&u).Rename("Bob")
```

反过来，如果你有指针，却调用值接收者方法：

```go
func (u User) NameLength() int {
	return len(u.Name)
}

p := &User{Name: "Tom"}
p.NameLength()
```

Go 会自动理解成：

```go
(*p).NameLength()
```

因此有两种常见自动转换：

```go
p.Field      // 自动解引用，类似 (*p).Field
value.Method() // 必要时自动取地址，类似 (&value).Method()
```

不过 `*` 和 `&` 并没有消失。在赋值、传参或明确操作指针时，仍然要写：

```go
p := &u       // 取地址
value := *p   // 解引用，复制出结构体值
```

所以准确地说：

> Go 对结构体的字段选择和方法调用提供了自动取地址、自动解引用；但一般的指针操作仍然需要显式写 `&` 和 `*`。

### 那真实实例化成指针比较方便啊，修改也会同步，不会造成困扰

对，**对于“会变化、需要共享同一个对象”的结构体，直接使用指针通常确实更方便**：

```go
user := &User{Name: "Tom"}
```

后面传给别的函数：

```go
func rename(u *User) {
	u.Name = "Bob"
}

rename(user)

fmt.Println(user.Name) // Bob
```

大家操作的是同一个 `User`，修改会同步，不会因为复制而改错对象。

所以这些类型经常用指针：

```go
*User
*Service
*Repository
*Client
*Server
```

它们通常有明确的“对象身份”，并且状态会变化。

不过指针也可能造成另一类困扰：**多个地方都能悄悄修改同一个对象**。

```go
a := &User{Name: "Tom"}
b := a

b.Name = "Bob"

fmt.Println(a.Name) // Bob
```

这里 `a`、`b` 是两个变量，但指向同一个结构体。代码复杂后，可能不知道究竟是谁修改了它。

指针还有 `nil` 的问题：

```go
var u *User

fmt.Println(u.Name) // panic
```

值则天然存在零值：

```go
var u User

fmt.Println(u.Name) // ""
```

因此更准确的选择标准不是“指针总是更方便”，而是：

> 希望大家操作同一个、会变化的对象，就用指针；希望它像数字、坐标、配置快照一样独立复制，就用值。

例如用户实体通常适合指针：

```go
func NewUser(name string) *User {
	return &User{Name: name}
}
```

坐标更适合值：

```go
type Point struct {
	X, Y float64
}

p1 := Point{1, 2}
p2 := p1
p2.X = 10

fmt.Println(p1.X) // 1
```

这里复制反而更符合直觉：修改 `p2` 不应该影响 `p1`。

所以你的感觉对了一半：

> 指针避免了“修改副本没生效”的困扰，但也引入了“多个地方共享修改”的困扰。

Go 需要你根据类型语义，在这两种困扰之间选择。

### 而且指针与值使用完全一样，几乎无感啊

对，Go **故意把结构体值和结构体指针的日常操作做得几乎一样**：

```go
u := User{Name: "Tom"}   // User
p := &User{Name: "Tom"}  // *User

fmt.Println(u.Name)
fmt.Println(p.Name)

u.Name = "Alice"
p.Name = "Bob"
```

字段访问都用 `.`，方法调用通常也都用 `.`。Go 会在合适的时候自动取地址或解引用。

所以使用体验确实很无感。但可以记一句：

> **表面语法几乎一样，复制与共享的语义完全不同。**

#### 赋值时区别最大

值会复制：

```go
u1 := User{Name: "Tom"}
u2 := u1

u2.Name = "Bob"

fmt.Println(u1.Name) // Tom
```

指针会共享：

```go
p1 := &User{Name: "Tom"}
p2 := p1

p2.Name = "Bob"

fmt.Println(p1.Name) // Bob
```

#### 普通函数传参不会自动转换

方法调用可以自动取地址：

```go
u.Rename("Bob") // Rename 接收 *User 时，自动看作 (&u).Rename(...)
```

但普通函数不行：

```go
func rename(u *User) {
	u.Name = "Bob"
}

user := User{Name: "Tom"}

rename(&user) // 必须明确写 &
```

不能写：

```go
rename(user) // 编译错误
```

反过来也一样：

```go
func printUser(u User) {}

p := &User{Name: "Tom"}
printUser(*p) // 必须显式解引用
```

#### 指针还能是 `nil`

```go
var p *User
fmt.Println(p.Name) // panic
```

值则始终至少有一个零值：

```go
var u User
fmt.Println(u.Name) // ""
```

还有接口方法集等更深层的区别。例如只有指针接收者方法时，往往是 `*User` 实现接口，而不是 `User`。

所以你的感觉没错：**Go 在字段和方法调用层面把指针隐藏得很好**。真正需要关注指针和值区别的地方，主要是：

```text
赋值、传参、共享修改、nil、接口实现
```

这也是 Go 比 C 指针好用很多的地方：该让你看到指针语义时让你看到，普通操作时又不逼你整天写 `(*p).Field`。

## go 的组合跟数据库的思想一致啊

对，**很像数据库里的“关系建模”**。

你原来的结构：

```go
type User struct {
	username string
	email    string
	group    Group
}
```

表达的是：

> 一个 `User` 拥有一个 `Group`

这就是典型的 **has-a 关系**，对应数据库里大概会设计成：

```text
users
- id
- username
- email
- group_id

groups
- id
- name
```

然后通过：

```text
users.group_id → groups.id
```

建立关联。

所以可以类比为：

```go
group Group
```

类似“用户关联一个组”。

如果是多个组：

```go
type User struct {
	groups []Group
}
```

就类似一对多或多对多关系。

不过 Go 结构体组合和数据库关系仍然有一个重要区别：

```go
type User struct {
	group Group
}
```

这是直接把完整的 `Group` 值放在 `User` 里面，比较像数据库里的**内嵌字段**或查询完成后的对象结果。

而数据库通常只保存外键：

```go
type User struct {
	groupID int
}
```

或者 ORM 中同时写：

```go
type User struct {
	GroupID int
	Group   Group
}
```

这里：

- `GroupID` 是真正存进数据库的外键
- `Group` 是 ORM 查询关联数据后填进去的对象

至于嵌入：

```go
type User struct {
	Group
}
```

它更像“把 `Group` 的字段和行为提升到 `User` 上”，不太像普通数据库外键。它有一点像扁平化：

```go
user.name
```

实际上来自：

```go
user.Group.name
```

因此你的感觉很准：

> Go 倾向于通过组合建立对象关系；关系数据库也倾向于通过表与外键组合数据。

它们背后都是同一种思想：

> 不把所有东西塞进一个庞大的类型或表里，而是拆成独立部分，再通过关系组合起来。

这也正是“组合优于继承”为什么很自然：数据库本身基本没有面向对象那种类继承，主要就是实体、字段和关系。

### 怪不得它那么适合写后端

对，Go 很多设计都天然贴着后端问题走。

后端本质上经常是在处理这些东西：

```text
请求
→ 校验
→ 调用服务
→ 查数据库
→ 拼装结构体
→ 返回 JSON
```

而 Go 的类型系统、结构体、组合、接口、错误返回，刚好都很适合这条链路。

比如数据库查出来的数据，很自然地装进结构体：

```go
type User struct {
	ID      int
	Name    string
	GroupID int
	Group   Group
}
```

业务之间也通常不是复杂继承，而是组合：

```go
type UserService struct {
	repo   UserRepository
	mailer Mailer
}
```

这和后端实际依赖关系非常一致：用户服务“拥有”仓储和邮件组件，而不是“继承”仓储。

接口也很贴合分层：

```go
type UserRepository interface {
	FindByID(id int) (*User, error)
}
```

Service 只依赖“能查用户”这个能力，不关心底层是 MySQL、PostgreSQL 还是内存实现。

错误处理也和请求链路很匹配：

```go
user, err := repo.FindByID(id)
if err != nil {
	return nil, err
}
```

错误沿着调用链一层层返回，很直白。虽然写起来有点重复，但后端出问题时容易定位，不太会像异常一样突然从深处跳出来。

再加上：

- 标准库里的 HTTP、JSON、网络支持很强
- goroutine 很适合大量并发请求
- 编译成单个二进制，部署省心
- 没有复杂运行时环境
- 代码风格统一，团队项目容易读
- 性能通常远远超过普通业务需求

所以 Go 并不是每个单独能力都最强，而是它整套设计和后端工作的形状特别一致。

甚至可以说：

> Go 不是把后端写得特别“魔法”，而是让后端代码看起来就像后端业务本身。

这也是你可能会觉得 Go 结构体、数据库模型、DDD 里的实体和服务之间非常顺：它们基本都在用“数据 + 组合 + 明确依赖”这一套思路。

## omitempty 常用吗？

很常用，尤其是 API 响应、配置结构体和 PATCH 请求里。

```go
type User struct {
	Name     string  `json:"name,omitempty"`
	Age      int     `json:"age,omitempty"`
	Nickname *string `json:"nickname,omitempty"`
}
```

序列化时，字段是零值就不输出：

```go
u := User{Name: "Tom"}
```

结果：

```json
{ "name": "Tom" }
```

常见零值包括：

```text
string      ""
int         0
bool        false
pointer     nil
slice       nil 或长度为 0
map         nil 或长度为 0
```

不过它有一个重要问题：**无法区分“没填”和“明确填了零值”**。

```go
type UpdateUserRequest struct {
	Age int `json:"age,omitempty"`
}
```

如果客户端明确传：

```json
{ "age": 0 }
```

再次序列化时，`age` 可能被省略，因为 `0` 是零值。

需要区分时，通常用指针：

```go
type UpdateUserRequest struct {
	Age    *int  `json:"age,omitempty"`
	Active *bool `json:"active,omitempty"`
}
```

这样：

```text
nil      表示没有提供
&0       表示明确提供了 0
&false   表示明确提供了 false
```

还有一个常见误区：普通结构体字段不一定会因为 `omitempty` 而省略。

```go
type Profile struct {
	Name string `json:"name"`
}

type User struct {
	Profile Profile `json:"profile,omitempty"`
}
```

即使 `Profile` 是零值结构体，也可能仍然输出：

```json
{ "profile": { "name": "" } }
```

想让整个对象可省略，通常写指针：

```go
type User struct {
	Profile *Profile `json:"profile,omitempty"`
}
```

所以可以这样用：

- API 响应中不想输出空字段：常用
- 可选嵌套对象：用指针配合 `omitempty`
- PATCH / 更新请求：常用，但字段最好用指针
- 数据库实体：不要为了 JSON 随便到处加，最好单独定义请求和响应结构体

一句话记：

> `omitempty` 很常用，但它表达的是“零值不输出”，不完全等于“可选字段”。

## 挂方法的时候，如果不修改就挂到结构体值上；如果修改就挂在结构体指针上？

对，**这是最基本的判断规则**：

```go
// 不修改原对象：值接收者
func (u User) Name() string {
	return u.Username
}

// 修改原对象：指针接收者
func (u *User) SetName(name string) {
	u.Username = name
}
```

值接收者拿到的是结构体副本：

```go
func (u User) SetName(name string) {
	u.Username = name // 只修改副本
}
```

调用后原值不会变：

```go
user := User{Username: "Tom"}
user.SetName("Bob")

fmt.Println(user.Username) // 仍然是 Tom
```

而指针接收者操作的是原对象：

```go
func (u *User) SetName(name string) {
	u.Username = name
}
```

```go
user.SetName("Bob")
fmt.Println(user.Username) // Bob
```

不过实际选择接收者时，还要考虑另外几件事。

#### 结构体很大时，也常用指针

即使方法不修改，也可能用指针避免复制：

```go
func (u *User) FullName() string {
	return u.Username
}
```

不过小结构体的复制成本通常很低，不必为了性能到处使用指针。

#### 同一个类型通常尽量统一

假如 `User` 有很多会修改状态的方法：

```go
func (u *User) SetName(name string)
func (u *User) Activate()
func (u *User) Disable()
```

那么读取方法通常也会统一写成指针接收者：

```go
func (u *User) Name() string
func (u *User) IsActive() bool
```

这样方法集、接口实现和使用方式会更一致。

#### 某些结构体不应该被复制

例如包含互斥锁：

```go
type Counter struct {
	mu    sync.Mutex
	value int
}
```

这种类型的方法都应该使用指针接收者：

```go
func (c *Counter) Value() int {
	c.mu.Lock()
	defer c.mu.Unlock()

	return c.value
}
```

因为复制锁可能导致严重问题。

所以可以这样判断：

```text
需要修改原对象              → 指针接收者
结构体很大或不应该被复制    → 指针接收者
类型整体具有对象身份        → 通常统一用指针接收者
小型、不可变式的值对象      → 可以用值接收者
```

例如：

```go
type Point struct {
	X, Y float64
}

func (p Point) Length() float64 {
	return math.Sqrt(p.X*p.X + p.Y*p.Y)
}
```

`Point` 很像数字或坐标，适合值接收者。

而：

```go
type User struct {
	Username string
	Active   bool
}
```

`User` 通常是会持续变化、有明确身份的业务对象，整体统一使用 `*User` 往往更自然。

### 那统一挂到指针上不就可以了

可以，而且对 `User`、`Service`、`Client` 这类有状态对象，**统一挂到指针上通常很合理**：

```go
type User struct {
	Username string
}

func (u *User) Name() string {
	return u.Username
}

func (u *User) SetName(name string) {
	u.Username = name
}
```

这样读写方法风格统一，也不会复制结构体。

不过代价是：**只有 `*User` 实现这些方法对应的接口，`User` 值不实现。**

```go
type Namer interface {
	Name() string
}

var u User
var p *User = &u

var _ Namer = p // 可以
var _ Namer = u // 不可以
```

因为指针接收者的方法只属于 `*User` 的方法集。

还有一些不可取地址的值，不能自动转成指针调用：

```go
func getUser() User {
	return User{Username: "Tom"}
}

getUser().SetName("Bob") // 错误：函数返回的临时值不可取地址
```

map 中的结构体值也类似：

```go
users := map[string]User{
	"tom": {Username: "Tom"},
}

users["tom"].SetName("Bob") // 错误：map 元素不可取地址
```

如果 map 里保存指针就没问题：

```go
users := map[string]*User{
	"tom": {Username: "Tom"},
}

users["tom"].SetName("Bob")
```

所以实际可以采用这个规则：

- `User`、`Order`、`Service` 等有身份、会变化的对象：通常统一用指针接收者。
- `Point`、`Color`、`Time` 这类小型值对象：通常用值接收者。

```go
type Point struct {
	X, Y float64
}

func (p Point) Length() float64 {
	return math.Sqrt(p.X*p.X + p.Y*p.Y)
}
```

你的思路对于后端业务实体很适用：**只要一个类型有方法需要修改自身，通常整套方法统一使用指针接收者，最省心。**

## 匿名嵌入的行为是不是就很像继承了？

对，**表面行为很像继承**，但 Go 官方和惯用思路仍然把它看作**组合 + 方法提升**，不是传统面向对象继承。

例如：

```go
type Group struct {
	Name string
}

func (g Group) Describe() string {
	return "组：" + g.Name
}

type User struct {
	Group
}
```

因为匿名嵌入了 `Group`，可以直接写：

```go
user := User{
	Group: Group{Name: "管理员"},
}

fmt.Println(user.Name)
fmt.Println(user.Describe())
```

实际上相当于：

```go
fmt.Println(user.Group.Name)
fmt.Println(user.Group.Describe())
```

Go 只是允许你省略中间的 `.Group`。这叫**字段和方法提升**。

### 为什么看起来像继承

它有几个类似继承的表现：

```go
type User struct {
	Group
}
```

`User` 可以直接使用 `Group` 的：

- 字段
- 方法
- 某些接口能力

例如：

```go
type Describer interface {
	Describe() string
}

var d Describer = User{
	Group: Group{Name: "管理员"},
}
```

因为 `Group.Describe()` 被提升了，`User` 也可能因此实现 `Describer`。

### 但它不是继承

最重要的区别是：**User 不是 Group 的子类型。**

```go
var g Group
var u User

g = u // 错误：User 不是 Group
```

必须明确取出内部字段：

```go
g = u.Group
```

传统继承通常表达：

```text
Dog is an Animal
```

而 Go 嵌入表达的是：

```text
User has a Group
```

只是访问语法被简化了。

### “重写方法”也不完全一样

外层类型可以定义同名方法：

```go
func (u User) Describe() string {
	return "用户：" + u.Username
}
```

此时：

```go
user.Describe()       // 调用 User.Describe
user.Group.Describe() // 仍然可以调用 Group.Describe
```

这更像**外层方法遮蔽了被提升的方法**，不是传统继承中的虚方法重写。

更明显的区别是：

```go
type Group struct{}

func (g Group) Print() {
	g.Describe()
}

func (g Group) Describe() {
	fmt.Println("Group")
}
```

即使 `User` 自己也定义了 `Describe()`，调用：

```go
user.Group.Print()
```

里面仍然调用 `Group.Describe()`，不会自动动态分派到 `User.Describe()`。

所以可以这样理解：

> Go 的匿名嵌入，在使用体验上像继承；在结构和语义上，仍然只是包含了另一个字段，并由编译器提供字段和方法转发的语法糖。

因此有人把它称为：

```text
implementation inheritance
```

但最好还是按 Go 的思路理解为：

```text
组合 + 提升
```

对于 `User` 和 `Group` 这种普通业务关系，通常还是具名字段更明确：

```go
type User struct {
	Group Group
}
```

匿名嵌入更适合你确实希望外层类型“直接获得内部类型能力”的场景。

## 组合是只组合字段，还是挂载的方法也一并组合进去了？

要分两种情况。

### 普通具名字段：只组合数据，不提升方法

```go
type Group struct {
	Name string
}

func (g Group) Describe() string {
	return "组：" + g.Name
}

type User struct {
	Group Group
}
```

`User` 里确实包含一个完整的 `Group`，所以它包含：

- `Group` 的字段数据
- 一个可以调用 `Group` 方法的对象

但方法不会直接提升到 `User`：

```go
user.Group.Name
user.Group.Describe()
```

不能写：

```go
user.Name       // 不行
user.Describe() // 不行
```

所以普通组合更准确地说是：

> `User` 拥有一个 `Group`，通过这个字段使用它的字段和方法。

---

### 匿名嵌入：字段和方法都会被提升

```go
type User struct {
	Group
}
```

现在可以写：

```go
user.Group.Name
user.Group.Describe()
```

也可以省略中间的 `Group`：

```go
user.Name
user.Describe()
```

所以匿名嵌入会把内部类型的：

- 字段
- 方法

都“提升”到外层使用。

但并不是把方法真正复制进了 `User`，而是编译器帮你转发：

```go
user.Describe()
```

大致相当于：

```go
user.Group.Describe()
```

### 指针方法也要看嵌入方式

```go
func (g *Group) Rename(name string) {
	g.Name = name
}
```

如果嵌入值：

```go
type User struct {
	Group
}

user.Rename("管理员")
```

只要 `user` 是可取地址变量，Go 通常会自动取地址，所以可以调用。

如果直接嵌入指针：

```go
type User struct {
	*Group
}
```

那么 `Group` 的值接收者和指针接收者方法都可以直接提升：

```go
user.Describe()
user.Rename("管理员")
```

但要小心嵌入的 `*Group` 可能是 `nil`。

### 一句话区别

```go
Group Group // 拥有 Group，方法通过 user.Group.Method() 调用
Group       // 嵌入 Group，方法可提升为 user.Method()
```

所以“组合”广义上包含字段和行为，但：

> **普通具名组合只显式包含字段对象；匿名嵌入才会把内部方法提升到外层。**

## 那匿名提升的方法能重写吗？能覆盖吗？

可以“覆盖调用入口”，但不是传统继承里的虚方法重写。

```go
type Group struct {
	Name string
}

func (g Group) Describe() string {
	return "Group: " + g.Name
}

type User struct {
	Group
	Username string
}

func (u User) Describe() string {
	return "User: " + u.Username
}
```

调用：

```go
user := User{
	Group:    Group{Name: "管理员"},
	Username: "qrkks",
}

fmt.Println(user.Describe())       // User: qrkks
fmt.Println(user.Group.Describe()) // Group: 管理员
```

此时：

```go
user.Describe()
```

优先调用 `User.Describe()`，嵌入的 `Group.Describe()` 被遮蔽了，但并没有消失，仍可显式调用：

```go
user.Group.Describe()
```

更准确的说法是：

> 外层类型的同名方法遮蔽了被提升的方法，而不是重写了它。

### 不会产生动态分派

例如：

```go
type Group struct{}

func (g Group) Print() {
	fmt.Println(g.Describe())
}

func (g Group) Describe() string {
	return "Group"
}

type User struct {
	Group
}

func (u User) Describe() string {
	return "User"
}
```

调用：

```go
user := User{}
user.Print()
```

输出仍然是：

```text
Group
```

因为 `Group.Print()` 里面的接收者是 `Group`，它调用的是：

```go
g.Describe()
```

不会自动跳到外层的 `User.Describe()`。

这和 Java、Python 继承中的虚方法重写不同：

```text
Go 嵌入：静态选择、方法提升、同名遮蔽
传统继承：可能根据真实对象类型动态调用子类方法
```

还要注意多个嵌入类型有同名方法时：

```go
type User struct {
	Group
	Profile
}
```

如果 `Group` 和 `Profile` 都有 `Describe()`：

```go
user.Describe() // 编译错误：有歧义
```

必须明确写：

```go
user.Group.Describe()
user.Profile.Describe()
```

或者由 `User` 自己定义一个：

```go
func (u User) Describe() string {
	return u.Group.Describe()
}
```

所以结论是：

> 能用外层同名方法“盖住”匿名嵌入提升的方法，但这叫遮蔽，不是真正的继承重写。

## 匿名组合一般常用吗？

常用，但不是“结构体组合的默认写法”。更准确地说：

> **具名字段非常普遍；匿名嵌入只在确实希望提升字段或方法时使用。**

普通业务关系一般不用匿名嵌入：

```go
type User struct {
	Group Group
}
```

因为语义是：

```text
User 拥有一个 Group
```

应该明确调用：

```go
user.Group.Name
user.Group.Validate()
```

若写成：

```go
type User struct {
	Group
}
```

就会让 `Group` 的字段和方法直接出现在 `User` 的 API 上：

```go
user.Name
user.Validate()
```

对于“用户属于用户组”这种关系，通常反而容易混淆。

### 常见的匿名嵌入场景

#### 包装一个类型，只改少数行为

例如包装 `http.ResponseWriter`：

```go
type LoggingWriter struct {
	http.ResponseWriter
	StatusCode int
}

func (w *LoggingWriter) WriteHeader(code int) {
	w.StatusCode = code
	w.ResponseWriter.WriteHeader(code)
}
```

因为嵌入了 `http.ResponseWriter`，未重写的方法仍然可以直接使用；这里只覆盖需要拦截的 `WriteHeader`。

这种场景非常适合嵌入：

> 保留原类型的大部分能力，只增强或修改少量行为。

#### 组合基础能力

```go
type SafeCounter struct {
	sync.Mutex
	value int
}

func (c *SafeCounter) Increment() {
	c.Lock()
	defer c.Unlock()

	c.value++
}
```

嵌入 `sync.Mutex` 后，可以直接：

```go
c.Lock()
c.Unlock()
```

不过这样也意味着 `Lock` 和 `Unlock` 成为了 `SafeCounter` 的公开方法。如果不希望暴露，可以用具名字段：

```go
type SafeCounter struct {
	mu    sync.Mutex
	value int
}
```

#### 框架或 ORM 的公共字段

例如 GORM 中经常见到：

```go
type User struct {
	gorm.Model
	Username string
}
```

`gorm.Model` 中的 ID、创建时间、更新时间等字段会被嵌入。这种写法很常见，但也会让数据库模型与框架类型绑定得更紧。

#### 让包装类型自动实现接口

```go
type ReaderWithStats struct {
	io.Reader
	BytesRead int64
}
```

因为嵌入了 `io.Reader`，外层类型可以保留读取能力，再添加统计逻辑。

### 实际使用判断

只表达“拥有、关联、属于”：

```go
type User struct {
	Group Group
}
```

希望外层类型主动把内部类型的能力当成自己的 API：

```go
type LoggingWriter struct {
	http.ResponseWriter
}
```

所以可以记成：

> **业务数据关系优先具名字段；能力包装、接口代理、基础设施复用时，匿名嵌入比较常见。**

在你现在写的 `User` 和 `Group` 例子里，建议用：

```go
type User struct {
	Username string
	Email    string
	Group    Group
}
```

因为用户只是“属于某个组”，并不应该表现得像用户自己就是一个组。

## Animal, Cat 这种就适合匿名嵌入了？

**可以，但要看你想表达什么。**

如果只是想复用动物的公共字段和方法，`Cat` 匿名嵌入 `Animal` 很自然：

```go
type Animal struct {
	Name string
	Age  int
}

func (a Animal) Eat() {
	fmt.Println(a.Name, "正在吃东西")
}

type Cat struct {
	Animal
	Color string
}
```

使用：

```go
cat := Cat{
	Animal: Animal{
		Name: "咪咪",
		Age:  3,
	},
	Color: "橘色",
}

fmt.Println(cat.Name) // 相当于 cat.Animal.Name
cat.Eat()             // 相当于 cat.Animal.Eat()
```

这在表面上很像：

```text
Cat 继承 Animal
```

但 Go 中真实结构仍然是：

```text
Cat 包含一个 Animal
```

`Cat` 并不是 `Animal` 的子类型：

```go
func printAnimal(a Animal) {}

cat := Cat{}
printAnimal(cat)        // 错误
printAnimal(cat.Animal) // 正确
```

### 要表达“猫也是一种动物”，接口通常更合适

```go
type Animal interface {
	Speak()
	Eat()
}

type Cat struct {
	Name string
}

func (c Cat) Speak() {
	fmt.Println("喵")
}

func (c Cat) Eat() {
	fmt.Println(c.Name, "正在吃东西")
}
```

现在：

```go
func feed(a Animal) {
	a.Eat()
}

cat := Cat{Name: "咪咪"}
feed(cat)
```

这里才真正表达：

> 任何实现了 `Animal` 能力的类型，都可以被当作 `Animal` 使用。

还可以再加 `Dog`：

```go
type Dog struct {
	Name string
}

func (d Dog) Speak() {
	fmt.Println("汪")
}

func (d Dog) Eat() {
	fmt.Println(d.Name, "正在吃东西")
}
```

然后：

```go
animals := []Animal{
	Cat{Name: "咪咪"},
	Dog{Name: "旺财"},
}

for _, animal := range animals {
	animal.Speak()
}
```

### 两者可以结合

公共数据用结构体嵌入，多态能力用接口：

```go
type AnimalBase struct {
	Name string
	Age  int
}

func (a AnimalBase) Eat() {
	fmt.Println(a.Name, "正在吃东西")
}

type Animal interface {
	Speak()
	Eat()
}

type Cat struct {
	AnimalBase
	Color string
}

func (c Cat) Speak() {
	fmt.Println("喵")
}
```

于是 `Cat`：

- 通过嵌入复用 `Name`、`Age`、`Eat`
- 通过实现接口成为一种 `Animal`

所以更准确的结论是：

> `Animal` 和 `Cat` 可以用匿名嵌入复用公共字段和行为；但要表达真正的“猫是一种动物”和多态，Go 更推荐使用接口。

## 其实就是想组合字段就具名，想组合方法就匿名

可以先这么记，**作为入门判断很实用**：

```go
type User struct {
	Group Group // 主要表达数据关系
}
```

表示：

> User 有一个 Group。

字段和方法都要显式经过 `Group`：

```go
user.Group.Name
user.Group.Validate()
```

而：

```go
type LoggingWriter struct {
	http.ResponseWriter // 希望继承式地使用它的能力
}
```

表示：

> 希望把内部类型的方法提升成外层类型自己的方法。

可以直接：

```go
writer.Write(...)
writer.Header()
```

不过更准确地说：

> **具名字段强调“拥有某个对象”；匿名嵌入强调“把内部类型的能力纳入外层 API”。**

因为匿名嵌入不仅提升方法，也会提升字段：

```go
type Animal struct {
	Name string
}

type Cat struct {
	Animal
}

cat.Name // 也被提升了
```

所以不能严格说匿名只组合方法。它会同时影响：

- 字段访问
- 方法调用
- 外层类型的方法集
- 外层类型实现了哪些接口

因此实践规则可以记成：

```text
业务关系、数据关联、has-a
→ 具名字段

包装类型、代理能力、复用接口实现
→ 匿名嵌入
```

例如：

```go
type User struct {
	Group Group
}
```

适合“用户属于组”。

```go
type LoggingWriter struct {
	http.ResponseWriter
}
```

适合“日志 Writer 保留原 Writer 的全部能力，并增强少数行为”。

还有一点：只是想复用几个方法时，不一定非得匿名嵌入，也可以显式委托：

```go
type Service struct {
	client Client
}

func (s *Service) Send() error {
	return s.client.Send()
}
```

这样虽然多写一点，但外层 API 更可控。匿名嵌入适合你确实愿意让内部方法成为外层公开能力的情况。
