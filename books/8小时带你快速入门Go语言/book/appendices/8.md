# 8

## init

`init` 是 Go 的**包初始化函数**。程序启动时自动执行，不需要也不能手动调用。

```go
package main

import "fmt"

func init() {
	fmt.Println("init")
}

func main() {
	fmt.Println("main")
}
```

输出：

```text
init
main
```

### 执行顺序

大致是：

```text
初始化被导入的包
→ 初始化当前包的全局变量
→ 执行当前包的 init()
→ 执行 main()
```

例如：

```go
package main

import "fmt"

var x = setup()

func setup() int {
	fmt.Println("初始化全局变量")
	return 10
}

func init() {
	fmt.Println("执行 init")
}

func main() {
	fmt.Println("执行 main")
}
```

输出：

```text
初始化全局变量
执行 init
执行 main
```

一个包可以有多个 `init`：

```go
func init() {
	fmt.Println("init 1")
}

func init() {
	fmt.Println("init 2")
}
```

通常按出现顺序执行：

```text
init 1
init 2
```

不过当它们分散在多个文件里时，不建议依赖文件之间的执行顺序。

`init` 有几个限制：

```go
func init() {} // 正确
```

不能有参数：

```go
func init(x int) {} // 错误
```

不能有返回值：

```go
func init() error {} // 错误
```

也不能这样调用：

```go
init() // 错误
```

如果 `init` 里面有 `defer`，它会在**这个 init 函数结束时**执行，而不是等到 `main` 结束：

```go
func init() {
	defer fmt.Println("init defer")
	fmt.Println("init body")
}

func main() {
	fmt.Println("main")
}
```

输出：

```text
init body
init defer
main
```

现在 Go 项目里一般尽量少用复杂的 `init`。它适合做简单的自动注册或不可避免的包初始化；数据库连接、配置读取等重要逻辑通常显式写在 `main` 里，更容易理解和测试。

## 如果有多个 defer，是后写的先执行？

对，多个 `defer` 按照**后进先出**的顺序执行：后写的先执行。

```go
func main() {
	defer fmt.Println("第一个")
	defer fmt.Println("第二个")
	defer fmt.Println("第三个")

	fmt.Println("函数正文")
}
```

输出：

```text
函数正文
第三个
第二个
第一个
```

可以把它想成叠盘子：

```text
最后放上去的，最先拿下来
```

`defer` 后面的函数是在当前函数准备结束时执行：

```go
func test() {
	defer fmt.Println("结束")

	fmt.Println("开始")
	return
}
```

输出：

```text
开始
结束
```

还有一个容易忽略的地方：**`defer` 的参数在写到 `defer` 那一刻就计算了**。

```go
func main() {
	x := 1
	defer fmt.Println(x)

	x = 2
}
```

输出仍然是：

```text
1
```

因为执行到：

```go
defer fmt.Println(x)
```

时，参数 `x` 的值 `1` 已经保存下来了。

## 如果多个包里有 init 呢，是汇总完了先执行 init，还是执行到哪个包运行哪个 init 呢？go 的 import 与 python 一样，是运行到 import 就开始执行了一遍吗？

都不完全是。Go 会先根据 `import` 建立**包依赖图**，然后在 `main()` 开始之前，按照依赖关系逐个初始化包：

> 被依赖的包先完整初始化，再初始化依赖它的包。

每个包内部的顺序是：

```text
包级变量初始化
→ 该包所有 init()
→ 这个包初始化完成
```

不会先把所有包的变量统一初始化，再集中执行所有 `init()`。

例如：

```text
main
├── import a
│   └── import c
└── import b
```

大致顺序是：

```text
c 的全局变量
c.init()

a 的全局变量
a.init()

b 的全局变量
b.init()

main 包的全局变量
main.init()

main()
```

同一个包即使被多个包导入，也只初始化一次；一个 `init()` 返回之后，才会继续执行下一个 `init()`。所有这些初始化都发生在 `main()` 调用之前。

### 和 Python 的区别

Python 的 `import` 是运行时语句：

```python
print("前面")

import mymodule  # 执行到这里时，运行 mymodule 的顶层代码

print("后面")
```

Go 的 `import` 不是运行到那一行才执行，它属于文件顶部的依赖声明：

```go
package main

import "mypackage"

func main() {
	fmt.Println("main")
}
```

你不能把 Go 的 `import` 放进函数或条件语句里：

```go
func main() {
	import "fmt" // 错误
}
```

程序进入 `main()` 时，`fmt` 以及它依赖的包早已初始化完成。Go 规范将 `import` 定义为包之间的依赖关系，并要求导入包先于当前包初始化。

所以可以这样记：

```text
Python：程序执行到 import → 首次加载并执行模块顶层代码
Go：程序启动前按依赖图初始化全部相关包 → 最后执行 main()
```

Go 的模式更像“程序启动阶段统一完成包初始化”，而不是普通代码运行过程中走到哪里初始化到哪里。

## 也就是说对于散落的 init 来说，运行 main 之前这些 init 都会被执行的

对。**只要这些包进入了当前程序的依赖图，它们的 `init()` 都会在 `main()` 开始之前执行完。**

大致顺序：

```text
依赖最深的包
  → 包级变量初始化
  → init()

依赖它的包
  → 包级变量初始化
  → init()

main 包
  → 包级变量初始化
  → init()

最后执行 main()
```

例如：

```go
package main

import (
	_ "example.com/a"
	_ "example.com/b"
)

func main() {
	println("main")
}
```

即使没有直接使用 `a`、`b`，这里用了空白导入 `_`，它们的初始化逻辑仍然会运行：

```text
a.init()
b.init()
main()
```

但要注意，并不是电脑里或项目里所有包的 `init()` 都会运行，只有：

- 当前程序直接导入的包
- 这些包继续依赖的包
- 用 `_` 空白导入的包

才会初始化。

同一个包无论被多少地方导入，都只初始化一次。

还有一个小例外：`init()` 函数本身一定会在 `main()` 前执行完，但如果它启动了 goroutine：

```go
func init() {
	go doSomething()
}
```

这个 goroutine 不一定会在 `main()` 前执行完，它可能和 `main()` 并发运行。
