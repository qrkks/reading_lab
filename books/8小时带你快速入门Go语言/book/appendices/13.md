## 只有一个协程相当于还是同步的？

**只有一个子 goroutine，不等于程序就是同步的。**  
因为还存在主 goroutine，所以实际上至少有两个 goroutine 可以并发执行：

```go
go add() // 子 goroutine

fmt.Println("main") // 主 goroutine 继续执行
```

`go add()` 不会等待 `add()` 完成，主 goroutine 会立刻往下走。

### 为什么有时看起来像同步？

如果启动后马上等待：

```go
go add(done)
<-done
fmt.Println(sum)
```

执行结构相当于：

```text
启动 add
main 立即等待
add 执行完成
main 继续
```

虽然 `add` 仍然运行在另一个 goroutine 中，但主 goroutine 没有趁这段时间做其他事情，所以**最终效果看起来和同步调用差不多**：

```go
add(done)
fmt.Println(sum)
```

甚至这种情况下开 goroutine 没什么收益，只是增加了复杂度。

### 真正体现并发的是 main 同时做别的事

```go
go add(done)

fmt.Println("main 在做其他工作")

<-done
fmt.Println(sum)
```

大致可能是：

```text
子 goroutine 计算 sum
主 goroutine 打印或处理其他任务
两边同时推进
最后 main 等待子 goroutine
```

因此可以这样记：

> 一个子 goroutine 仍然是并发；但如果启动后立刻等待，它在使用效果上近似同步。

你刚才的任务如果只是：

```go
go add()
<-done
```

那确实没必要使用 goroutine，直接：

```go
add()
fmt.Println(sum)
```

更简单。goroutine 的价值在于：**等待它的时候，当前 goroutine 还有其他事情可做，或者需要同时启动多个任务。**

## 如何给程序计时

Go 里最常用的是：

```go
start := time.Now()

// 要计时的代码
doWork()

elapsed := time.Since(start)
fmt.Println("耗时：", elapsed)
```

完整例子：

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	start := time.Now()

	time.Sleep(1500 * time.Millisecond)

	fmt.Println("程序耗时：", time.Since(start))
}
```

输出大约是：

```text
程序耗时：1.5008s
```

### 用 `defer` 给整个函数计时

```go
func doWork() {
	start := time.Now()

	defer func() {
		fmt.Println("doWork 耗时：", time.Since(start))
	}()

	time.Sleep(time.Second)
}
```

`defer` 会在函数退出时执行，因此即使函数中间有多个 `return`，也能打印时间：

```go
func query() error {
	start := time.Now()
	defer func() {
		fmt.Println("query 耗时：", time.Since(start))
	}()

	if somethingWrong {
		return errors.New("出错了")
	}

	return nil
}
```

还可以抽成工具函数：

```go
func trackTime(name string) func() {
	start := time.Now()

	return func() {
		fmt.Printf("%s 耗时：%v\n", name, time.Since(start))
	}
}
```

使用：

```go
func doWork() {
	defer trackTime("doWork")()

	time.Sleep(time.Second)
}
```

这里看起来有两对括号：

```go
defer trackTime("doWork")()
```

执行过程是：

```go
finish := trackTime("doWork")
defer finish()
```

第一个调用立即记录开始时间，返回的函数在 `doWork` 退出时执行。

### 给你刚才的 goroutine 程序计时

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	start := time.Now()

	workCh := make(chan bool, 1)
	done := make(chan struct{})

	go func() {
		time.Sleep(1 * time.Second)
		workCh <- true
		done <- struct{}{}
	}()

	select {
	case result := <-workCh:
		fmt.Println("工作完成了，结果：", result)

	case <-time.After(2 * time.Second):
		fmt.Println("工作超时了")
	}

	<-done

	fmt.Println("程序结束")
	fmt.Println("程序总耗时：", time.Since(start))
}
```

这里测量的是从 `main` 开始计时，到接收到 `done` 为止的总时间。

需要更精确地测试函数性能时，一般不用手动计时，而是使用 Go 的 benchmark：

```go
func BenchmarkDoWork(b *testing.B) {
	for i := 0; i < b.N; i++ {
		doWork()
	}
}
```

然后运行：

```bash
go test -bench=.
```

普通学习和观察运行时间，用 `time.Now()` 加 `time.Since()` 就够了。

## channel 有点像个临时容器

对，**可以先把 channel 理解成“协程之间的临时传递容器”**，这个直觉没问题。

但它和普通容器不一样，关键在于它还负责**同步**。

普通容器：

```go
slice := []int{1, 2, 3}
```

主要作用是存数据，你可以随时按索引查看：

```go
fmt.Println(slice[0])
```

channel：

```go
ch := make(chan int, 1)
ch <- 10
```

主要作用是：

> 一个 goroutine 把值交给另一个 goroutine。

接收：

```go
v := <-ch
```

这个值被取走后，就不在 channel 里了。

所以它更像：

> 临时中转站 + 等待机制。

容量为 1 的缓冲通道确实很像一个只能放一个东西的临时盒子：

```go
ch := make(chan int, 1)

ch <- 10  // 放进去
v := <-ch // 拿出来
```

但是无缓冲通道：

```go
ch := make(chan int)
```

连“盒子”都没有，更像两个人当面交接：

```text
发送者拿着数据等待
        ↓
接收者伸手接
        ↓
交接完成，两边继续
```

因此更准确的分类是：

```text
无缓冲 channel：交接点
有缓冲 channel：有限容量的消息队列
```

它不太适合作为普通数据容器，因为：

- 不能查看其中某个元素；
- 不能偷看而不消费；
- 不能按索引访问；
- 接收顺序通常是先进先出；
- 发送和接收可能阻塞；
- 关闭后不能再发送。

所以可以记成：

> **channel 看起来像临时容器，本质上是带同步能力的消息传递队列。**

## 在哪里加锁？

要保护的是共享变量 `sum`，所以锁要围住对 `sum` 的读写。

最直接的写法：

```go
var (
	sum int
	wg  sync.WaitGroup
	mu  sync.Mutex
)

func add() {
	defer wg.Done()

	for i := 0; i < 10_000_000; i++ {
		mu.Lock()
		sum++
		mu.Unlock()
	}
}
```

完整结构：

```go
import (
	"fmt"
	"sync"
	"time"
)

var (
	sum int
	wg  sync.WaitGroup
	mu  sync.Mutex
)

func add() {
	defer wg.Done()

	for i := 0; i < 10_000_000; i++ {
		mu.Lock()
		sum++
		mu.Unlock()
	}
}

start := time.Now()

wg.Add(2)

go add()
go add()

wg.Wait()

fmt.Println(sum)
fmt.Println(time.Since(start))
```

这样结果会稳定是：

```text
20000000
```

不过每次 `sum++` 都加锁、解锁，非常慢。

更合理的做法是每个 goroutine 先计算自己的局部结果，最后只加锁一次：

```go
func add() {
	defer wg.Done()

	localSum := 0

	for i := 0; i < 10_000_000; i++ {
		localSum++
	}

	mu.Lock()
	sum += localSum
	mu.Unlock()
}
```

这里的逻辑是：

```text
localSum
```

只属于当前 goroutine，不需要锁。

只有：

```go
sum += localSum
```

会修改共享变量，因此只锁这一小段。

一般遵循：

> 锁住共享数据的最小必要范围，不要把不需要保护的计算也锁进去。

另外，最后：

```go
wg.Wait()
fmt.Println(sum)
```

这里打印 `sum` 不需要再加锁，因为 `wg.Wait()` 已经确保两个 goroutine 都完成了，不会再有人修改 `sum`。

## 加锁之后特别慢，跑了 30 多秒

对，正常。你现在相当于执行了大约 **2000 万次加锁和解锁**：

```go
for i := 0; i < 10_000_000; i++ {
	mu.Lock()
	sum++
	mu.Unlock()
}
```

两个 goroutine 一共就是：

```text
2000 万次 Lock
2000 万次 Unlock
```

而且两个 goroutine 还在不断争抢同一把锁，实际效果接近：

> 你加一次，我加一次，谁也不能真正并行。

所以不仅没有并发收益，还额外付出了：

- 锁竞争
- goroutine 调度
- CPU 缓存同步
- 互斥锁本身的开销

### 正确做法：局部计算，最后只锁一次

```go
var (
	sum int
	wg  sync.WaitGroup
	mu  sync.Mutex
)

func add() {
	defer wg.Done()

	localSum := 0

	for i := 0; i < 10_000_000; i++ {
		localSum++
	}

	mu.Lock()
	sum += localSum
	mu.Unlock()
}
```

完整版本：

```go
import (
	"fmt"
	"sync"
	"time"
)

var (
	sum int
	wg  sync.WaitGroup
	mu  sync.Mutex
)

func add() {
	defer wg.Done()

	localSum := 0

	for i := 0; i < 10_000_000; i++ {
		localSum++
	}

	mu.Lock()
	sum += localSum
	mu.Unlock()
}

start := time.Now()

wg.Add(2)

go add()
go add()

wg.Wait()

fmt.Println(sum)
fmt.Println(time.Since(start))
```

现在每个 goroutine 只会：

```text
独立计算 1000 万次
最后争抢一次锁
```

总共只有两次加锁，速度会快很多。

### 甚至可以完全不用锁

让每个 goroutine 把结果发到 channel：

```go
func add(resultCh chan<- int) {
	localSum := 0

	for i := 0; i < 10_000_000; i++ {
		localSum++
	}

	resultCh <- localSum
}

resultCh := make(chan int, 2)

go add(resultCh)
go add(resultCh)

sum := <-resultCh + <-resultCh

fmt.Println(sum)
```

这里没有共享写入：

```text
goroutine 1 → 自己的 localSum
goroutine 2 → 自己的 localSum
main        → 汇总两个结果
```

因此不需要锁。

你这次实验正好说明：

> **并发不是把同一个变量交给多个 goroutine 抢着修改。**

更高效的并发通常是：

> 各自独立计算，最后再汇总结果。

锁应该保护很短的共享操作，而不是包住高频循环中的每一次修改。

#### resultCh := make(chan int, 2) 这里用 2 就是正好装两个 add 里的结果？

对，可以这么理解。

```go
resultCh := make(chan int, 2)
```

这里的 `2` 表示：

> 这个 channel 的缓冲区最多可以先存 **2 个 `int` 值**，而不用立刻被接收。

因为你启动了两个 `add`：

```go
go add(resultCh)
go add(resultCh)
```

而每个 `add` 最后只发送一次结果：

```go
resultCh <- localSum
```

所以总共正好会发送 **2 个结果**。

因此容量写成 `2` 的效果就是：

> 两个 goroutine 都可以把结果放进去，然后结束；  
> 即使 main 还没来得及接收，也不会阻塞。

---

##### 为什么这里 `2` 很合适？

因为发送总次数就是 2 次：

```text
add 1 -> 发送 1 次
add 2 -> 发送 1 次
总共 -> 2 次
```

而缓冲区容量也是 2，所以：

```go
resultCh := make(chan int, 2)
```

就能刚好装下两个结果。

---

##### 如果写成 1 呢？

也能跑，只是行为不同：

```go
resultCh := make(chan int, 1)
```

这时：

- 第一个 goroutine 发送成功，结果先放进缓冲区；
- 第二个 goroutine 再发送时，如果 main 还没接收，就会阻塞等待。

也就是说：

> 容量 1 不是错，只是第二个发送可能要等等。

---

##### 如果写成 0 呢？

也就是无缓冲：

```go
resultCh := make(chan int)
```

这时每次发送都必须等接收方“当面接手”。

所以两个 goroutine 在执行：

```go
resultCh <- localSum
```

时，都可能阻塞，直到 main 执行：

```go
a := <-resultCh
b := <-resultCh
```

---

##### 所以这里的 2 不是“必须”，而是“正好合适”

你可以这样理解：

- `0`：不存货，必须当面交接
- `1`：能先放 1 个结果
- `2`：正好能把两个 `add` 的结果都先装进去

所以你的理解：

> “这里用 2 就是正好装两个 add 里的结果？”

**对，完全对。**

---

##### 顺便补一句

虽然这里写 `2` 很自然，但并不是说：

> channel 容量必须等于 goroutine 数量

只是这个例子里：

- 有 2 个生产者
- 每个生产者发送 1 次
- 所以容量 2 很顺手

如果后面你是边生产边消费，容量就不一定要和任务数相等了。
