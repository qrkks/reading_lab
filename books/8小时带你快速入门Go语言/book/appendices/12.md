## 有点像引用计数啊。add 是加，done 是减，wait 是清 0 了就继续后面的。

对，这个类比很直观：

```text
Add(1)  → 待完成任务数 +1
Done()  → 待完成任务数 -1
Wait()  → 阻塞，等计数变成 0 后继续
```

而且：

```go
wg.Done()
```

本质上就等价于：

```go
wg.Add(-1)
```

例如：

```go
var wg sync.WaitGroup

wg.Add(3)

go func() {
	defer wg.Done()
	// 任务 1
}()

go func() {
	defer wg.Done()
	// 任务 2
}()

go func() {
	defer wg.Done()
	// 任务 3
}()

wg.Wait()
fmt.Println("三个任务都完成了")
```

可以想象计数变化：

```text
初始：0
Add(3)：3
任务完成：2
任务完成：1
任务完成：0
Wait 解除阻塞
```

不过它和引用计数只是在“加减计数”这一点上相似。

引用计数通常是在统计：

> 还有多少地方持有这个对象。

计数归零后，可能释放对象。

`WaitGroup` 统计的是：

> 还有多少项任务没有完成。

归零后只是唤醒正在 `Wait()` 的 goroutine，不涉及内存释放。

还有两条重要规则：

```go
wg.Add(3) // 一般要在启动 goroutine 之前完成
```

不要让计数减成负数，否则会 panic：

```go
wg.Add(1)
wg.Done()
wg.Done() // panic：negative WaitGroup counter
```

所以可以把 `WaitGroup` 理解成一个非常简单的**并发任务倒计时器**。

## 传 wg 必须传指针

对，**传给函数时通常必须传指针**：

```go
func shopping(name string, wg *sync.WaitGroup) {
	defer wg.Done()
	// ...
}
```

调用：

```go
go shopping("张三", &wg)
```

原因是所有 goroutine 必须操作**同一个 `WaitGroup` 计数器**。

如果按值传递：

```go
func shopping(name string, wg sync.WaitGroup) {
	defer wg.Done()
}
```

每次调用都会复制一份 `wg`。`Done()` 减少的是副本里的计数，而 `main` 中原来的计数没有变化：

```text
main 中的 wg：3，一直不变
shopping 的副本：3 → 2
```

于是：

```go
wg.Wait()
```

可能永远等不到零。

而传指针时：

```text
main ─┐
张三 ─┼→ 同一个 WaitGroup
李四 ─┤
王五 ─┘
```

每个 `Done()` 都减少同一个计数：

```text
3 → 2 → 1 → 0
```

另外，`sync.WaitGroup` 使用后不应该复制。Go 里的同步类型，如这些，通常也都通过指针使用：

```go
*sync.WaitGroup
*sync.Mutex
*sync.RWMutex
```

不过也可以不把 `wg` 作为参数，而让匿名函数直接使用外层变量：

```go
var wg sync.WaitGroup

wg.Add(1)

go func() {
	defer wg.Done()
	fmt.Println("购物")
}()

wg.Wait()
```

这里虽然没显式传 `&wg`，闭包使用的仍然是外面的同一个 `wg`，并没有复制。

## 这协程有点方便啊

是，Go 的 goroutine 确实非常方便，最爽的地方就是：

```go
go shopping("张三", &wg)
```

只加一个 `go`，普通函数调用就变成并发执行了。

这比很多语言都轻：

- 不用手动创建线程对象
- 不用配置线程池才能开始
- goroutine 本身很轻量
- Go 运行时负责把大量 goroutine 调度到少量系统线程上

所以特别适合后端里这种任务：

```go
go sendEmail()
go writeLog()
go fetchRemoteData()
go processFile()
```

不过也正因为启动太容易，容易让人产生一种错觉：

> 加个 `go` 就万事大吉了。

实际上真正麻烦的是后面的管理：

```text
什么时候结束？
怎么拿返回值？
怎么传错误？
怎么取消？
会不会同时修改同一个变量？
会不会泄漏 goroutine？
```

于是 Go 又配套了这些东西：

```text
WaitGroup  等待完成
channel    传值、传结果
Mutex      保护共享数据
context    取消、超时
select     同时等待多个 channel
```

所以 goroutine 本身很简单：

```go
go f()
```

并发控制才是后面的重点。

但这个设计特别符合 Go 的风格：

> 先让简单场景极其简单，复杂场景再逐步加入控制工具。

这也是 Go 写网络服务很舒服的重要原因。每来一个请求，底层就可以由独立 goroutine 处理，你写业务代码时通常不用直接操心线程。

## 只要数据需要一致的是不是就得传指针

不完全是。

更准确地说：

> **需要操作同一份数据时，通常要共享同一个对象；指针是共享对象的一种方式，但数据一致性还需要同步机制。**

例如：

```go
func add(count *int) {
	*count++
}
```

这里传指针，多个地方确实能修改同一个 `count`。

但如果多个 goroutine 同时修改：

```go
count := 0

for range 1000 {
	go func() {
		count++
	}()
}
```

即使大家操作的是同一个变量，也会发生数据竞争，结果可能不正确。

要加锁：

```go
var (
	count int
	mu    sync.Mutex
	wg    sync.WaitGroup
)

for range 1000 {
	wg.Add(1)

	go func() {
		defer wg.Done()

		mu.Lock()
		count++
		mu.Unlock()
	}()
}

wg.Wait()
```

所以是两层问题：

```text
指针：让大家访问同一份数据
锁 / channel：保证大家访问这份数据时不打架
```

`WaitGroup` 必须传指针，是因为所有 goroutine 必须修改同一个任务计数器：

```go
func work(wg *sync.WaitGroup) {
	defer wg.Done()
}
```

但指针并不是唯一的共享方式。`map`、`slice`、`channel` 本身就带有引用语义：

```go
func add(m map[string]int) {
	m["a"] = 1
}
```

这里不用 `*map`，调用者仍然能看到修改。

不过 `map` 在多个 goroutine 里同时读写，同样需要锁：

```go
type SafeMap struct {
	mu   sync.Mutex
	data map[string]int
}
```

另外，有时为了避免共享数据，反而会把值复制给各个 goroutine：

```go
for _, name := range names {
	go func(name string) {
		fmt.Println(name)
	}(name)
}
```

每个 goroutine 拿自己的 `name`，就不需要同步。

所以可以这样记：

```text
需要修改调用者的普通值
→ 通常传指针

多个 goroutine 共享修改同一数据
→ 指针/引用 + Mutex、原子操作或 channel

不需要共享
→ 尽量传值副本，反而更安全
```

**指针解决“是不是同一份数据”，同步机制解决“同时修改会不会乱”。**

## 感觉同步代码在写异步工作

对，Go 很大一个魅力就是：

> **用同步、顺序的代码外观，写并发执行的工作。**

例如：

```go
func shopping(name string) {
	fmt.Println(name, "开始购物")
	time.Sleep(time.Second)
	fmt.Println(name, "购物结束")
}
```

这个函数内部完全是同步的：

```text
打印开始
→ 等待 1 秒
→ 打印结束
```

但只要调用时加上：

```go
go shopping("张三")
go shopping("李四")
go shopping("王五")
```

三个同步函数就会并发运行。

所以 Go 不是要求你把函数内部写成各种回调或 `async/await`，而是：

```text
每个 goroutine 内部：按同步思维写
多个 goroutine 之间：并发执行
```

这和 JavaScript 很不一样。

JavaScript 通常是：

```js
async function shopping(name) {
  console.log(name, "开始购物");
  await sleep(1000);
  console.log(name, "购物结束");
}
```

你需要一直意识到：

```text
这个函数是 async
这里需要 await
返回的是 Promise
```

Go 里：

```go
func shopping(name string) {
	time.Sleep(time.Second)
}
```

仍然就是普通函数。是否并发，取决于调用方式：

```go
shopping("张三")    // 当前 goroutine 同步执行
go shopping("张三") // 新 goroutine 并发执行
```

甚至 channel 的读取看起来也像普通同步代码：

```go
result := <-resultCh
fmt.Println(result)
```

如果结果还没来，当前 goroutine 就停在这里。但不是整个程序停住，Go 运行时会去执行其他 goroutine。

可以想成：

```text
当前 goroutine 阻塞
        ↓
Go 调度器把线程拿去执行其他 goroutine
        ↓
数据到了，再回来继续执行
```

因此 Go 的并发代码读起来经常像普通顺序代码：

```go
result := request()
save(result)
sendNotification()
```

只是这整段代码可能运行在一个 goroutine 里。

这也是 Go 很适合后端的原因之一。处理一个 HTTP 请求时，你可以像写同步代码一样：

```go
func handler(w http.ResponseWriter, r *http.Request) {
	user, err := findUser(r.Context())
	if err != nil {
		// 处理错误
		return
	}

	data, err := loadData(user)
	if err != nil {
		return
	}

	writeResponse(w, data)
}
```

但服务器底层可以让许多这样的 handler 同时在不同 goroutine 中运行。

所以你这个感觉很准确：

> Go 把“异步工作”包装成了“多个同步流程并发运行”。

真正需要额外处理的主要是它们之间的协调：

```text
WaitGroup：等大家完成
channel：互相传数据
Mutex：避免同时改乱共享数据
context：取消和超时
```

而每个单独任务内部，仍然可以按照最自然的同步逻辑来写。

## chan 需要 import 吗？

不需要。`chan` 是 Go 的内置关键字，和 `map`、`struct`、`func` 一样，不属于某个包。

```go
ch := make(chan int)
```

这里：

```go
chan int
```

表示“传递 `int` 的 channel”。

发送和接收：

```go
ch <- 10      // 发送
value := <-ch // 接收
```

关闭 channel 也不需要导包：

```go
close(ch)
```

完整例子只因为要打印，所以才导入 `fmt`：

```go
import "fmt"

ch := make(chan int)

go func() {
	ch <- 10
}()

value := <-ch
fmt.Println(value)
```

还可以限制方向：

```go
chan int   // 可发送、可接收
chan<- int // 只能发送
<-chan int // 只能接收
```

所以 `chan` 本身不用 `import`；你在程序里还用了什么包的功能，才导入什么包。

## Channel

你这段代码会死锁，因为 `moneyChan` 是**无缓冲 channel**：

```go
moneyChan := make(chan int)
```

发送：

```go
moneyChan <- money
```

必须同时有另一个 goroutine 接收。但此时 `main` 正卡在：

```go
wg.Wait()
```

于是形成：

```text
pay 等待 main 接收 money
main 等待 pay 执行 Done
```

而 `Done()` 是 `defer`，只有 `pay` 返回时才执行；`pay` 又卡在发送上，所以谁也走不了。

推荐这样改：主 goroutine 一边接收金额，另一个 goroutine 等待任务完成后关闭 channel。

```go
import (
	"fmt"
	"sync"
	"time"
)

func pay(
	name string,
	money int,
	moneyChan chan<- int,
	wg *sync.WaitGroup,
) {
	defer wg.Done()

	fmt.Printf("%s 开始购物\n", name)
	time.Sleep(time.Second)
	fmt.Printf("%s 购物结束，消费 %d 元\n", name, money)

	moneyChan <- money
}
func main() {
	var wg sync.WaitGroup
	moneyChan := make(chan int)

	wg.Add(3)

	startTime := time.Now()

	go pay("张三", 2, moneyChan, &wg)
	go pay("李四", 3, moneyChan, &wg)
	go pay("王五", 5, moneyChan, &wg)

	// 等所有付款任务完成后关闭 channel
	go func() {
		wg.Wait()
		close(moneyChan)
	}()

	total := 0

	// 不断接收，直到 channel 被关闭
	for money := range moneyChan {
		total += money
	}

	fmt.Printf(
		"购买完成，总计 %d 元，耗时 %v\n",
		total,
		time.Since(startTime),
	)
}

main()
```

输出顺序不固定，但结果大致是：

```text
张三 开始购物
李四 开始购物
王五 开始购物
张三 购物结束，消费 2 元
李四 购物结束，消费 3 元
王五 购物结束，消费 5 元
购买完成，总计 10 元，耗时约 1 秒
```

这里的关键结构是：

```text
3 个 pay goroutine → 向 moneyChan 发送金额
main              → 从 moneyChan 接收并累加
等待协程           → wg 归零后关闭 moneyChan
```

参数里的：

```go
moneyChan chan<- int
```

表示这个函数只能向 channel 发送数据，不能从中接收。这样比直接写 `chan int` 更能说明 `pay` 的职责。

### `moneyChan chan<- int`, 这是啥意思

`moneyChan chan<- int` 表示：

> 参数 `moneyChan` 是一个**只能发送 `int` 的 channel**。

拆开看：

```go
moneyChan chan<- int
```

- `moneyChan`：参数名
- `chan<- int`：只能发送 `int` 的 channel 类型
- 箭头指向 `chan` 外面，可以理解为“数据进入 channel”

所以函数里可以写：

```go
moneyChan <- money
```

但不能接收：

```go
money := <-moneyChan // 编译错误：只能发送
```

三种 channel 类型对比：

```go
chan int   // 双向：能发送，也能接收
chan<- int // 只发送
<-chan int // 只接收
```

例如：

```go
func send(ch chan<- int) {
	ch <- 10
}

func receive(ch <-chan int) {
	value := <-ch
	fmt.Println(value)
}
```

使用时可以把一个普通的双向 channel 传进去：

```go
ch := make(chan int)

go send(ch)
receive(ch)
```

这里 `ch` 本身仍然是：

```go
chan int
```

只是传入 `send` 后，函数只能把它当成 `chan<- int` 使用；传入 `receive` 后，只能当成 `<-chan int` 使用。

你的代码里：

```go
func pay(
	name string,
	money int,
	moneyChan chan<- int,
	wg *sync.WaitGroup,
)
```

是在明确表达：

> `pay` 的职责只是把消费金额发送出去，不允许它偷着从 channel 里读取数据。

这是编译期限制，让函数职责更清楚、更安全。箭头可以这样记：

```go
ch <- value // 值发送进 ch
chan<- int  // 只能发送

value := <-ch // 值从 ch 出来
<-chan int    // 只能接收
```

### channel 在 go 里面算难的吗？

算是 **Go 里入门容易、真正用好比较难** 的部分。

最基础的 channel 很简单：

```go
ch := make(chan int)

go func() {
	ch <- 10
}()

value := <-ch
```

核心就三件事：

```go
ch <- value   // 发送
value := <-ch // 接收
close(ch)     // 关闭
```

你刚才已经碰到了 channel 最关键的地方：**发送和接收会不会互相等待**。

无缓冲 channel：

```go
ch := make(chan int)
```

发送方和接收方必须碰头：

```text
发送方等接收方
接收方也可能等发送方
```

因此很容易写出死锁。你刚才的代码就是：

```text
pay 等 main 接收
main 等 pay Done
```

双方互相等。

带缓冲 channel：

```go
ch := make(chan int, 3)
```

可以先暂存 3 个值：

```go
ch <- 1
ch <- 2
ch <- 3
```

第四次发送才会等待有人接收。

真正开始变难的是这些问题：

- 谁负责关闭 channel？
- 发送方会不会永远等不到接收方？
- 接收方会不会永远等不到数据？
- goroutine 提前退出后，其他 goroutine 会不会泄漏？
- 应该用 channel，还是用 `Mutex`？
- 多个 channel 同时有数据时怎么处理？

后面还会遇到：

```go
select {
case value := <-resultCh:
	fmt.Println(value)

case err := <-errCh:
	fmt.Println(err)

case <-time.After(time.Second):
	fmt.Println("超时")
}
```

`select` 会同时等待多个 channel，这时并发结构就开始复杂了。

不过你不需要一开始掌握所有高级模式。先掌握下面这条主线就够了：

```text
goroutine 负责做事
channel 负责传递结果
WaitGroup 负责等待结束
context 负责取消和超时
Mutex 负责保护共享数据
```

还有一个很实用的判断：

```text
只是等待任务结束
→ WaitGroup

需要把结果传回来
→ channel

多个 goroutine 修改同一个变量
→ Mutex 或重新设计成通过 channel 传递

需要超时或取消
→ context
```

所以 channel 本身语法不难，难的是**设计谁等谁、谁发谁收，以及什么时候结束**。你刚才已经自己运行并看懂了一个“多个生产者把结果传给主流程”的完整例子，实际上已经迈过最容易卡人的第一道坎了。

### channel, wg 各管各的，互不影响？

对，**`channel` 和 `WaitGroup` 是两个独立工具，各管各的，不会自动互相影响。**

```text
WaitGroup：还有多少任务没结束
channel：goroutine 之间如何传递数据
```

### `WaitGroup` 不管数据

```go
wg.Add(3)
wg.Done()
wg.Wait()
```

它只维护一个计数器：

```text
3 → 2 → 1 → 0
```

它不知道 goroutine 发了多少钱，也不知道有没有 channel。

### channel 不管任务数量

```go
moneyChan <- money
money := <-moneyChan
```

它只负责发送和接收数据。它不知道总共有三个购物任务，也不会自动调用 `wg.Done()`。

不过，我们可以通过代码把它们**组织在一起**：

```go
go func() {
	wg.Wait()
	close(moneyChan)
}()
```

意思是：

> 等 `WaitGroup` 确认所有付款任务完成后，再人为关闭 channel。

这不是它们自动联动，而是你写代码建立了关系。

你当前程序的流程是：

```text
pay goroutine
    ↓
向 channel 发送金额
    ↓
函数结束，defer wg.Done()

另一个 goroutine
    ↓
wg.Wait() 等计数归零
    ↓
close(moneyChan)

main goroutine
    ↓
持续从 channel 收金额
    ↓
channel 关闭后结束 range
```

它们虽然各管各的，但执行顺序会互相牵制。例如无缓冲 channel 中：

```go
moneyChan <- money
```

必须有人接收，`pay` 才能继续执行到函数结束，随后 `defer wg.Done()` 才会运行。因此如果主流程先：

```go
wg.Wait()
```

却不接收 channel，就会互相等待而死锁。

所以准确概括是：

> **机制上互相独立，程序流程中可以通过阻塞关系和你写的代码相互协调。**

有时只需要其中一个：

```go
// 只关心任务做完，不需要结果
WaitGroup
```

```go
// 需要接收固定数量的结果
channel
```

需要“收集未知数量的结果，等全部生产者结束后关闭 channel”时，二者配合就很自然。

## for 有哪些语法，怎么还有 for 后面直接花括号的

Go 只有一个循环关键字 `for`，但有几种写法。

#### 1. 经典三段式

```go
for i := 0; i < 3; i++ {
	fmt.Println(i)
}
```

三部分分别是：

```text
初始化；继续条件；每轮结束后的操作
```

相当于很多语言里的 `for`。

---

#### 2. 只有条件，类似 `while`

```go
i := 0

for i < 3 {
	fmt.Println(i)
	i++
}
```

Go 没有 `while`，就用这种写法代替。

---

#### 3. `for {}`：无限循环

```go
for {
	fmt.Println("一直运行")
}
```

因为没有写结束条件，所以默认一直循环。

通常配合 `break`：

```go
i := 0

for {
	if i >= 3 {
		break
	}

	fmt.Println(i)
	i++
}
```

`for {}` 常用于：

- 服务器持续监听请求
- 持续读取消息
- 后台工作 goroutine
- 不确定何时结束的循环

例如：

```go
for {
	conn, err := listener.Accept()
	if err != nil {
		continue
	}

	go handle(conn)
}
```

---

#### 4. `range` 遍历集合

遍历切片：

```go
nums := []int{10, 20, 30}

for index, value := range nums {
	fmt.Println(index, value)
}
```

只需要值：

```go
for _, value := range nums {
	fmt.Println(value)
}
```

只需要下标：

```go
for index := range nums {
	fmt.Println(index)
}
```

也能遍历 `map`：

```go
for key, value := range ages {
	fmt.Println(key, value)
}
```

---

#### 5. 遍历 channel

你刚才的代码中：

```go
for money := range moneyChan {
	total += money
}
```

意思是：

> 不断从 `moneyChan` 接收数据，直到 channel 被关闭。

大致相当于：

```go
for {
	money, ok := <-moneyChan
	if !ok {
		break
	}

	total += money
}
```

所以必须有人最终调用：

```go
close(moneyChan)
```

否则这个 `for range` 会一直等待。

---

#### 6. 重复固定次数

现在也可以写：

```go
for range 3 {
	fmt.Println("执行一次")
}
```

会执行三次。

需要序号时：

```go
for i := range 3 {
	fmt.Println(i)
}
```

输出：

```text
0
1
2
```

所以最常见的几种可以记成：

```go
for i := 0; i < n; i++ {} // 经典循环
for condition {}          // 类似 while
for {}                    // 无限循环
for _, value := range xs {} // 遍历集合
for value := range ch {}    // 持续读取 channel
```

`for {}` 并不是特殊函数，而是把初始化、条件和更新全部省略后形成的无限循环。

### 为什么无限循环，没有 while 吗？

Go 没有 `while`。它把所有循环统一成了一个关键字：`for`。

很多语言里的：

```python
while condition:
    ...
```

在 Go 里就是：

```go
for condition {
	// ...
}
```

例如：

```go
i := 0

for i < 3 {
	fmt.Println(i)
	i++
}
```

这其实就是 Go 版的 `while`。

而：

```go
for {
	// ...
}
```

相当于其他语言里的：

```python
while True:
    ...
```

因为没有条件限制，所以会一直循环，通常用 `break` 退出：

```go
i := 0

for {
	if i >= 3 {
		break
	}

	fmt.Println(i)
	i++
}
```

Go 这么设计主要是为了减少语法：经典 `for`、`while`、无限循环、集合遍历，全都统一使用 `for`。

```go
for i := 0; i < 3; i++ {} // 普通 for
for i < 3 {}               // 相当于 while
for {}                     // 无限循环
for _, v := range values {} // 遍历
```

所以不是 Go 没有 `while` 的能力，而是把 `while` 合并进 `for` 了。

## 协程有关的有几块内容？

Go 里和 goroutine 有关的内容，大致可以分成 **7 块**。你现在已经学到前三块了。

### 1. 启动 goroutine

```go
go work()
```

理解：

- 普通调用 `work()`：当前流程等它完成
- 并发调用 `go work()`：新建 goroutine 执行
- `main` 结束后，其他 goroutine 也会直接结束

---

### 2. 等待任务完成：`WaitGroup`

```go
var wg sync.WaitGroup

wg.Add(1)

go func() {
	defer wg.Done()
	work()
}()

wg.Wait()
```

它是任务倒计时器：

```text
Add   增加待完成任务
Done  完成一个任务
Wait  等待计数归零
```

你现在已经基本掌握了。

---

### 3. goroutine 之间传数据：channel

```go
ch := make(chan int)

go func() {
	ch <- 10
}()

value := <-ch
```

包括：

```go
chan int    // 双向
chan<- int  // 只发送
<-chan int  // 只接收
```

以及：

```go
close(ch)

for value := range ch {
	fmt.Println(value)
}
```

你目前正在学这一块。

---

### 4. 无缓冲与有缓冲 channel

无缓冲：

```go
ch := make(chan int)
```

发送方和接收方必须碰头，类似当面交货。

有缓冲：

```go
ch := make(chan int, 3)
```

可以先在 channel 里暂存 3 个值，类似快递柜。

这一块重点是理解：

- 什么时候发送会阻塞
- 什么时候接收会阻塞
- 为什么会产生死锁

---

### 5. 同时等待多个事件：`select`

```go
select {
case value := <-resultCh:
	fmt.Println("收到结果：", value)

case err := <-errCh:
	fmt.Println("发生错误：", err)

case <-time.After(time.Second):
	fmt.Println("超时")
}
```

`select` 很像专门用于 channel 的 `switch`：

> 哪个 channel 先准备好，就执行哪个分支。

它常用于结果、错误、超时、取消等多路协调。

---

### 6. 共享数据同步：锁和原子操作

多个 goroutine 同时改一份数据时，channel 不一定是最佳方案，有时直接加锁：

```go
var mu sync.Mutex

mu.Lock()
count++
mu.Unlock()
```

相关内容包括：

```go
sync.Mutex
sync.RWMutex
sync.Once
sync.Map
atomic
```

这里要理解：

> 指针负责共享同一份数据，锁负责防止同时修改造成混乱。

---

### 7. 取消、超时与并发模式

后端实际项目里非常重要的是 `context`：

```go
ctx, cancel := context.WithTimeout(
	context.Background(),
	time.Second,
)
defer cancel()
```

goroutine 监听取消：

```go
select {
case <-ctx.Done():
	return
case result := <-resultCh:
	fmt.Println(result)
}
```

然后还有一些常用模式：

```text
worker pool     固定数量的工作协程
fan-out         把任务分给多个 goroutine
fan-in          汇总多个 goroutine 的结果
semaphore       限制最大并发量
errgroup        等待多个任务并收集错误
pipeline        多阶段流水线
```

学习顺序可以这样排：

```text
go
→ WaitGroup
→ channel
→ 缓冲与关闭
→ select
→ Mutex
→ context
→ 常见并发模式
→ 数据竞争和 goroutine 泄漏
```

你现在已经掌握了：

```text
go + WaitGroup + channel 基础
```

下一步最适合继续弄清楚的是：

```text
无缓冲 channel 为什么阻塞
有缓冲 channel 又改变了什么
close 和 range 为什么要配合
```

把这三件事弄明白，Go 并发的骨架基本就立起来了。

## go 并发是 go 的核心吗？

是，**并发确实是 Go 的核心之一**，而且不是后来加上的库功能，而是从语言、运行时到标准库一起设计进去的。

最明显的是这几个直接进入语言语法的东西：

```go
go work()             // 启动 goroutine
ch := make(chan int)  // channel
ch <- value           // 发送
value := <-ch         // 接收

select {
case value := <-ch:
	// ...
}
```

很多语言把并发放在框架或库里，Go 则直接让你用关键字表达。

### 为什么说它是核心

#### 1. 普通函数随时可以并发运行

```go
work()    // 当前 goroutine 执行
go work() // 新 goroutine 执行
```

函数本身不用声明成 `async`，也不用改变返回类型。

#### 2. 运行时专门负责调度

Go 运行时会把大量 goroutine 调度到较少的系统线程上：

```text
大量 goroutine
      ↓
Go 调度器
      ↓
少量系统线程
      ↓
CPU
```

某个 goroutine 等待网络或 channel 时，调度器可以先运行其他 goroutine。

#### 3. 标准库也围绕并发设计

例如 HTTP 服务器收到多个请求时，通常会并发处理每个请求。你写的 handler 仍然是普通同步代码：

```go
func handler(w http.ResponseWriter, r *http.Request) {
	user := loadUser()
	data := loadData(user)
	writeResponse(w, data)
}
```

但许多请求可以同时运行。

#### 4. 有完整的协调工具

```text
goroutine     负责并发执行
channel       负责传递数据
select        等待多个事件
WaitGroup     等待任务结束
Mutex         保护共享数据
context       取消和超时
```

这不是一个孤立的 `go` 关键字，而是一整套并发模型。

### 但 Go 不只是并发语言

Go 的其他核心设计同样重要：

- 简单、统一的语法
- 结构体和组合
- 小接口与隐式实现
- 显式错误处理
- 快速编译
- 单二进制部署
- 垃圾回收
- 强大的标准库与工具链

所以更准确地说：

> **并发是 Go 最有辨识度的核心能力之一；简单工程化则是 Go 更大的整体核心。**

还有一点要区分：

```text
并发：多个任务交错推进
并行：多个任务在多个 CPU 核上同时执行
```

goroutine 首先让并发很容易；是否真正并行，还取决于 CPU 核数、任务类型和调度情况。

你刚才会因为运行 goroutine 特别开心，正好碰到了 Go 最有魅力的那个设计点：**让很多独立的同步流程自然地同时运行。**

## 整个 go 学习中我需要搞清楚哪些事情

你整个 Go 学习里，真正需要搞清楚的不是“所有语法”，而是下面这些骨架。把它们串起来，你就算真正入门了。

#### 1. 值到底怎么流动

这是 Go 最底层、最值得搞清楚的一块：

```go
x := 10
y := x      // 复制值

p := &x     // 保存 x 的地址
*p = 20     // 修改原来的 x
```

需要弄清：

- 值传递与复制
- 指针 `&`、解引用 `*`
- 结构体值和结构体指针
- `slice`、`map`、`channel` 为什么看起来有引用语义
- 什么时候修改会影响原对象

你现在已经学得比较深入了。

---

#### 2. Go 怎么组织数据和行为

核心是：

```go
type User struct {
	Name string
}

func (u *User) Rename(name string) {
	u.Name = name
}
```

需要掌握：

- `struct`
- 自定义具名类型
- 值接收者与指针接收者
- 具名字段和匿名嵌入
- 组合而不是继承
- 构造函数本质是普通函数

核心判断：

```text
数据属于谁？
行为应该挂在哪个类型上？
这个类型是值，还是有身份、会变化的对象？
```

---

#### 3. 接口到底解决什么问题

接口不是为了给结构体“贴标签”，而是为了：

> 让调用方依赖能力，而不是依赖具体类型。

```go
type UserRepository interface {
	FindByID(id int64) (*User, error)
}
```

需要弄清：

- 隐式实现
- 方法集
- `T` 与 `*T` 实现接口的区别
- 小接口
- 接口应该由使用方定义
- 类型断言和 type switch
- 什么时候不需要接口

你目前已经理解到：

> 接口通过行为定义了一组类型。

下一步只需要在真实项目里看到它如何用于解耦。

---

#### 4. 错误如何沿调用链传播

Go 后端里会反复写：

```go
user, err := repo.FindByID(id)
if err != nil {
	return nil, fmt.Errorf("find user: %w", err)
}
```

需要搞清：

- `error` 是接口
- `(value, error)` 模式
- `errors.New`
- `fmt.Errorf` 和 `%w`
- `errors.Is`
- `errors.As`
- 自定义错误
- 哪一层处理，哪一层继续返回
- `panic` 不应该代替普通错误处理

这是写后端非常核心的一块。

---

#### 5. 切片、map 和字符串

这三种数据结构会占普通业务代码的大部分。

##### 切片

```go
[]User
append
len
cap
range
```

要理解底层数组、共享和扩容，否则后面容易遇到“怎么改了原切片”的困惑。

##### map

```go
map[string]User
value, ok := users[id]
delete(users, id)
```

要知道：

- map 是引用语义
- nil map 可以读，不能写
- map 并发读写不安全
- map 元素不可取地址

##### 字符串

需要区分：

```go
byte
rune
string
```

尤其是中文字符不能简单按字节处理。

---

#### 6. goroutine 和并发协调

这是 Go 最有特色的一块，但不要只停留在：

```go
go work()
```

需要逐步搞清：

```text
goroutine   谁在执行
WaitGroup   谁还没完成
channel     数据怎么传
Mutex       共享数据怎么保护
select      多个事件等谁
context     怎么取消和超时
```

你现在已经理解了：

- `go`
- `WaitGroup`
- channel 基础
- 无缓冲 channel 会阻塞
- `Done()` 和 channel 各管各的

接下来重点是：

1. 有缓冲和无缓冲 channel
2. `close` 与 `range`
3. `select`
4. `context`
5. 数据竞争
6. goroutine 泄漏
7. 限制并发数量

并发不是把所有函数都加上 `go`，而是设计清楚：

> 谁启动、谁等待、谁取消、谁关闭、结果流向哪里。

---

#### 7. 包、模块和可见性

需要搞清 Go 项目是如何组织的：

```text
module
package
file
identifier
```

重点包括：

- `go mod init`
- `go.mod`
- 一个目录通常对应一个包
- 大写导出、小写包内可见
- 包名应该简短
- 如何避免循环依赖
- `internal` 目录
- `cmd` 目录
- 包级变量应少用
- `init` 应谨慎使用

这是从 REPL 进入真实项目时最明显的一道坎。

#### 8. JSON、HTTP 和数据库

##### JSON

#### 9. 测试和工具链

#### 10. 后端架构怎么保持简单
