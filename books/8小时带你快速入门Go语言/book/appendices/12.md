## 有点像引用计数啊。add 是加，done 是减，wait 是清 0 了就继续后面的。

对，这个类比很直观：

```text
Add(1)  → 待完成任务数 +1
Done()  → 待完成任务数 -1
Wait()  → 阻塞，等计数变成 0 后继续
```

而且：

```go
wg.Done()
```

本质上就等价于：

```go
wg.Add(-1)
```

例如：

```go
var wg sync.WaitGroup

wg.Add(3)

go func() {
	defer wg.Done()
	// 任务 1
}()

go func() {
	defer wg.Done()
	// 任务 2
}()

go func() {
	defer wg.Done()
	// 任务 3
}()

wg.Wait()
fmt.Println("三个任务都完成了")
```

可以想象计数变化：

```text
初始：0
Add(3)：3
任务完成：2
任务完成：1
任务完成：0
Wait 解除阻塞
```

不过它和引用计数只是在“加减计数”这一点上相似。

引用计数通常是在统计：

> 还有多少地方持有这个对象。

计数归零后，可能释放对象。

`WaitGroup` 统计的是：

> 还有多少项任务没有完成。

归零后只是唤醒正在 `Wait()` 的 goroutine，不涉及内存释放。

还有两条重要规则：

```go
wg.Add(3) // 一般要在启动 goroutine 之前完成
```

不要让计数减成负数，否则会 panic：

```go
wg.Add(1)
wg.Done()
wg.Done() // panic：negative WaitGroup counter
```

所以可以把 `WaitGroup` 理解成一个非常简单的**并发任务倒计时器**。

## 传 wg 必须传指针

对，**传给函数时通常必须传指针**：

```go
func shopping(name string, wg *sync.WaitGroup) {
	defer wg.Done()
	// ...
}
```

调用：

```go
go shopping("张三", &wg)
```

原因是所有 goroutine 必须操作**同一个 `WaitGroup` 计数器**。

如果按值传递：

```go
func shopping(name string, wg sync.WaitGroup) {
	defer wg.Done()
}
```

每次调用都会复制一份 `wg`。`Done()` 减少的是副本里的计数，而 `main` 中原来的计数没有变化：

```text
main 中的 wg：3，一直不变
shopping 的副本：3 → 2
```

于是：

```go
wg.Wait()
```

可能永远等不到零。

而传指针时：

```text
main ─┐
张三 ─┼→ 同一个 WaitGroup
李四 ─┤
王五 ─┘
```

每个 `Done()` 都减少同一个计数：

```text
3 → 2 → 1 → 0
```

另外，`sync.WaitGroup` 使用后不应该复制。Go 里的同步类型，如这些，通常也都通过指针使用：

```go
*sync.WaitGroup
*sync.Mutex
*sync.RWMutex
```

不过也可以不把 `wg` 作为参数，而让匿名函数直接使用外层变量：

```go
var wg sync.WaitGroup

wg.Add(1)

go func() {
	defer wg.Done()
	fmt.Println("购物")
}()

wg.Wait()
```

这里虽然没显式传 `&wg`，闭包使用的仍然是外面的同一个 `wg`，并没有复制。

## 这协程有点方便啊

是，Go 的 goroutine 确实非常方便，最爽的地方就是：

```go
go shopping("张三", &wg)
```

只加一个 `go`，普通函数调用就变成并发执行了。

这比很多语言都轻：

- 不用手动创建线程对象
- 不用配置线程池才能开始
- goroutine 本身很轻量
- Go 运行时负责把大量 goroutine 调度到少量系统线程上

所以特别适合后端里这种任务：

```go
go sendEmail()
go writeLog()
go fetchRemoteData()
go processFile()
```

不过也正因为启动太容易，容易让人产生一种错觉：

> 加个 `go` 就万事大吉了。

实际上真正麻烦的是后面的管理：

```text
什么时候结束？
怎么拿返回值？
怎么传错误？
怎么取消？
会不会同时修改同一个变量？
会不会泄漏 goroutine？
```

于是 Go 又配套了这些东西：

```text
WaitGroup  等待完成
channel    传值、传结果
Mutex      保护共享数据
context    取消、超时
select     同时等待多个 channel
```

所以 goroutine 本身很简单：

```go
go f()
```

并发控制才是后面的重点。

但这个设计特别符合 Go 的风格：

> 先让简单场景极其简单，复杂场景再逐步加入控制工具。

这也是 Go 写网络服务很舒服的重要原因。每来一个请求，底层就可以由独立 goroutine 处理，你写业务代码时通常不用直接操心线程。

## 只要数据需要一致的是不是就得传指针

不完全是。

更准确地说：

> **需要操作同一份数据时，通常要共享同一个对象；指针是共享对象的一种方式，但数据一致性还需要同步机制。**

例如：

```go
func add(count *int) {
	*count++
}
```

这里传指针，多个地方确实能修改同一个 `count`。

但如果多个 goroutine 同时修改：

```go
count := 0

for range 1000 {
	go func() {
		count++
	}()
}
```

即使大家操作的是同一个变量，也会发生数据竞争，结果可能不正确。

要加锁：

```go
var (
	count int
	mu    sync.Mutex
	wg    sync.WaitGroup
)

for range 1000 {
	wg.Add(1)

	go func() {
		defer wg.Done()

		mu.Lock()
		count++
		mu.Unlock()
	}()
}

wg.Wait()
```

所以是两层问题：

```text
指针：让大家访问同一份数据
锁 / channel：保证大家访问这份数据时不打架
```

`WaitGroup` 必须传指针，是因为所有 goroutine 必须修改同一个任务计数器：

```go
func work(wg *sync.WaitGroup) {
	defer wg.Done()
}
```

但指针并不是唯一的共享方式。`map`、`slice`、`channel` 本身就带有引用语义：

```go
func add(m map[string]int) {
	m["a"] = 1
}
```

这里不用 `*map`，调用者仍然能看到修改。

不过 `map` 在多个 goroutine 里同时读写，同样需要锁：

```go
type SafeMap struct {
	mu   sync.Mutex
	data map[string]int
}
```

另外，有时为了避免共享数据，反而会把值复制给各个 goroutine：

```go
for _, name := range names {
	go func(name string) {
		fmt.Println(name)
	}(name)
}
```

每个 goroutine 拿自己的 `name`，就不需要同步。

所以可以这样记：

```text
需要修改调用者的普通值
→ 通常传指针

多个 goroutine 共享修改同一数据
→ 指针/引用 + Mutex、原子操作或 channel

不需要共享
→ 尽量传值副本，反而更安全
```

**指针解决“是不是同一份数据”，同步机制解决“同时修改会不会乱”。**

## 感觉同步代码在写异步工作

对，Go 很大一个魅力就是：

> **用同步、顺序的代码外观，写并发执行的工作。**

例如：

```go
func shopping(name string) {
	fmt.Println(name, "开始购物")
	time.Sleep(time.Second)
	fmt.Println(name, "购物结束")
}
```

这个函数内部完全是同步的：

```text
打印开始
→ 等待 1 秒
→ 打印结束
```

但只要调用时加上：

```go
go shopping("张三")
go shopping("李四")
go shopping("王五")
```

三个同步函数就会并发运行。

所以 Go 不是要求你把函数内部写成各种回调或 `async/await`，而是：

```text
每个 goroutine 内部：按同步思维写
多个 goroutine 之间：并发执行
```

这和 JavaScript 很不一样。

JavaScript 通常是：

```js
async function shopping(name) {
  console.log(name, "开始购物");
  await sleep(1000);
  console.log(name, "购物结束");
}
```

你需要一直意识到：

```text
这个函数是 async
这里需要 await
返回的是 Promise
```

Go 里：

```go
func shopping(name string) {
	time.Sleep(time.Second)
}
```

仍然就是普通函数。是否并发，取决于调用方式：

```go
shopping("张三")    // 当前 goroutine 同步执行
go shopping("张三") // 新 goroutine 并发执行
```

甚至 channel 的读取看起来也像普通同步代码：

```go
result := <-resultCh
fmt.Println(result)
```

如果结果还没来，当前 goroutine 就停在这里。但不是整个程序停住，Go 运行时会去执行其他 goroutine。

可以想成：

```text
当前 goroutine 阻塞
        ↓
Go 调度器把线程拿去执行其他 goroutine
        ↓
数据到了，再回来继续执行
```

因此 Go 的并发代码读起来经常像普通顺序代码：

```go
result := request()
save(result)
sendNotification()
```

只是这整段代码可能运行在一个 goroutine 里。

这也是 Go 很适合后端的原因之一。处理一个 HTTP 请求时，你可以像写同步代码一样：

```go
func handler(w http.ResponseWriter, r *http.Request) {
	user, err := findUser(r.Context())
	if err != nil {
		// 处理错误
		return
	}

	data, err := loadData(user)
	if err != nil {
		return
	}

	writeResponse(w, data)
}
```

但服务器底层可以让许多这样的 handler 同时在不同 goroutine 中运行。

所以你这个感觉很准确：

> Go 把“异步工作”包装成了“多个同步流程并发运行”。

真正需要额外处理的主要是它们之间的协调：

```text
WaitGroup：等大家完成
channel：互相传数据
Mutex：避免同时改乱共享数据
context：取消和超时
```

而每个单独任务内部，仍然可以按照最自然的同步逻辑来写。

## chan 需要 import 吗？

不需要。`chan` 是 Go 的内置关键字，和 `map`、`struct`、`func` 一样，不属于某个包。

```go
ch := make(chan int)
```

这里：

```go
chan int
```

表示“传递 `int` 的 channel”。

发送和接收：

```go
ch <- 10      // 发送
value := <-ch // 接收
```

关闭 channel 也不需要导包：

```go
close(ch)
```

完整例子只因为要打印，所以才导入 `fmt`：

```go
import "fmt"

ch := make(chan int)

go func() {
	ch <- 10
}()

value := <-ch
fmt.Println(value)
```

还可以限制方向：

```go
chan int   // 可发送、可接收
chan<- int // 只能发送
<-chan int // 只能接收
```

所以 `chan` 本身不用 `import`；你在程序里还用了什么包的功能，才导入什么包。

## Channel

你这段代码会死锁，因为 `moneyChan` 是**无缓冲 channel**：

```go
moneyChan := make(chan int)
```

发送：

```go
moneyChan <- money
```

必须同时有另一个 goroutine 接收。但此时 `main` 正卡在：

```go
wg.Wait()
```

于是形成：

```text
pay 等待 main 接收 money
main 等待 pay 执行 Done
```

而 `Done()` 是 `defer`，只有 `pay` 返回时才执行；`pay` 又卡在发送上，所以谁也走不了。

推荐这样改：主 goroutine 一边接收金额，另一个 goroutine 等待任务完成后关闭 channel。

```go
import (
	"fmt"
	"sync"
	"time"
)

func pay(
	name string,
	money int,
	moneyChan chan<- int,
	wg *sync.WaitGroup,
) {
	defer wg.Done()

	fmt.Printf("%s 开始购物\n", name)
	time.Sleep(time.Second)
	fmt.Printf("%s 购物结束，消费 %d 元\n", name, money)

	moneyChan <- money
}
func main() {
	var wg sync.WaitGroup
	moneyChan := make(chan int)

	wg.Add(3)

	startTime := time.Now()

	go pay("张三", 2, moneyChan, &wg)
	go pay("李四", 3, moneyChan, &wg)
	go pay("王五", 5, moneyChan, &wg)

	// 等所有付款任务完成后关闭 channel
	go func() {
		wg.Wait()
		close(moneyChan)
	}()

	total := 0

	// 不断接收，直到 channel 被关闭
	for money := range moneyChan {
		total += money
	}

	fmt.Printf(
		"购买完成，总计 %d 元，耗时 %v\n",
		total,
		time.Since(startTime),
	)
}

main()
```

输出顺序不固定，但结果大致是：

```text
张三 开始购物
李四 开始购物
王五 开始购物
张三 购物结束，消费 2 元
李四 购物结束，消费 3 元
王五 购物结束，消费 5 元
购买完成，总计 10 元，耗时约 1 秒
```

这里的关键结构是：

```text
3 个 pay goroutine → 向 moneyChan 发送金额
main              → 从 moneyChan 接收并累加
等待协程           → wg 归零后关闭 moneyChan
```

参数里的：

```go
moneyChan chan<- int
```

表示这个函数只能向 channel 发送数据，不能从中接收。这样比直接写 `chan int` 更能说明 `pay` 的职责。

### `moneyChan chan<- int`, 这是啥意思

`moneyChan chan<- int` 表示：

> 参数 `moneyChan` 是一个**只能发送 `int` 的 channel**。

拆开看：

```go
moneyChan chan<- int
```

- `moneyChan`：参数名
- `chan<- int`：只能发送 `int` 的 channel 类型
- 箭头指向 `chan` 外面，可以理解为“数据进入 channel”

所以函数里可以写：

```go
moneyChan <- money
```

但不能接收：

```go
money := <-moneyChan // 编译错误：只能发送
```

三种 channel 类型对比：

```go
chan int   // 双向：能发送，也能接收
chan<- int // 只发送
<-chan int // 只接收
```

例如：

```go
func send(ch chan<- int) {
	ch <- 10
}

func receive(ch <-chan int) {
	value := <-ch
	fmt.Println(value)
}
```

使用时可以把一个普通的双向 channel 传进去：

```go
ch := make(chan int)

go send(ch)
receive(ch)
```

这里 `ch` 本身仍然是：

```go
chan int
```

只是传入 `send` 后，函数只能把它当成 `chan<- int` 使用；传入 `receive` 后，只能当成 `<-chan int` 使用。

你的代码里：

```go
func pay(
	name string,
	money int,
	moneyChan chan<- int,
	wg *sync.WaitGroup,
)
```

是在明确表达：

> `pay` 的职责只是把消费金额发送出去，不允许它偷着从 channel 里读取数据。

这是编译期限制，让函数职责更清楚、更安全。箭头可以这样记：

```go
ch <- value // 值发送进 ch
chan<- int  // 只能发送

value := <-ch // 值从 ch 出来
<-chan int    // 只能接收
```

### channel 在 go 里面算难的吗？

算是 **Go 里入门容易、真正用好比较难** 的部分。

最基础的 channel 很简单：

```go
ch := make(chan int)

go func() {
	ch <- 10
}()

value := <-ch
```

核心就三件事：

```go
ch <- value   // 发送
value := <-ch // 接收
close(ch)     // 关闭
```

你刚才已经碰到了 channel 最关键的地方：**发送和接收会不会互相等待**。

无缓冲 channel：

```go
ch := make(chan int)
```

发送方和接收方必须碰头：

```text
发送方等接收方
接收方也可能等发送方
```

因此很容易写出死锁。你刚才的代码就是：

```text
pay 等 main 接收
main 等 pay Done
```

双方互相等。

带缓冲 channel：

```go
ch := make(chan int, 3)
```

可以先暂存 3 个值：

```go
ch <- 1
ch <- 2
ch <- 3
```

第四次发送才会等待有人接收。

真正开始变难的是这些问题：

- 谁负责关闭 channel？
- 发送方会不会永远等不到接收方？
- 接收方会不会永远等不到数据？
- goroutine 提前退出后，其他 goroutine 会不会泄漏？
- 应该用 channel，还是用 `Mutex`？
- 多个 channel 同时有数据时怎么处理？

后面还会遇到：

```go
select {
case value := <-resultCh:
	fmt.Println(value)

case err := <-errCh:
	fmt.Println(err)

case <-time.After(time.Second):
	fmt.Println("超时")
}
```

`select` 会同时等待多个 channel，这时并发结构就开始复杂了。

不过你不需要一开始掌握所有高级模式。先掌握下面这条主线就够了：

```text
goroutine 负责做事
channel 负责传递结果
WaitGroup 负责等待结束
context 负责取消和超时
Mutex 负责保护共享数据
```

还有一个很实用的判断：

```text
只是等待任务结束
→ WaitGroup

需要把结果传回来
→ channel

多个 goroutine 修改同一个变量
→ Mutex 或重新设计成通过 channel 传递

需要超时或取消
→ context
```

所以 channel 本身语法不难，难的是**设计谁等谁、谁发谁收，以及什么时候结束**。你刚才已经自己运行并看懂了一个“多个生产者把结果传给主流程”的完整例子，实际上已经迈过最容易卡人的第一道坎了。

### channel, wg 各管各的，互不影响？

对，**`channel` 和 `WaitGroup` 是两个独立工具，各管各的，不会自动互相影响。**

```text
WaitGroup：还有多少任务没结束
channel：goroutine 之间如何传递数据
```

### `WaitGroup` 不管数据

```go
wg.Add(3)
wg.Done()
wg.Wait()
```

它只维护一个计数器：

```text
3 → 2 → 1 → 0
```

它不知道 goroutine 发了多少钱，也不知道有没有 channel。

### channel 不管任务数量

```go
moneyChan <- money
money := <-moneyChan
```

它只负责发送和接收数据。它不知道总共有三个购物任务，也不会自动调用 `wg.Done()`。

不过，我们可以通过代码把它们**组织在一起**：

```go
go func() {
	wg.Wait()
	close(moneyChan)
}()
```

意思是：

> 等 `WaitGroup` 确认所有付款任务完成后，再人为关闭 channel。

这不是它们自动联动，而是你写代码建立了关系。

你当前程序的流程是：

```text
pay goroutine
    ↓
向 channel 发送金额
    ↓
函数结束，defer wg.Done()

另一个 goroutine
    ↓
wg.Wait() 等计数归零
    ↓
close(moneyChan)

main goroutine
    ↓
持续从 channel 收金额
    ↓
channel 关闭后结束 range
```

它们虽然各管各的，但执行顺序会互相牵制。例如无缓冲 channel 中：

```go
moneyChan <- money
```

必须有人接收，`pay` 才能继续执行到函数结束，随后 `defer wg.Done()` 才会运行。因此如果主流程先：

```go
wg.Wait()
```

却不接收 channel，就会互相等待而死锁。

所以准确概括是：

> **机制上互相独立，程序流程中可以通过阻塞关系和你写的代码相互协调。**

有时只需要其中一个：

```go
// 只关心任务做完，不需要结果
WaitGroup
```

```go
// 需要接收固定数量的结果
channel
```

需要“收集未知数量的结果，等全部生产者结束后关闭 channel”时，二者配合就很自然。

## for 有哪些语法，怎么还有 for 后面直接花括号的

Go 只有一个循环关键字 `for`，但有几种写法。

#### 1. 经典三段式

```go
for i := 0; i < 3; i++ {
	fmt.Println(i)
}
```

三部分分别是：

```text
初始化；继续条件；每轮结束后的操作
```

相当于很多语言里的 `for`。

---

#### 2. 只有条件，类似 `while`

```go
i := 0

for i < 3 {
	fmt.Println(i)
	i++
}
```

Go 没有 `while`，就用这种写法代替。

---

#### 3. `for {}`：无限循环

```go
for {
	fmt.Println("一直运行")
}
```

因为没有写结束条件，所以默认一直循环。

通常配合 `break`：

```go
i := 0

for {
	if i >= 3 {
		break
	}

	fmt.Println(i)
	i++
}
```

`for {}` 常用于：

- 服务器持续监听请求
- 持续读取消息
- 后台工作 goroutine
- 不确定何时结束的循环

例如：

```go
for {
	conn, err := listener.Accept()
	if err != nil {
		continue
	}

	go handle(conn)
}
```

---

#### 4. `range` 遍历集合

遍历切片：

```go
nums := []int{10, 20, 30}

for index, value := range nums {
	fmt.Println(index, value)
}
```

只需要值：

```go
for _, value := range nums {
	fmt.Println(value)
}
```

只需要下标：

```go
for index := range nums {
	fmt.Println(index)
}
```

也能遍历 `map`：

```go
for key, value := range ages {
	fmt.Println(key, value)
}
```

---

#### 5. 遍历 channel

你刚才的代码中：

```go
for money := range moneyChan {
	total += money
}
```

意思是：

> 不断从 `moneyChan` 接收数据，直到 channel 被关闭。

大致相当于：

```go
for {
	money, ok := <-moneyChan
	if !ok {
		break
	}

	total += money
}
```

所以必须有人最终调用：

```go
close(moneyChan)
```

否则这个 `for range` 会一直等待。

---

#### 6. 重复固定次数

现在也可以写：

```go
for range 3 {
	fmt.Println("执行一次")
}
```

会执行三次。

需要序号时：

```go
for i := range 3 {
	fmt.Println(i)
}
```

输出：

```text
0
1
2
```

所以最常见的几种可以记成：

```go
for i := 0; i < n; i++ {} // 经典循环
for condition {}          // 类似 while
for {}                    // 无限循环
for _, value := range xs {} // 遍历集合
for value := range ch {}    // 持续读取 channel
```

`for {}` 并不是特殊函数，而是把初始化、条件和更新全部省略后形成的无限循环。

### 为什么无限循环，没有 while 吗？

Go 没有 `while`。它把所有循环统一成了一个关键字：`for`。

很多语言里的：

```python
while condition:
    ...
```

在 Go 里就是：

```go
for condition {
	// ...
}
```

例如：

```go
i := 0

for i < 3 {
	fmt.Println(i)
	i++
}
```

这其实就是 Go 版的 `while`。

而：

```go
for {
	// ...
}
```

相当于其他语言里的：

```python
while True:
    ...
```

因为没有条件限制，所以会一直循环，通常用 `break` 退出：

```go
i := 0

for {
	if i >= 3 {
		break
	}

	fmt.Println(i)
	i++
}
```

Go 这么设计主要是为了减少语法：经典 `for`、`while`、无限循环、集合遍历，全都统一使用 `for`。

```go
for i := 0; i < 3; i++ {} // 普通 for
for i < 3 {}               // 相当于 while
for {}                     // 无限循环
for _, v := range values {} // 遍历
```

所以不是 Go 没有 `while` 的能力，而是把 `while` 合并进 `for` 了。

## 协程有关的有几块内容？

Go 里和 goroutine 有关的内容，大致可以分成 **7 块**。你现在已经学到前三块了。

### 1. 启动 goroutine

```go
go work()
```

理解：

- 普通调用 `work()`：当前流程等它完成
- 并发调用 `go work()`：新建 goroutine 执行
- `main` 结束后，其他 goroutine 也会直接结束

---

### 2. 等待任务完成：`WaitGroup`

```go
var wg sync.WaitGroup

wg.Add(1)

go func() {
	defer wg.Done()
	work()
}()

wg.Wait()
```

它是任务倒计时器：

```text
Add   增加待完成任务
Done  完成一个任务
Wait  等待计数归零
```

你现在已经基本掌握了。

---

### 3. goroutine 之间传数据：channel

```go
ch := make(chan int)

go func() {
	ch <- 10
}()

value := <-ch
```

包括：

```go
chan int    // 双向
chan<- int  // 只发送
<-chan int  // 只接收
```

以及：

```go
close(ch)

for value := range ch {
	fmt.Println(value)
}
```

你目前正在学这一块。

---

### 4. 无缓冲与有缓冲 channel

无缓冲：

```go
ch := make(chan int)
```

发送方和接收方必须碰头，类似当面交货。

有缓冲：

```go
ch := make(chan int, 3)
```

可以先在 channel 里暂存 3 个值，类似快递柜。

这一块重点是理解：

- 什么时候发送会阻塞
- 什么时候接收会阻塞
- 为什么会产生死锁

---

### 5. 同时等待多个事件：`select`

```go
select {
case value := <-resultCh:
	fmt.Println("收到结果：", value)

case err := <-errCh:
	fmt.Println("发生错误：", err)

case <-time.After(time.Second):
	fmt.Println("超时")
}
```

`select` 很像专门用于 channel 的 `switch`：

> 哪个 channel 先准备好，就执行哪个分支。

它常用于结果、错误、超时、取消等多路协调。

---

### 6. 共享数据同步：锁和原子操作

多个 goroutine 同时改一份数据时，channel 不一定是最佳方案，有时直接加锁：

```go
var mu sync.Mutex

mu.Lock()
count++
mu.Unlock()
```

相关内容包括：

```go
sync.Mutex
sync.RWMutex
sync.Once
sync.Map
atomic
```

这里要理解：

> 指针负责共享同一份数据，锁负责防止同时修改造成混乱。

---

### 7. 取消、超时与并发模式

后端实际项目里非常重要的是 `context`：

```go
ctx, cancel := context.WithTimeout(
	context.Background(),
	time.Second,
)
defer cancel()
```

goroutine 监听取消：

```go
select {
case <-ctx.Done():
	return
case result := <-resultCh:
	fmt.Println(result)
}
```

然后还有一些常用模式：

```text
worker pool     固定数量的工作协程
fan-out         把任务分给多个 goroutine
fan-in          汇总多个 goroutine 的结果
semaphore       限制最大并发量
errgroup        等待多个任务并收集错误
pipeline        多阶段流水线
```

学习顺序可以这样排：

```text
go
→ WaitGroup
→ channel
→ 缓冲与关闭
→ select
→ Mutex
→ context
→ 常见并发模式
→ 数据竞争和 goroutine 泄漏
```

你现在已经掌握了：

```text
go + WaitGroup + channel 基础
```

下一步最适合继续弄清楚的是：

```text
无缓冲 channel 为什么阻塞
有缓冲 channel 又改变了什么
close 和 range 为什么要配合
```

把这三件事弄明白，Go 并发的骨架基本就立起来了。

## go 并发是 go 的核心吗？

是，**并发确实是 Go 的核心之一**，而且不是后来加上的库功能，而是从语言、运行时到标准库一起设计进去的。

最明显的是这几个直接进入语言语法的东西：

```go
go work()             // 启动 goroutine
ch := make(chan int)  // channel
ch <- value           // 发送
value := <-ch         // 接收

select {
case value := <-ch:
	// ...
}
```

很多语言把并发放在框架或库里，Go 则直接让你用关键字表达。

### 为什么说它是核心

#### 1. 普通函数随时可以并发运行

```go
work()    // 当前 goroutine 执行
go work() // 新 goroutine 执行
```

函数本身不用声明成 `async`，也不用改变返回类型。

#### 2. 运行时专门负责调度

Go 运行时会把大量 goroutine 调度到较少的系统线程上：

```text
大量 goroutine
      ↓
Go 调度器
      ↓
少量系统线程
      ↓
CPU
```

某个 goroutine 等待网络或 channel 时，调度器可以先运行其他 goroutine。

#### 3. 标准库也围绕并发设计

例如 HTTP 服务器收到多个请求时，通常会并发处理每个请求。你写的 handler 仍然是普通同步代码：

```go
func handler(w http.ResponseWriter, r *http.Request) {
	user := loadUser()
	data := loadData(user)
	writeResponse(w, data)
}
```

但许多请求可以同时运行。

#### 4. 有完整的协调工具

```text
goroutine     负责并发执行
channel       负责传递数据
select        等待多个事件
WaitGroup     等待任务结束
Mutex         保护共享数据
context       取消和超时
```

这不是一个孤立的 `go` 关键字，而是一整套并发模型。

### 但 Go 不只是并发语言

Go 的其他核心设计同样重要：

- 简单、统一的语法
- 结构体和组合
- 小接口与隐式实现
- 显式错误处理
- 快速编译
- 单二进制部署
- 垃圾回收
- 强大的标准库与工具链

所以更准确地说：

> **并发是 Go 最有辨识度的核心能力之一；简单工程化则是 Go 更大的整体核心。**

还有一点要区分：

```text
并发：多个任务交错推进
并行：多个任务在多个 CPU 核上同时执行
```

goroutine 首先让并发很容易；是否真正并行，还取决于 CPU 核数、任务类型和调度情况。

你刚才会因为运行 goroutine 特别开心，正好碰到了 Go 最有魅力的那个设计点：**让很多独立的同步流程自然地同时运行。**

## 如果需要交接，缓存使用 1 和无缓存通道是不是一样的效果，还更安全？

不一样。**容量为 1 的缓冲通道只是看起来也能传一个值，但它没有“当面交接”的同步保证。**

无缓冲通道：

```go
ch := make(chan int)
ch <- 1
```

发送者必须等到接收者执行：

```go
x := <-ch
```

两边才同时继续。相当于：

> 我把东西交到你手里之后，我才走。

所以无缓冲通道同时完成两件事：

1. 传递数据；
2. 同步两个协程。

---

容量为 1：

```go
ch := make(chan int, 1)
ch <- 1
```

只要缓冲区是空的，发送者立刻就能继续，接收者可能还完全没运行。

相当于：

> 我把东西放进快递柜，然后走了，你什么时候拿不确定。

因此它们的关键区别是：

```text
无缓冲：发送完成 = 接收者已经接手
缓冲 1：发送完成 = 数据已经放进缓冲区
```

### 缓冲 1 不一定“更安全”

它确实有一个优点：发送方不容易因为接收方稍晚执行而阻塞。例如：

```go
resultCh := make(chan Result, 1)

go func() {
    resultCh <- doWork()
}()
```

即使主协程因为超时已经不再接收，工作协程仍然可以把一个结果放进缓冲区并退出，避免协程永久卡在发送操作上。这种场景下，缓冲 1 的确更安全。

但如果你需要严格交接，例如：

```go
ch <- data
// 下面开始修改或释放 data
```

你希望保证接收方已经拿到数据，那么容量 1 就不够，因为发送完成时，数据可能仍然躺在缓冲区里。此时无缓冲通道更合适。

### 可以这样选择

- **必须确认对方已经接手**：无缓冲通道。
- **允许先放下，对方稍后处理**：缓冲通道。
- **只会产生一个异步结果，并且接收方可能超时离开**：容量 1 通常很好用。
- **流水线需要严格背压，一个处理完下一个才能继续**：无缓冲或小缓冲。

所以你说的“交接”如果真的是**所有权、任务或控制权的当面交接**，无缓冲更准确；如果只是“把结果交出去，不要求对方此刻接住”，容量 1 更灵活。

## 为什么要使用<-而不是直接赋值？

因为 `ch` 这个变量里装的不是“通道中的值”，而是**通道本身的引用/句柄**。

```go
ch := make(chan int)
```

此时可以理解为：

```text
ch → 某个通道对象
```

所以如果写普通赋值：

```go
ch = ...
```

含义应该是：

> 让变量 `ch` 改为指向另一个通道。

例如：

```go
ch1 := make(chan int)
ch2 := make(chan int)

ch1 = ch2
```

这不是往 `ch1` 里放东西，而是让 `ch1` 和 `ch2` 指向同一个通道。

---

## `<-` 表示数据流动

Go 单独设计了 `<-`，用来表示值在通道中的流动方向：

```go
ch <- 10
```

可以直观看作：

```text
ch ← 10
```

把 `10` 送入通道。

接收时：

```go
x := <-ch
```

可以看作：

```text
x ← ch
```

从通道中取出值，赋给 `x`。

所以：

```go
ch <- 10
x := <-ch
```

箭头都指向数据流动的目标。

---

## 为什么不能设计成这样？

理论上语言设计者也可以设计成：

```go
ch = 10
```

但会产生歧义，因为 `ch` 的类型是：

```go
chan int
```

而 `10` 的类型是：

```go
int
```

普通赋值通常表示修改变量本身：

```go
x = 10
```

但通道发送并不是修改变量 `ch`，而是通过 `ch` 操作它背后的通道对象。

它更像：

```go
slice = append(slice, 10)
map[key] = 10
file.Write(data)
ch <- 10
```

其中 `ch <- 10` 是一种专门的并发操作，并且它可能阻塞。普通赋值通常不会表达这种同步语义。

---

## `<-` 还能够限制通道方向

它不仅用于发送和接收，也能写在类型里：

```go
chan int   // 可发送、可接收
chan<- int // 只能发送
<-chan int // 只能接收
```

例如：

```go
func producer(out chan<- int) {
    out <- 10
}

func consumer(in <-chan int) {
    x := <-in
    fmt.Println(x)
}
```

这种语法非常统一：

```go
chan<- int
```

箭头指向通道：值只能进入通道。

```go
<-chan int
```

箭头从通道出来：值只能从通道出来。

所以 `<-` 的核心意思不是“特殊赋值符”，而是：

> **通过通道发送或接收数据，并标明数据流动方向。**

## 我咋感觉这里的 ch 应该传指针？

不用传指针。因为 **channel 本身就是引用语义的值**。

```go
ch := make(chan int, 1)
```

这里 `ch` 不是整个通道的数据本体，而更像一个指向运行时通道对象的“句柄”：

```text
ch ──→ 通道对象
```

当你把它传进函数：

```go
go func(ch chan int) {
	fmt.Println(<-ch)
}(ch)
```

复制的是这个句柄，不是复制一个新通道：

```text
main 中的 ch ─────┐
                  ├──→ 同一个通道对象
函数参数 ch ──────┘
```

所以函数中的：

```go
<-ch
```

读取的就是外面那个通道里的 `1`。

可以用普通赋值理解：

```go
ch1 := make(chan int, 1)
ch2 := ch1

ch1 <- 10
fmt.Println(<-ch2) // 10
```

`ch1` 和 `ch2` 是两个变量，但指向同一个通道。

### 为什么结构体有时要传指针？

普通结构体传值会复制全部字段：

```go
type User struct {
	Name string
}

func change(u User) {
	u.Name = "new"
}
```

这里修改的是副本。

但通道、map、slice、函数这些值，内部本来就包含了对底层对象的引用。复制它们后，通常仍然操作同一个底层对象。

因此一般写：

```go
func worker(ch chan int)
```

而不是：

```go
func worker(ch *chan int)
```

`*chan int` 只有在你需要修改“通道变量本身”时才可能使用，例如让调用者的变量指向另一个通道：

```go
func replace(ch *chan int) {
	*ch = make(chan int)
}
```

这种需求极少见，也通常有更自然的设计。

你的代码甚至可以直接捕获外层变量，不传参数：

```go
go func() {
	fmt.Println(<-ch)
	close(done)
}()
```

两种写法都正确。传参的写法只是更明确地表达：“这个 goroutine 使用这个通道”。

## done 的三个语句是什么作用？

`done` 这三个语句合起来，就是让主协程等待子协程执行完。

```go
done := make(chan struct{})
```

创建一个无缓冲通道，专门用于发送“完成信号”。

这里用：

```go
struct{}
```

是因为它是空结构体，不携带任何数据，几乎不占空间。我们只关心“完成了没有”，不需要传具体值。

---

子协程里的：

```go
close(done)
```

表示：

> 工作已经完成，关闭 `done` 通道。

关闭通道后，所有正在等待接收这个通道的 goroutine 都会立刻继续执行。

---

主协程里的：

```go
<-done
```

表示从 `done` 接收数据。

因为 `done` 是无缓冲通道，而且暂时没有值，所以这里会阻塞：

```go
<-done // 等待
```

直到子协程执行：

```go
close(done)
```

主协程才会解除阻塞，继续向下运行：

```go
fmt.Println("程序执行完了")
```

整个执行顺序大致是：

```text
main 创建 done
        ↓
启动 goroutine
        ↓
main 执行 <-done，开始等待
        ↓
goroutine 打印 ch 中的值
        ↓
goroutine close(done)
        ↓
main 解除等待
        ↓
打印“程序执行完了”
```

这里其实也可以不关闭，而是发送一个空值：

```go
done <- struct{}{}
```

对应主协程仍然写：

```go
<-done
```

完整写法：

```go
done := make(chan struct{})

go func() {
	fmt.Println(<-ch)
	done <- struct{}{}
}()

<-done
```

两者区别是：

```go
close(done)
```

可以同时唤醒所有等待者；

```go
done <- struct{}{}
```

一次发送通常只唤醒一个接收者。

你这里只有一个等待者，所以两种都可以。`close(done)` 更像是在广播：

> 这件事已经彻底完成了。

### 简化版的 wg?

对，**可以把这个 `done` 通道理解成单个 goroutine 场景下的简化版 `WaitGroup`**。

你的写法：

```go
done := make(chan struct{})

go func() {
	fmt.Println(<-ch)
	close(done)
}()

<-done
```

表达的是：

> 启动一个 goroutine，main 等它完成。

对应 `WaitGroup`：

```go
var wg sync.WaitGroup

wg.Add(1)

go func() {
	defer wg.Done()
	fmt.Println(<-ch)
}()

wg.Wait()
```

二者效果基本相同。

#### 区别

`done` 通道更适合：

- 只等待一个 goroutine；
- 除了完成，还需要配合 `select`、超时或取消；
- 想广播“已经完成”。

`WaitGroup` 更适合：

- 等待多个 goroutine；
- 只关心全部完成，不需要传递结果；
- 典型的并发任务汇总。

例如等待 10 个任务，`WaitGroup` 很自然：

```go
var wg sync.WaitGroup

for i := 0; i < 10; i++ {
	wg.Add(1)

	go func(i int) {
		defer wg.Done()
		fmt.Println(i)
	}(i)
}

wg.Wait()
```

用一个 `done` 通道做这件事，就要自己计数：

```go
done := make(chan struct{})

for i := 0; i < 10; i++ {
	go func(i int) {
		fmt.Println(i)
		done <- struct{}{}
	}(i)
}

for range 10 {
	<-done
}
```

能写，但明显没有 `WaitGroup` 自然。

所以可以记成：

> 一个 goroutine：`done` 很轻便。  
> 多个 goroutine：通常用 `WaitGroup`。  
> 需要超时、取消、`select`：通道更灵活。

### 感觉 1 个的时候代码量也差不多啊

对，**单个 goroutine 时，代码量确实差不多，甚至 `WaitGroup` 还更啰嗦一点。**

`done`：

```go
done := make(chan struct{})

go func() {
	defer close(done)
	doWork()
}()

<-done
```

`WaitGroup`：

```go
var wg sync.WaitGroup
wg.Add(1)

go func() {
	defer wg.Done()
	doWork()
}()

wg.Wait()
```

所以单个任务时，选择主要不是看代码量，而是看语义。

`done` 表达的是：

> 某件事完成了。

`WaitGroup` 表达的是：

> 等待一组 goroutine 的计数归零。

因此只有一个 goroutine，并且只是简单等待完成，`done` 很自然。但如果没有超时、取消、`select` 等需求，直接用 `WaitGroup` 也完全可以，而且以后增加 goroutine 更方便。

真正体现 `done` 优势的是可以放进 `select`：

```go
select {
case <-done:
	fmt.Println("完成")
case <-time.After(time.Second):
	fmt.Println("超时")
}
```

`wg.Wait()` 不能直接放进 `select`。

所以可以简单记：

- 单个任务、单纯等待：两者差不多。
- 预计以后会增加多个任务：`WaitGroup`。
- 需要配合超时、取消或其他通道：`done`。

## 如何打印 done 的值，但不影响它

如果你的 `done` 是这样用的：

```go
done := make(chan struct{})
close(done)
```

那么它其实**没有一个需要打印的“值”**。`struct{}{}` 只是完成信号，真正有意义的是：

> `done` 有没有被关闭。

可以非阻塞地检查：

```go
select {
case <-done:
	fmt.Println("任务已经完成")
default:
	fmt.Println("任务尚未完成")
}
```

因为关闭后的通道可以被无限次接收，所以这里不会把它“用掉”：

```go
close(done)

<-done
<-done
<-done // 都会立即通过
```

打印接收到的值也可以：

```go
v, ok := <-done
fmt.Println(v, ok)
```

输出：

```text
{} false
```

其中：

- `{}` 是 `struct{}{}` 的零值；
- `false` 表示通道已经关闭。

更常见的是只打印 `ok`：

```go
_, ok := <-done
fmt.Println(ok) // false
```

不过，如果通道不是通过 `close(done)` 通知，而是这样发送：

```go
done <- struct{}{}
```

那么接收：

```go
<-done
```

会把这个信号消费掉。Go 的 channel 没有“偷看但不取出”的操作。

所以：

- **关闭通道作为完成信号**：可以被多个地方反复检查，不会消耗。
- **向通道发送一个值作为信号**：接收一次就会消费一次。

你的 `done` 场景更适合 `close(done)`，因为它本质上是广播“已经完成”。

### 如何打印 ch 的值，但不影响它

**不能直接“偷看”channel 里的值而不取走它。** Go 的 channel 没有 `peek` 操作。

```go
v := <-ch
fmt.Println(v)
```

这一定会消费一个值。

你可能想到“取出来再放回去”：

```go
v := <-ch
fmt.Println(v)
ch <- v
```

但这并不等价于没影响：

- 改变了值在队列中的顺序；
- 其他 goroutine 可能在你放回前读走其他值；
- 无缓冲通道可能直接阻塞；
- 通道已关闭时不能放回，会 panic。

例如容量为 2：

```go
ch <- 1
ch <- 2

v := <-ch // 取出 1
ch <- v   // 现在顺序变成 2、1
```

你只能无损查看通道的部分状态：

```go
fmt.Println(len(ch)) // 当前缓冲区中的元素数量
fmt.Println(cap(ch)) // 缓冲区容量
```

但不能查看具体元素。

如果确实需要随时查看“当前值”，通常应把值另外保存在变量中，并通过锁保护：

```go
var (
	mu      sync.RWMutex
	current int
)

func send(ch chan<- int, v int) {
	mu.Lock()
	current = v
	mu.Unlock()

	ch <- v
}

func printCurrent() {
	mu.RLock()
	defer mu.RUnlock()

	fmt.Println(current)
}
```

所以可以记住：

> channel 是传递队列，不是可查看的容器；接收就意味着消费。

## 关闭通道会影响哪些？

关闭一个 channel，会影响所有持有这个 channel 的 goroutine，因为它们指向的是同一个底层通道。

```go
ch1 := make(chan int)
ch2 := ch1

close(ch1)
```

此时 `ch2` 也已经关闭了，并不是只关闭变量 `ch1`。

### 对接收方的影响

关闭后，接收方仍然可以接收。

如果缓冲区还有值，会先把剩余值取完：

```go
ch := make(chan int, 2)
ch <- 10
ch <- 20
close(ch)

fmt.Println(<-ch) // 10
fmt.Println(<-ch) // 20
```

缓冲区取完后，再接收会立即返回该类型的零值：

```go
fmt.Println(<-ch) // 0
fmt.Println(<-ch) // 0
```

不会阻塞。

通常用两个返回值判断：

```go
v, ok := <-ch

fmt.Println(v)  // 0
fmt.Println(ok) // false
```

其中：

- `ok == true`：正常接收到值；
- `ok == false`：通道已关闭，而且缓冲区已经为空。

### 对发送方的影响

向关闭的 channel 发送会直接 panic：

```go
close(ch)
ch <- 1 // panic: send on closed channel
```

因此一般遵守：

> 由发送方关闭 channel，接收方不要关闭。

因为发送方最清楚后面是否还会继续发送。

### 对 `range` 的影响

`range` 会持续读取 channel：

```go
for v := range ch {
	fmt.Println(v)
}
```

它会：

1. 读取剩余缓冲值；
2. 缓冲区清空；
3. 检测到 channel 已关闭；
4. 自动结束循环。

例如：

```go
ch := make(chan int, 2)
ch <- 1
ch <- 2
close(ch)

for v := range ch {
	fmt.Println(v)
}

fmt.Println("结束")
```

输出：

```text
1
2
结束
```

如果不关闭 channel，而又没有新的发送者，`range` 会一直等下去，最终可能死锁。

### 对 `select` 的影响

从已关闭 channel 接收的 `case` 会立即就绪：

```go
select {
case v, ok := <-ch:
	fmt.Println(v, ok)
case <-other:
	fmt.Println("other")
}
```

如果 `ch` 已关闭且已清空，第一个 case 可以立即执行，并得到零值和 `false`。

所以在循环中要处理关闭状态：

```go
for {
	select {
	case v, ok := <-ch:
		if !ok {
			fmt.Println("ch 已关闭")
			return
		}
		fmt.Println(v)
	}
}
```

否则可能一直读取零值，形成死循环：

```go
for {
	select {
	case v := <-ch:
		fmt.Println(v) // 关闭后不停打印 0
	}
}
```

### 对等待中的接收者的影响

假设多个 goroutine 都在等待：

```go
<-ch
```

执行：

```go
close(ch)
```

以后，所有等待接收的 goroutine 都会被唤醒。

因此关闭 channel 经常用作广播信号：

```go
done := make(chan struct{})

go func() {
	<-done
	fmt.Println("worker 1 停止")
}()

go func() {
	<-done
	fmt.Println("worker 2 停止")
}()

close(done)
```

两个 goroutine 都会收到“完成”信号。

而：

```go
done <- struct{}{}
```

通常只会唤醒一个接收者。

### 关闭 channel 不会做什么

关闭 channel 不会：

- 自动停止 goroutine；
- 清理 goroutine；
- 取消正在执行的函数；
- 删除缓冲区中已有的数据；
- 让 channel 变量变成 `nil`。

它只是表示：

> 这个 channel 从此不会再有新值发送进来。

### 几个会 panic 的操作

```go
close(ch)
close(ch) // panic：重复关闭
```

```go
var ch chan int
close(ch) // panic：关闭 nil channel
```

```go
close(ch)
ch <- 1 // panic：向关闭的 channel 发送
```

但是从关闭的 channel 接收不会 panic。

可以记成：

> **关闭后：不能再发，但还能收；剩余值收完后，接收立即得到零值和 `false`。**

## select 是什么场景下用的？

`select` 用在：**一个 goroutine 同时等待多个 channel 操作，哪个先准备好就执行哪个。**

它很像只服务于 channel 的 `switch`。

### 1. 同时等待多个结果

```go
select {
case v := <-ch1:
	fmt.Println("收到 ch1：", v)

case v := <-ch2:
	fmt.Println("收到 ch2：", v)
}
```

此时当前 goroutine 会阻塞等待：

- `ch1` 先有数据，就执行第一个分支；
- `ch2` 先有数据，就执行第二个分支。

适合多个并发任务竞争返回结果：

```go
result1 := make(chan string)
result2 := make(chan string)

go func() {
	result1 <- "任务 1 完成"
}()

go func() {
	result2 <- "任务 2 完成"
}()

select {
case r := <-result1:
	fmt.Println(r)
case r := <-result2:
	fmt.Println(r)
}
```

注意：这里只接收最先完成的那个，另一个 goroutine 可能还会继续运行。

---

### 2. 设置超时

这是非常常见的场景：

```go
select {
case v := <-ch:
	fmt.Println("收到结果：", v)

case <-time.After(2 * time.Second):
	fmt.Println("等待超时")
}
```

含义是：

> 最多等待 `ch` 两秒；两秒内没结果，就执行超时分支。

例如网络请求、数据库查询、后台任务，都可能需要超时控制。

不过在循环中一般不要反复使用 `time.After`，更常用计时器：

```go
timer := time.NewTimer(2 * time.Second)
defer timer.Stop()

select {
case v := <-ch:
	fmt.Println(v)

case <-timer.C:
	fmt.Println("超时")
}
```

---

### 3. 响应取消信号

例如使用 `context`：

```go
func worker(ctx context.Context, jobs <-chan int) {
	for {
		select {
		case job := <-jobs:
			fmt.Println("处理任务：", job)

		case <-ctx.Done():
			fmt.Println("任务取消")
			return
		}
	}
}
```

这个 worker 同时等待：

- 新任务；
- 取消信号。

这在 Go 后端中非常常见。比如 HTTP 请求被取消后，相关 goroutine 也应停止工作。

---

### 4. 同时等待发送和接收

`select` 不仅能接收，也能发送：

```go
select {
case v := <-input:
	fmt.Println("收到：", v)

case output <- result:
	fmt.Println("结果发送成功")
}
```

哪个操作能立即完成，就选择哪个。

例如一个 goroutine 既可能接到新数据，也可能把之前计算好的结果发出去。

---

### 5. 非阻塞操作

加入 `default` 后，`select` 不会等待：

```go
select {
case v := <-ch:
	fmt.Println("收到：", v)

default:
	fmt.Println("现在没有数据")
}
```

它相当于：

> 有数据就取；没有数据就立即离开。

非阻塞发送也一样：

```go
select {
case ch <- value:
	fmt.Println("发送成功")

default:
	fmt.Println("通道暂时无法接收")
}
```

不过 `default` 要谨慎使用。尤其在循环中：

```go
for {
	select {
	case v := <-ch:
		fmt.Println(v)
	default:
	}
}
```

这会不停空转，占满 CPU。

---

### 6. 循环监听事件

这是 `select` 最典型的样子：

```go
for {
	select {
	case job := <-jobs:
		fmt.Println("处理：", job)

	case err := <-errors:
		fmt.Println("错误：", err)

	case <-done:
		fmt.Println("结束")
		return
	}
}
```

可以把它理解成一个 goroutine 的“事件循环”：

```text
新任务来了 → 处理任务
错误来了   → 处理错误
完成了     → 退出
```

### 多个 case 同时准备好怎么办？

Go 会从已经准备好的 case 中**伪随机选择一个**：

```go
select {
case <-ch1:
case <-ch2:
}
```

如果 `ch1` 和 `ch2` 此刻都有数据，不保证永远先选第一个。

所以 `select` 不是按从上到下的优先级执行。

### 什么时候不需要 `select`？

只有一个 channel，而且只是单纯等待：

```go
v := <-ch
```

就够了，不需要写：

```go
select {
case v := <-ch:
	fmt.Println(v)
}
```

单 channel 使用 `select` 通常是因为还需要：

- 超时；
- 取消；
- 非阻塞；
- 同时尝试发送。

可以把它记成一句话：

> 普通 `<-ch` 是“等这一个通道”；`select` 是“同时等几个可能发生的通道事件”。

## 写一段 mvp 我学习一下

先看一个最小可运行例子：**等待任务结果，但最多只等 2 秒。**

```go
package main

import (
	"fmt"
	"time"
)

func main() {
	// 创建一个用于接收任务结果的 channel。
	//
	// channel 中传递的值类型是 string，
	// 缓冲区容量是 1。
	//
	// 为什么容量设置为 1？
	//
	// 因为主 goroutine 可能会先超时，离开下面的 select。
	// 如果这里使用无缓冲 channel：
	//
	//     result := make(chan string)
	//
	// 那么任务 goroutine 后来执行发送操作时：
	//
	//     result <- "任务完成"
	//
	// 可能因为已经没有接收者而永久阻塞。
	//
	// 容量为 1 时，即使主 goroutine 已经超时离开，
	// 工作 goroutine 仍然可以把一个结果放入缓冲区，然后正常结束。
	result := make(chan string, 1)

	// 启动一个新的 goroutine，模拟一个耗时任务。
	//
	// main 函数本身运行在主 goroutine 中。
	// 下面这个匿名函数会运行在另一个 goroutine 中，
	// 因此不会阻塞 main goroutine 继续向下执行。
	go func() {
		// 模拟任务需要执行 1 秒。
		//
		// 在真实项目中，这里可能是：
		//
		// - 查询数据库
		// - 请求外部 API
		// - 读取文件
		// - 执行复杂计算
		time.Sleep(1 * time.Second)

		// 任务完成后，把结果发送到 result channel。
		//
		// 因为 result 的缓冲区容量是 1，
		// 所以只要缓冲区还是空的，这个发送操作就可以立即完成，
		// 不要求主 goroutine 此刻正在接收。
		result <- "任务完成"
	}()

	// select 会同时等待多个 channel 操作。
	//
	// 它不会按照代码从上到下依次执行，
	// 而是等待下面哪个 case 先准备好。
	//
	// 当前同时等待两个事件：
	//
	// 1. result 收到任务结果；
	// 2. 2 秒的超时时间到达。
	//
	// 哪个先发生，就执行哪个 case。
	select {
	case value := <-result:
		// 如果任务在 2 秒内完成，
		// 工作 goroutine 会向 result 发送一个字符串。
		//
		// 此时这个接收操作可以执行，
		// select 就会选择这个分支。
		//
		// value 保存从 result channel 中收到的值，
		// 在这里就是字符串 "任务完成"。
		fmt.Println("收到结果：", value)

	case <-time.After(2 * time.Second):
		// time.After 会返回一个只读的时间 channel：
		//
		//     <-chan time.Time
		//
		// 2 秒以后，计时器会向这个 channel 发送一个时间值。
		//
		// 这里我们不关心具体的时间值，
		// 只关心“2 秒已经到了”，
		// 所以直接写：
		//
		//     <-time.After(...)
		//
		// 而没有把收到的时间保存到变量中。
		//
		// 如果任务超过 2 秒仍未返回结果，
		// select 就会执行这个超时分支。
		fmt.Println("任务超时")
	}

	// select 只会执行其中一个 case。
	//
	// 无论是成功收到结果，还是发生超时，
	// 执行完相应的 case 后都会继续运行这里。
	// 需要注意：这里的“超时”只是主 goroutine 不再等待结果，并没有真正取消工作 goroutine。工作 goroutine 仍会继续睡满 3 秒，然后把结果放进缓冲区并结束。
	fmt.Println("程序结束")
}
main()
```

运行结果大致是：

```text
任务完成
程序结束
```

核心是：

```go
select {
case value := <-result:
	fmt.Println(value)

case <-time.After(2 * time.Second):
	fmt.Println("任务超时")
}
```

它同时等待两个 channel：

```text
result             任务完成时收到结果
time.After(...)    两秒后收到超时信号
```

哪个先有值，就执行哪个 `case`。

把任务时间改成 3 秒：

```go
time.Sleep(3 * time.Second)
```

输出就会变成：

```text
任务超时
程序结束
```

这里 `result` 使用容量为 1 的缓冲通道很重要：

```go
result := make(chan string, 1)
```

假设主协程已经因超时离开 `select`，工作 goroutine 后来仍然可以把结果放入缓冲区并正常结束。要是无缓冲通道：

```go
result := make(chan string)
```

超时后没人接收结果，工作 goroutine 就可能永远卡在：

```go
result <- "任务完成"
```

这段代码可以记成一个常见模板：

```go
result := make(chan Result, 1)

go func() {
	result <- doWork()
}()

select {
case value := <-result:
	// 使用结果
case <-time.After(timeout):
	// 处理超时
}
```

### select 是哪个先到就执行哪个，执行完就结束了，不再等其它的了？

对，**默认就是这样**。

`select` 的行为可以理解成：

1. 同时等多个 `case`
2. **哪个先就绪，就执行哪个**
3. **只执行这一个**
4. 执行完之后，`select` 整体就结束了
5. 不会继续在这一次 `select` 里等别的 `case`

例如：

```go
select {
case v := <-ch1:
	fmt.Println("收到 ch1:", v)

case v := <-ch2:
	fmt.Println("收到 ch2:", v)
}
```

含义是：

- 等 `ch1` 或 `ch2`
- 谁先准备好就选谁
- 打印完就离开这个 `select`

---

### 它不是“持续监听器”

它**只处理一次事件**。

如果你想持续监听多个通道，就要放进循环里：

```go
for {
	select {
	case v := <-ch1:
		fmt.Println("收到 ch1:", v)

	case v := <-ch2:
		fmt.Println("收到 ch2:", v)

	case <-done:
		fmt.Println("结束")
		return
	}
}
```

这个才是：

> 不断等待多个事件，来一个处理一个。

---

### 如果多个同时就绪呢？

如果好几个 `case` 同时都可以执行，Go 会**伪随机选一个**，不是按顺序优先。

```go
select {
case <-ch1:
	fmt.Println("1")
case <-ch2:
	fmt.Println("2")
}
```

如果此刻 `ch1` 和 `ch2` 都有值，不保证永远先选 `ch1`。

---

### 你的理解可以总结成一句话

> **一次 `select` 只处理一个先到的 channel 事件，处理完就结束。**

## select 里 case 后面是什么？

`select` 里的 `case` 后面必须是一个 **channel 操作**，主要有三种写法。

### 1. 从 channel 接收值

```go
case value := <-ch:
	fmt.Println(value)
```

意思是：

> 等待从 `ch` 中收到一个值，并赋给 `value`。

也可以同时判断通道是否关闭：

```go
case value, ok := <-ch:
	if !ok {
		fmt.Println("通道关闭了")
		return
	}
	fmt.Println(value)
```

---

### 2. 只等待信号，不接收具体值

```go
case <-done:
	fmt.Println("结束")
```

这里仍然是在从 `done` 接收，只是没有把收到的值保存下来。

常用于：

- 完成信号；
- 取消信号；
- 超时信号。

例如：

```go
case <-time.After(2 * time.Second):
	fmt.Println("超时")
```

---

### 3. 向 channel 发送值

```go
case ch <- value:
	fmt.Println("发送成功")
```

意思是：

> 等待 `value` 能够发送进 `ch`。

例如：

```go
select {
case jobs <- job:
	fmt.Println("任务已提交")

case <-done:
	fmt.Println("停止提交")
}
```

---

另外还有一个特殊分支：

```go
default:
	fmt.Println("当前没有 channel 可以操作")
```

`default` 后面没有 channel 操作。它表示：

> 如果其他所有 `case` 现在都不能立即执行，就执行这里，不等待。

所以可以记成：

```go
select {
case 接收操作:
case 发送操作:
default:
}
```

不能像普通 `switch` 那样写一般条件：

```go
select {
case x > 10: // 错误
}
```

因为 `select` 专门用于等待 channel，而普通条件判断应使用 `switch` 或 `if`。

## `done <- struct{}{}` 什么意思？

```go
done <- struct{}{}
```

意思是：

> 向 `done` 通道发送一个空结构体值。

拆开看：

```go
done
```

是通道，例如：

```go
done := make(chan struct{})
```

它的类型是：

```go
chan struct{}
```

表示这个通道只能传递 `struct{}`。

然后：

```go
struct{}{}
```

是一个空结构体值。

前面的：

```go
struct{}
```

是类型。

后面的：

```go
{}
```

是在创建这个类型的一个值。

所以：

```go
struct{}{}
```

类似于：

```go
User{}
```

只是它没有任何字段。

完整的数据流就是：

```go
done <- struct{}{}
```

可以读成：

```text
把一个 struct{} 空值发送进 done
```

接收方写：

```go
<-done
```

表示等待这个完成信号。

完整例子：

```go
done := make(chan struct{})

go func() {
	fmt.Println("开始工作")

	// 工作完成后发送信号
	done <- struct{}{}
}()

// 等待完成信号
<-done

fmt.Println("程序结束")
```

这里并不关心传递了什么数据，只关心：

> 发送动作发生了，说明工作完成了。

之所以用 `struct{}{}`，是因为不需要携带 `true`、数字或字符串。它只是一个纯粹的信号。

可以和 `bool` 对比：

```go
done := make(chan bool)
done <- true
```

也能用，但这个 `true` 没有实际信息，因为只要收到值，就已经知道完成了。

因此 Go 中常用：

```go
chan struct{}
```

表达“只传信号，不传数据”。

还要注意，无缓冲通道中：

```go
done <- struct{}{}
```

会一直阻塞，直到另一个 goroutine 执行：

```go
<-done
```

所以它相当于一次一对一的交接。

## 噢，给一个值和关闭都能让等待的`<-done`结束等待

对，正是这样。

这两种都能让：

```go
<-done
```

解除阻塞。

发送一个值：

```go
done <- struct{}{}
```

相当于：

> 给一个等待者发送一次完成信号。

关闭通道：

```go
close(done)
```

相当于：

> 宣布这件事已经完成，所有等待者都可以继续。

主要区别是：

```go
done <- struct{}{}
```

通常只唤醒一个接收者，而且这个值会被消费掉。

```go
close(done)
```

会唤醒所有接收者；关闭后，任何新的：

```go
<-done
```

也都会立即通过。

例如：

```go
done := make(chan struct{})

go func() {
	<-done
	fmt.Println("worker 1 结束")
}()

go func() {
	<-done
	fmt.Println("worker 2 结束")
}()

close(done)
```

两个 worker 都会被唤醒。

但改成：

```go
done <- struct{}{}
```

通常只有其中一个 worker 能收到，另一个还会继续等待。

所以可以记成：

> 发送值是一次通知；关闭通道是广播完成状态。
