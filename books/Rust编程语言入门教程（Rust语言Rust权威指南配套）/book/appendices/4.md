# 4

## `String::from`

```rust
String::from("hello")
```

意思是：

> 用字符串切片 `"hello"` 创建一个拥有所有权的 `String`。

拆开看：

```rust
String      // 字符串类型
::          // 访问这个类型上的关联函数
from(...)   // 调用 from 函数进行转换
```

所以：

```rust
let s = String::from("hello");
```

这里 `s` 的类型是：

```rust
String
```

而单独的：

```rust
let s = "hello";
```

类型则是：

```rust
&str
```

区别大致是：

| 类型     | 特点                                     |
| -------- | ---------------------------------------- |
| `&str`   | 借用的一段字符串内容，通常不可扩展       |
| `String` | 自己拥有内容，存放在堆上，可以修改和增长 |

例如：

```rust
let mut s = String::from("hello");
s.push_str(" world");

dbg!(s);
```

结果是：

```text
"hello world"
```

但这样不行：

```rust
let mut s = "hello";
s.push_str(" world");
```

因为 `s` 是 `&str`，不是可增长的 `String`。

`String::from` 的 `from` 来自 `From` trait。下面写法基本等价：

```rust
let a = String::from("hello");
let b = "hello".to_string();
let c: String = "hello".into();
```

你现在可以先把它记成：

```text
"hello"                → 字符串切片 &str
String::from("hello")  → 拥有内容的 String
```

这里的 `::` 也很重要：`from` 是属于 `String` 类型的关联函数，不是某个具体字符串对象的方法。

## 什么是 trait

`trait` 可以先理解成：

> **一组“某种能力”的约定。**

它规定：

> 只要某个类型实现了这个 trait，它就必须具备这些方法或行为。

很像 Java/C# 的 `interface`，但 Rust 的 trait 能力更强一些。

### 一个最简单的 trait

```rust
trait Speak {
    fn speak(&self);
}
```

这表示定义了一种叫 `Speak` 的能力：

> 实现 `Speak` 的类型，必须提供一个 `speak` 方法。

现在定义两个类型：

```rust
struct Dog;
struct Cat;
```

分别为它们实现 `Speak`：

```rust
impl Speak for Dog {
    // 关键字是用来标注一块实现代码，描述“这是一份实现”，对应名词 implementation；
    // 动词 implement 含义是“去执行实现这个动作”，无法指代一段代码块，不适合做关键字。
    fn speak(&self) {
        println!("汪");
    }
}

impl Speak for Cat {
    fn speak(&self) {
        println!("喵");
    }
}
```

然后：

```rust
let dog = Dog;
let cat = Cat;

dog.speak();
cat.speak();
```

输出：

```text
汪
喵
```

这里的结构是：

```text
trait Speak       定义“会说话”这种能力
Dog implements    狗具体怎么说
Cat implements    猫具体怎么说
```

### trait 解决了什么问题

假如我们想写一个函数，参数不要求必须是 `Dog` 或 `Cat`，只要求它“会说话”：

```rust
fn make_speak(item: &impl Speak) {
    item.speak();
}
```

现在两个类型都能传进去：

```rust
make_speak(&dog);
make_speak(&cat);
```

这相当于说：

> 我不关心你具体是什么类型，只关心你有没有实现 `Speak`。

这就是 trait 最核心的用途：**按能力编程，而不是按具体类型编程。**

### `impl` 的两个含义

你会经常看到两种 `impl`。

给类型定义自己的方法：

```rust
impl Dog {
    fn new() -> Dog {
        Dog
    }
}
```

这是：

> 为 `Dog` 添加固有方法。

给类型实现 trait：

```rust
impl Speak for Dog {
    fn speak(&self) {
        println!("汪");
    }
}
```

这是：

> 为 `Dog` 实现 `Speak` 这项能力。

注意中间多了一个 `for`：

```text
impl Dog             给 Dog 写自己的方法
impl Speak for Dog   给 Dog 实现 Speak trait
```

### trait 可以提供默认实现

```rust
trait Speak {
    fn speak(&self) {
        println!("默认声音");
    }
}
```

某个类型实现它时，可以直接使用默认方法：

```rust
struct Unknown;

impl Speak for Unknown {}
```

也可以覆盖：

```rust
impl Speak for Dog {
    fn speak(&self) {
        println!("汪");
    }
}
```

所以 trait 不仅可以声明方法，还可以直接提供一部分通用实现。

### 你已经见过很多 trait

例如：

```rust
String::from("hello")
```

这里的 `from` 来自 `From` trait。

```rust
format!("{:?}", value)
dbg!(value)
```

要求类型实现 `Debug` trait。

比较两个值：

```rust
a == b
```

通常依赖 `PartialEq` trait。

复制一个简单值：

```rust
let y = x;
```

如果类型实现了 `Copy`，就会复制，而不是转移所有权。

常见 trait 有：

```text
Debug       可以调试打印
Display     可以面向用户格式化输出
Clone       可以显式复制
Copy        可以隐式按位复制
Iterator    可以逐个产生元素
From        可以从另一种类型转换而来
Into        可以转换成另一种类型
PartialEq   可以使用 ==
```

### `derive` 是自动实现 trait

例如：

```rust
#[derive(Debug)]
struct Person {
    name: String,
    age: u32,
}
```

这里：

```rust
#[derive(Debug)]
```

表示让编译器自动为 `Person` 实现 `Debug` trait。

于是可以：

```rust
let person = Person {
    name: String::from("Alice"),
    age: 20,
};

dbg!(person);
```

还可以一次自动实现多个：

```rust
#[derive(Debug, Clone, PartialEq)]
struct Person {
    name: String,
    age: u32,
}
```

### 和“继承”不一样

trait 不是说：

```text
Dog 是 Speak 的子类
```

而是说：

```text
Dog 具备 Speak 这种能力
```

一个类型可以实现很多 trait：

```rust
impl Speak for Dog {
    /* ... */
}
impl Run for Dog {
    /* ... */
}
impl Eat for Dog {
    /* ... */
}
```

所以 trait 更像给类型贴能力标签：

```text
Dog
├── Speak
├── Run
└── Eat
```

最值得先记住的一句话是：

> **struct 描述“它有什么数据”，trait 描述“它会做什么”。**

## 跟 go 差不多

对，核心感觉很像 Go 的 `interface`：

> 都是在描述“一个类型具备哪些行为”，函数可以只依赖这些行为，而不依赖具体类型。

Go：

```go
type Speaker interface {
    Speak()
}
```

Rust：

```rust
trait Speaker {
    fn speak(&self);
}
```

最大的区别是实现方式。

Go 是**隐式实现**：

```go
type Dog struct{}

func (Dog) Speak() {
    fmt.Println("汪")
}
```

只要方法集合符合，`Dog` 就自动实现了 `Speaker`，不需要声明。

Rust 是**显式实现**：

```rust
struct Dog;

impl Speaker for Dog {
    fn speak(&self) {
        println!("汪");
    }
}
```

必须明确写出：

```rust
impl Speaker for Dog
```

这种差别很符合两门语言的气质：

```text
Go：你长得像这个接口，就算实现了
Rust：你必须明确声明实现了这个 trait
```

Rust 的 trait 也比 Go interface 更“重型”，它还可以包含：

- 默认方法实现
- 关联类型
- 关联常量
- 泛型约束
- 运算符重载
- 为一整类类型统一实现 trait

例如：

```rust
fn print_it(value: &impl std::fmt::Debug) {
    dbg!(value);
}
```

很像 Go：

```go
func PrintIt(value interface{ /* 某些方法 */ }) {
}
```

但 Rust 中更常见的写法是把 trait 当作编译期泛型约束：

```rust
fn make_speak<T: Speaker>(value: &T) {
    value.speak();
}
```

所以可以先直接类比：

> **Rust trait ≈ 加强版、显式实现的 Go interface。**

你有 Go 基础的话，trait 的基本思想几乎不用重新学，真正需要适应的是 Rust 后面的泛型约束和 `dyn Trait`。

## Rust 实现 Copy Trait 的类型清单

### 一、基础标量类型（全部自动实现 Copy）

1. 所有整数：`u8/u16/u32/u64/u128/usize`、`i8/i16/i32/i64/i128/isize`
2. 浮点数：`f32`、`f64`
3. 布尔：`bool`
4. 字符：`char`
5. 单元类型：`()`

### 二、复合容器类型（内部所有元素都实现 Copy，则整体自动 Copy）

#### 1. 定长数组 `[T; N]`

只要 `T: Copy`，数组就自动实现 Copy

```rust
let a = [1,2,3];
let b = a; // 拷贝，a 仍可用
```

#### 2. 元组 `(T1, T2, ..., Tn)`

**每一个成员**都实现 Copy，元组才是 Copy；只要一个成员不 Copy，整体就不 Copy

- `(i32, bool, char)` ✅ Copy
- `(i32, String)` ❌ String 无 Copy，整个元组不能拷贝

### 三、引用类型 & 指针

1. **不可变引用 `&T`**：永远实现 Copy（不管 T 是什么）
   ```rust
   let s = String::new();
   let r1 = &s;
   let r2 = r1; // 拷贝引用，r1 仍有效
   ```
2. **原始指针 `*const T` / `*mut T`**：实现 Copy
3. **可变引用 `&mut T`**：❌ 不实现 Copy（独占借用规则）

### 四、其他标准库自带 Copy 类型

1. 范围类型：`Range`/`RangeFrom`/`RangeTo`/`RangeFull` 等区间
2. 枚举：无堆数据的简单枚举（仅标量成员）
3. `Option<T>`：若 `T: Copy`，则 `Option<T>` 自动 Copy
4. `Result<T, E>`：T 和 E 都 Copy 时，Result 实现 Copy
5. `NonZeroU8`/`NonZeroI32` 等非零整数
6. `PhantomData<T>`

### 五、绝对**不实现 Copy** 的常见类型（堆分配/独占所有权）

- `String`、`Vec<T>`、`Box<T>`、`HashMap`、`BTreeMap`
- 自定义包含堆数据的结构体
- `&mut T` 可变借用

### 六、核心规则

1. Copy 是隐式 trait：只要类型所有组成部分都 Copy，编译器**自动派生 Copy**；
2. Copy 必须先实现 Clone（Copy 是 Clone 的子集，拷贝成本极低）；
3. 实现 Copy 的变量赋值时**复制副本**，原变量继续有效；
   无 Copy 的类型赋值触发 **Move（所有权转移）**，原变量失效。

#### 示例对比

```rust
// i32 实现 Copy
let x = 10;
let y = x;
println!("{}", x); // 正常使用 ✅

// String 无 Copy，发生 move
let s1 = String::from("hi");
let s2 = s1;
// println!("{}", s1); 报错，s1 已失效 ❌
```

## 是不是压在栈上的类型是有 copy triat 的，在堆上的没有

不完全是。**`Copy` 和“数据在栈还是堆上”有关联，但不是判断标准。**

更准确地说：

> `Copy` 表示这个值可以安全地按位复制一份，而且复制后新旧两个值都能继续使用。

例如整数：

```rust
let a = 10;
let b = a;

dbg!(a, b);
```

`i32` 实现了 `Copy`，因此这里相当于复制了一份 `10`。

而 `String`：

```rust
let a = String::from("hello");
let b = a;

// dbg!(a); // 报错：a 已经被移动
```

一个 `String` 本身在栈上大致保存三个字段：

```text
指向堆内存的指针
字符串长度
内存容量
```

真正的字符内容在堆上。

如果直接按位复制 `String`，就会出现两个 `String` 指向同一块堆内存：

```text
a ──┐
    ├──> "hello"
b ──┘
```

最后 `a` 和 `b` 都会尝试释放同一块内存，产生重复释放。因此 `String` 不实现 `Copy`，赋值时只能转移所有权。

但不能简单概括成“栈上就是 `Copy`，堆上就不是”。

### 在栈上，也可能不是 `Copy`

例如我们可以定义一个只有整数的结构体：

```rust
struct Ticket {
    id: u32,
}
```

它的数据完全可以放在栈上，但它默认仍然不是 `Copy`：

```rust
let a = Ticket { id: 1 };
let b = a;

// dbg!(a.id); // a 已移动
```

只有显式声明：

```rust
#[derive(Copy, Clone)]
struct Ticket {
    id: u32,
}
```

它才成为 `Copy` 类型。

而且一个类型只要实现了 `Drop`，就不能实现 `Copy`，即使它的数据本身只是一个整数：

```rust
struct Ticket(u32);

impl Drop for Ticket {
    fn drop(&mut self) {
        println!("ticket 被销毁");
    }
}
```

因为 Rust 必须保证这份资源只被销毁一次。

### 指向堆的东西，也可能是 `Copy`

引用：

```rust
let s = String::from("hello");

let a = &s;
let b = a;

dbg!(a, b);
```

这里 `a` 和 `b` 都是 `&String`，共享引用本身实现了 `Copy`。

它们可能指向包含堆内存的 `String`，但复制的只是引用，不是复制或接管那个 `String`。

所以：

```text
String    不 Copy
&String   Copy
```

原始指针 `*const T`、`*mut T` 也通常是 `Copy`，虽然它们可能指向堆上的数据。

### 常见规律

通常实现 `Copy` 的有：

```text
整数、浮点数
bool、char
共享引用 &T
原始指针
函数指针
成员全是 Copy 的元组、数组和结构体
```

通常不实现 `Copy` 的有：

```text
String
Vec<T>
Box<T>
Rc<T>
文件句柄
锁守卫
需要负责释放资源的类型
```

元组是否 `Copy`，取决于内部成员：

```rust
let a = (1, true); // i32 和 bool 都 Copy，所以整个元组 Copy
let b = a;
dbg!(a, b);
```

但：

```rust
let a = (1, String::from("hello"));
let b = a;

// dbg!(a); // 已移动
```

因为其中的 `String` 不 `Copy`，所以整个元组也不 `Copy`。

最合适的记法是：

> **`Copy` 不取决于值放在哪里，而取决于“简单复制这几个字节后，两个值能否各自安全地继续存在”。**

堆资源通常需要唯一所有者，所以拥有堆资源的类型往往不 `Copy`；这只是常见结果，不是定义。

## 那 move 属性的函数只能用一次就消失了？

不是函数只能用一次，而是：

> **一个不实现 `Copy` 的值，被按值传进函数后，原变量的所有权转移了，因此原变量不能再用。函数本身仍然可以反复调用。**

例如：

```rust
fn consume(s: String) {
    println!("{s}");
}

let s = String::from("hello");

consume(s);
consume(s); // 报错：s 的所有权已经转移
```

第一次调用时：

```text
s ──所有权转移──> consume 函数
```

函数结束后，如果没有把这个 `String` 再传出去，它就会被销毁并释放堆内存。

但函数没有消失，你可以给它另一个值：

```rust
consume(String::from("hello"));
consume(String::from("world"));
```

### 想继续使用原值：传引用

```rust
fn inspect(s: &String) {
    println!("{s}");
}

let s = String::from("hello");

inspect(&s);
inspect(&s);
dbg!(s);
```

这里函数只是借用：

```text
inspect(&s) → 借来看一下
consume(s)  → 把所有权交出去
```

### 想把所有权还回来：返回它

```rust
fn inspect_and_return(s: String) -> String {
    println!("{s}");
    s
}

let s = String::from("hello");
let s = inspect_and_return(s);

dbg!(s);
```

虽然所有权进入了函数，但函数又把它返回了。

### 方法也是同样规则

```rust
fn method(self)        // 消耗对象
fn method(&self)       // 只读借用
fn method(&mut self)   // 可变借用
```

例如方法参数是 `self`：

```rust
struct User {
    name: String,
}

impl User {
    fn consume(self) {
        println!("{}", self.name);
    }
}

let user = User {
    name: String::from("Alice"),
};

user.consume();
// user 不能再使用
```

所以准确的说法是：

> **按值接收非 `Copy` 类型的函数，会消耗传进去的那个值；不是函数本身只能用一次。**

另外，`move` 闭包是另一回事：它表示闭包取得捕获变量的所有权，但闭包是否只能调用一次，还取决于它是 `Fn`、`FnMut` 还是 `FnOnce`。

## 是不是说我默认你用完就释放，你不说我就释放，你要想继续使用，你得另外跟我说

对，**这个直觉已经很接近了**。可以把 Rust 想成在说：

> 这份资源现在归你。你离开作用域时，我默认帮你释放。  
> 你把它交给别人后，就不再归你；别人用完时再负责释放。

例如：

```rust
fn consume(s: String) {
    println!("{s}");
}

let s = String::from("hello");
consume(s);
```

过程是：

```text
s 创建 String
→ s 拥有它
→ consume(s) 把所有权交给函数参数
→ 函数结束
→ 参数离开作用域
→ String 自动释放
```

所以这之后不能再写：

```rust
println!("{s}");
```

因为不是“`s` 已经被释放了”这么简单，而是：

> `s` 已经不再拥有那个值了，Rust 禁止你继续通过 `s` 使用它。

你说的“要想继续使用，就得另外说”，主要有三种说法。

#### 只借给函数用一下

```rust
fn inspect(s: &String) {
    println!("{s}");
}

let s = String::from("hello");

inspect(&s);
println!("{s}");
```

`&s` 的意思是：

> 给你看看，但所有权还是我的。

#### 真的复制一份

```rust
let a = String::from("hello");
let b = a.clone();

println!("{a}");
println!("{b}");
```

这时堆上有两份独立内容，两边都能继续用。

#### 交出去，再让函数还回来

```rust
fn use_and_return(s: String) -> String {
    println!("{s}");
    s
}

let s = String::from("hello");
let s = use_and_return(s);

println!("{s}");
```

所以最简洁的总结是：

> **默认规则不是“用一次就立即释放”，而是“当前所有者离开作用域时自动释放”。如果把所有权交出去，原变量就不能继续用；想继续用，就借用、复制，或者把所有权再拿回来。**

## 什么叫所有者离开作用域

“所有者离开作用域”就是：

> **拥有这个值的变量，走出了它所在的那对大括号 `{}`。**

例如：

```rust
{
    let s = String::from("hello");
    println!("{s}");
} // 到这里，s 离开作用域
```

执行过程是：

```text
进入 {
→ 创建 s
→ s 拥有 String
→ 使用 s
→ 执行到 }
→ s 不再存在
→ String 的堆内存自动释放
```

这里 `{ ... }` 就是一个作用域。出了右大括号，里面定义的变量就不能再用了：

```rust
{
    let s = String::from("hello");
}

println!("{s}"); // 报错：找不到 s
```

函数体也是一个作用域：

```rust
fn test() {
    let s = String::from("hello");
} // 函数结束，s 离开作用域，String 被释放
```

因此“离开作用域”通常发生在：

```text
执行到右大括号 }
函数 return
函数自然执行结束
```

### 如果中途发生 move

```rust
{
    let a = String::from("hello");
    let b = a;

    println!("{b}");
}
```

这里所有权变化是：

```text
创建后：a 是所有者
let b = a：所有权从 a 移动给 b
离开作用域时：b 是所有者
所以由 b 负责释放
```

`a` 虽然变量名所在的作用域还没有结束，但它已经不是所有者了：

```rust
let a = String::from("hello");
let b = a;

println!("{a}"); // 不允许，a 已失去所有权
```

所以要区分两件事：

```text
变量离开作用域
→ 变量本身消失

所有权被 move
→ 变量名可能还在作用域里，但已经没有权利使用原值
```

可以把作用域理解成变量的“活动区域”：

```rust
{
    let s = String::from("hello");
    // s 从这里开始有效

    println!("{s}");

    // 到这里为止有效
}
```

只要 `s` 到结束时仍然拥有那个 `String`，Rust 就在 `}` 处自动释放它。Evcxr 顶层比较特殊，它会尝试把变量保留到后续代码框，所以顶层变量的作用域不像普通函数那样很快结束。

## 感觉不是所有者离开，所有者不是代码吗？不是死的吗？是程序执行行离开的感觉呢怎么是

对，你这个感觉更准确。**“所有者离开作用域”只是 Rust 教材里的简写，严格说不是代码自己走了，而是程序执行流程离开了那个作用域。**

比如：

```rust
{
    let s = String::from("hello");
    println!("{s}");
}
```

代码当然一直静静地写在那里，并没有移动。真正发生的是：

```text
程序执行进入 {
→ 创建运行时变量 s
→ s 拥有一个 String
→ 执行 println!
→ 执行流程越过 }
→ 这一轮运行中的 s 被销毁
→ String 被释放
```

所以更准确的说法应该是：

> 当程序执行离开变量所在的作用域时，这个变量对应的运行时实例会被销毁。

### “所有者”不是那一行代码

这一行：

```rust
let s = String::from("hello");
```

是静态的源代码。

程序运行时，每执行一次这行代码，才会创建一个实际的变量实例：

```text
源代码中的 s
        ↓ 执行
运行时产生一个 s
        ↓
这个 s 拥有某个 String
```

例如函数调用两次：

```rust
fn test() {
    let s = String::from("hello");
}

test();
test();
```

源码中只有一个 `s`，但运行时实际上先后创建了两个不同的 `s`：

```text
第一次调用：
s₁ 拥有 String₁
函数结束 → 销毁 s₁

第二次调用：
s₂ 拥有 String₂
函数结束 → 销毁 s₂
```

因此“所有者”更像是：

> 某一次程序运行过程中产生的变量绑定。

而不是源代码文本本身。

### 你说的“程序执行行离开”更接近本质

不过也不能只理解成“执行完某一行”，因为变量通常会活过很多行：

```rust
{
    let s = String::from("hello"); // s 从这里开始存在

    println!("{s}");
    println!("{s}");
    println!("{s}");

} // 执行到这里，s 才被销毁
```

不是执行完 `let` 那一行，`s` 就消失，而是执行流程离开整个 `{ ... }` 后才消失。

所以可以这样表述：

```text
作用域 = 变量允许存在的一段代码区域
执行进入作用域 = 可能创建变量实例
执行离开作用域 = 销毁这一轮运行中的变量实例
```

### `move` 又是另一件事

```rust
{
    let a = String::from("hello");
    let b = a;

    println!("{b}");
}
```

执行到：

```rust
let b = a;
```

后，运行时关系变成：

```text
原来：a 拥有 String
后来：b 拥有 String
```

此时程序还没离开大括号，但 `a` 已经不再拥有那个值。

到了右大括号：

```text
执行流程离开作用域
→ b 被销毁
→ b 所拥有的 String 被释放
```

所以你可以把教材那句话替换成更精确的一句：

> **当执行流程离开某个作用域时，该作用域中仍然有效的变量会被销毁；如果变量此时仍拥有资源，该资源也会随之释放。**

你的直觉没错：“离开”的主体不是代码，也不是一个人格化的所有者，而是**程序的执行流程**。

## 不是执行完那一行消失，其实感觉是执行完一个作用域，这里面没跟我声明要保留的我全清理掉

对，你这个感觉已经很接近本质了：

> **程序执行完一个作用域后，这个作用域里创建的局部变量都会失效；其中仍然拥有资源的变量，会负责清理自己的资源。**

例如：

```rust
{
    let a = String::from("A");
    let b = String::from("B");

    println!("{a} {b}");
} // 执行到这里，清理 b，再清理 a
```

大致可以想象成编译器替你安排了：

```rust
drop(b);
drop(a);
```

通常按创建顺序的反方向销毁。

不过“没声明要保留”需要稍微改成：

> **没有把值移动到更长寿命的位置，它就会随当前作用域一起销毁。**

Rust 不是写一个 `keep` 来保留，而是通过**所有权转移**把值带出作用域。

例如返回出去：

```rust
fn create_string() -> String {
    let s = String::from("hello");
    s
}

let result = create_string();
```

函数作用域结束时：

- 局部变量名 `s` 消失；
- 但字符串没有被清理；
- 因为所有权已经从 `s` 移动给了外面的 `result`。

过程是：

```text
函数内部 s 拥有 String
        ↓ 返回 s
外面的 result 接过所有权
        ↓
函数作用域结束，但资源已有新主人
```

也可以移动给外层已有的数据结构：

```rust
let mut values = Vec::new();

{
    let s = String::from("hello");
    values.push(s);
} // s 消失，但 String 已经归 values 所有
```

所以作用域结束时，Rust 实际处理的是：

```text
局部变量绑定全部失效
├─ 仍拥有资源的变量：执行 drop，清理资源
├─ 已经 move 出去的变量：这里没有资源可清理
└─ 只是引用的变量：引用失效，但不会清理它所指向的资源
```

例如：

```rust
let s = String::from("hello");

{
    let r = &s;
    println!("{r}");
} // 只清理引用 r，不清理 s 所拥有的字符串

println!("{s}"); // s 仍然有效
```

因此你可以把 Rust 的作用域想成一个自动资源管理盒子：

```text
进入作用域
→ 创建局部资源
→ 使用、借用或把所有权移出去
→ 离开作用域
→ 清理仍留在盒子里的资源
```

这也是 RAII 的核心思想：**不需要手动记住释放，作用域结束自动结账；想把某个资源带走，就把所有权移出去。**

## 不是 keep，keep 没有主人，必须得声明谁拥有，这样我才知道那个所有者在哪，我执行完所有所有者所在的作用域我就知道要清理了

对，你这次说得更准确。

Rust 不是在问：

> “这个值要不要保留？”

而是在问：

> **“这个值现在归谁所有？”**

只要始终能确定唯一的所有者，Rust 就能确定什么时候清理：

```text
找到当前所有者
→ 确定所有者的生命周期
→ 所有者被销毁
→ 它拥有的资源随之清理
```

例如：

```rust
{
    let s = String::from("hello");
}
```

关系是：

```text
s ──拥有──> String ──拥有──> 堆内存
```

执行流程离开这个作用域后，`s` 被销毁，于是它拥有的 `String` 也被销毁，堆内存被释放。

而发生移动时：

```rust
{
    let a = String::from("hello");
    let b = a;
}
```

所有权关系发生了变化：

```text
最初：

a ──拥有──> String

执行 let b = a 之后：

b ──拥有──> String
a ──不再拥有任何值
```

最后退出作用域时，由 `b` 清理这个字符串。

如果移到外层：

```rust
let result;

{
    let s = String::from("hello");
    result = s;
}

println!("{result}");
```

关系是：

```text
原来：s 拥有 String
后来：result 拥有 String
```

里面的作用域结束时，`s` 消失，但它已经不是所有者，所以不会清理字符串。等 `result` 所在的外层作用域结束，字符串才会被清理。

所有者也不一定是一个简单变量，还可能是容器：

```rust
let mut values = Vec::new();

{
    let s = String::from("hello");
    values.push(s);
}
```

此时：

```text
s
↓ move
values 中的某个元素
↓ 拥有
String
```

所以内部作用域结束后字符串仍然存在。将来 `values` 被销毁时，它会依次销毁自己拥有的元素。

因此可以把所有权系统理解成一条清晰的责任链：

```text
谁拥有资源
→ 谁负责保证它有效
→ 谁被销毁时负责释放它
```

而借用：

```rust
let r = &s;
```

只是：

```text
r ──暂时引用──> s 所拥有的值
```

`r` 不是主人，所以 `r` 消失时不会释放那个值。

你现在这句话已经抓到了所有权的核心：

> **资源必须始终有明确的主人；Rust 跟踪主人在哪里，当拥有它的那个位置被销毁时，就能确定资源应当被清理。**

## 这会增加心智负担吗？感觉很不稳定的感觉

会增加一点心智负担，尤其刚开始时。你会不断想：

> 这个值现在归谁？传进去以后还能不能用？这里应该 move、借用，还是 clone？

但它带来的通常不是“不稳定”，而是恰好相反：**资源变化非常确定，只是规则被暴露出来了。**

比如：

```rust
let a = String::from("hello");
let b = a;
```

所有权只会在明确写出的地方变化：

```text
创建 a       → a 拥有 String
let b = a    → 所有权明确移动给 b
离开作用域   → b 清理 String
```

它不会在后台随机搬动，也不会突然被垃圾回收器收走。编译器还禁止你在移动后继续使用 `a`：

```rust
println!("{a}"); // 编译期直接报错
```

所以这种“所有者不断变化”的感觉，实际上是一条静态、可追踪的责任链。

### 心智负担主要出现在写代码时

在 Python、Go、Java 里，你常常可以先不想资源关系：

```text
先把引用到处传
→ 程序照样运行
→ 以后再由 GC 处理
```

Rust 要你提前讲清楚：

```text
这是交出所有权
这是只读借用
这是可变借用
这是明确复制
```

所以 Rust 是：

> **写的时候更费脑，运行时更省心。**

很多原本可能在运行时才暴露的问题，会变成编译错误，例如：

- 使用已经释放或移动的值；
- 悬空引用；
- 一边读取、一边不安全地修改；
- 多线程数据竞争；
- 重复释放。

### 实际上不用每行都想所有权

普通代码的数据流很直时：

```rust
let text = String::from("hello");
let length = text.len();
println!("{text}: {length}");
```

几乎感觉不到所有权。

你主要记住三个动作就够了：

```rust
use_value(s);      // 按值传入：可能交出所有权
use_value(&s);     // 共享借用：看一下
change_value(&mut s); // 可变借用：暂时修改
```

真的需要独立副本时：

```rust
let copy = s.clone();
```

### 为什么你现在觉得“不稳定”

因为你刚刚开始把变量理解成“资源的主人”，于是会觉得主人似乎随时在换。

但 Rust 的规则其实非常机械：

```text
赋值给另一个变量
传给按值接收的函数
返回出函数
放进容器
```

这些地方可能发生 move。除此之外，值不会自己换主人。

而借用：

```rust
let r = &s;
```

又不会改变主人：

```text
s 仍然拥有 String
r 只是暂时使用
```

因此，熟悉后反而会有一种很强的稳定感：

> 看函数签名，大致就知道它会不会拿走、修改或仅仅读取我的值。

例如：

```rust
fn consume(s: String)       // 拿走
fn inspect(s: &String)      // 只看
fn modify(s: &mut String)   // 暂时修改
```

这比很多语言中“函数内部到底会不会保存、修改或共享这个对象”更透明。

所以你的感觉可以总结成：

> **所有权确实把一部分运行时复杂性搬到了程序员和编译器面前；刚开始像增加了不稳定感，熟悉后通常会变成更强的可预测感。**

## & 是什么？

在你现在这个所有权语境里，`&` 表示：

> **借用一个值，不取得它的所有权。**

例如：

```rust
let s = String::from("hello");
let r = &s;
```

此时关系是：

```text
s ──拥有──> String
r ──借用──> String
```

`r` 是一个引用，类型是：

```rust
&String
```

因为只是借用，所以 `s` 仍然可以继续使用：

```rust
let s = String::from("hello");
let r = &s;

println!("{r}");
println!("{s}");
```

### `&` 出现在两个位置

#### 在值前面：创建引用

```rust
&s
```

意思是：

> 借用 `s` 指向的值。

#### 在类型前面：表示引用类型

```rust
fn inspect(s: &String) {
    println!("{s}");
}
```

这里 `s: &String` 表示：

> 参数 `s` 是对一个 `String` 的引用，不接管 `String` 的所有权。

调用时：

```rust
let text = String::from("hello");

inspect(&text);
inspect(&text);

println!("{text}");
```

函数可以反复借用，`text` 不会被 move。

对比一下：

```rust
fn consume(s: String) {
    println!("{s}");
}
```

调用：

```rust
consume(text);
```

是把所有权交给函数。

而：

```rust
fn inspect(s: &String) {
    println!("{s}");
}
```

调用：

```rust
inspect(&text);
```

只是暂时借给函数看。

### `&mut`：可变借用

普通 `&` 只能读，不能修改：

```rust
fn inspect(s: &String) {
    println!("{s}");
}
```

要修改，需要：

```rust
fn modify(s: &mut String) {
    s.push_str(" world");
}

let mut text = String::from("hello");

modify(&mut text);

println!("{text}");
```

可以这样记：

```text
String       → 把值交出去
&String      → 借来看，不许修改
&mut String  → 借去修改，之后归还
```

所以 `&` 本质上是在对 Rust 说：

> **别拿走所有权，我只是暂时给你一个访问入口。**

## go 里是什么来着，取指针吗？

对，Go 里 `&x` 是**取地址**，得到指向 `x` 的指针。

```go
x := 10
p := &x
```

此时：

```go
p  // 类型是 *int
*p // 通过指针取得 x 的值
```

Rust 写法很像：

```rust
let x = 10;
let r = &x;
```

此时：

```rust
r  // 类型是 &i32
*r // 解引用，取得 x 的值
```

但概念上有一点区别：

| Go                     | Rust                             |
| ---------------------- | -------------------------------- |
| `&x` 得到指针 `*T`     | `&x` 得到引用 `&T`               |
| 指针可以是 `nil`       | 普通引用不能是空引用             |
| 生命周期主要由 GC 管理 | 生命周期由借用检查器在编译期检查 |
| 可以长期保存指针       | 引用不能比被引用的值活得更久     |
| `&x` 常理解为取地址    | `&x` 更强调借用                  |

例如 Go：

```go
func inspect(s *string) {
    fmt.Println(*s)
}

text := "hello"
inspect(&text)
```

Rust：

```rust
fn inspect(s: &String) {
    println!("{s}");
}

let text = String::from("hello");
inspect(&text);
```

所以可以先类比：

```text
Go:   &x → 取得指向 x 的指针
Rust: &x → 创建对 x 的借用引用
```

底层都包含地址信息，但 Rust 的 `&` 额外带着严格的借用和生命周期规则。

## rust 本质也是取指针吗？

从底层实现看，**很多时候可以把 Rust 的 `&x` 理解成“取得指向 `x` 的地址”**；但从语言语义看，它不只是普通指针，而是一个受严格约束的**引用**。

```rust
let x = 10;
let r = &x;
```

这里：

```rust
r: &i32
```

运行时通常会保存 `x` 的地址，所以在机器层面很像指针：

```text
r ──> x
```

也可以解引用：

```rust
let y = *r;
```

### 为什么不直接叫指针

因为 `&T` 比普通地址多了很多保证：

```text
&T
├── 不能为空
├── 必须指向有效的 T
├── 必须正确对齐
├── 不能比原值活得更久
└── 存在期间遵守借用规则
```

例如：

```rust
let r: &i32;
{
    let x = 10;
    r = &x;
}

println!("{r}");
```

这会被编译器拒绝，因为 `x` 已经消失，`r` 会变成悬空引用。

### `&T` 和 `&mut T`

```rust
&x
```

是共享引用：可以读，通常不能通过它修改。

```rust
&mut x
```

是可变引用：可以修改，但在它有效期间，Rust 要求它对这份数据具有排他的访问权。

```rust
let mut x = 10;
let r = &mut x;

*r += 1;
```

这种排他规则是普通指针本身不会自动保证的。

### Rust 也有真正的裸指针

```rust
*const T
*mut T
```

例如：

```rust
let x = 10;
let p: *const i32 = &x;
```

裸指针更接近 Go/C 里“地址”的概念：

- 可以为空；
- 编译器不替你检查生命周期；
- 解引用通常需要 `unsafe`；
- 更容易形成悬空指针。

所以可以这样记：

```text
&x          = 取得地址，并建立一个受编译器保护的借用
*const T    = 普通只读裸指针
*mut T      = 普通可变裸指针
```

还有一个小细节：引用不一定永远只是一个机器地址。比如 `&str` 和 `&[T]` 通常是“胖指针”，里面同时保存：

```text
数据地址 + 长度
```

因此最准确的一句话是：

> **Rust 的引用底层通常以指针实现，但语义上是带生命周期、有效性和别名规则的安全指针。**

## 那函数都临时借用就可以了是吧，如果不修改值的话

对，**如果函数只是读取数据，不需要拿走所有权，也不需要修改，通常就传不可变引用 `&T`。**

```rust
fn print_text(s: &String) {
    println!("{s}");
}

let s = String::from("Hello");

print_text(&s);
print_text(&s);
println!("{s}");
```

函数只是临时借用，调用结束后借用结束，`s` 仍然归原变量所有。

不过对字符串，更惯用的是接收 `&str`：

```rust
fn print_text(s: &str) {
    println!("{s}");
}

let s = String::from("Hello");

print_text(&s);
print_text("World");
```

因为 `&str` 既能接收 `String` 的借用，也能接收字符串字面量，范围更广。

可以先这样选：

```rust
fn read(x: &T)       // 只读取，临时借用
fn modify(x: &mut T) // 需要修改，临时可变借用
fn consume(x: T)     // 需要拿走、保存或销毁这个值
```

但小型 `Copy` 类型通常直接按值传，不必借用：

```rust
fn double(x: i32) -> i32 {
    x * 2
}
```

因为复制一个 `i32` 比传引用更简单，原变量也不会失效。

所以不是“所有函数都借用”，而是：

> **大对象只读时优先借用；需要取得所有权时按值传；小型 `Copy` 类型通常直接按值传。**

## 字符串先转成字面量？那怎么在里面操作？

不是“把 `String` 转成字符串字面量”，而是：

> `&str` 是对一段字符串内容的只读视图；字符串字面量只是 `&str` 的一种来源。

例如：

```rust
let s = String::from("Hello");
print_text(&s);
```

这里 `&s` 原本是 `&String`，Rust 会自动把它转换为指向字符串内容的 `&str`：

```text
String
  ↓ 借用其中的文本内容
&str
```

并没有重新创建 `"Hello"`，也没有复制字符串。

### 在 `&str` 里面可以做只读操作

```rust
fn inspect(s: &str) {
    println!("内容：{s}");
    println!("字节长度：{}", s.len());
    println!("是否包含 ell：{}", s.contains("ell"));

    for word in s.split_whitespace() {
        println!("{word}");
    }
}
```

调用时，两种都可以：

```rust
let text = String::from("Hello Rust");

inspect(&text);       // 借用 String
inspect("Hello Rust"); // 字符串字面量本来就是 &str
```

这就是为什么参数写 `&str` 比 `&String` 更通用。

### 不能通过 `&str` 修改原字符串

下面不行：

```rust
fn modify(s: &str) {
    s.push_str(" world"); // 报错
}
```

因为 `&str` 只是不可变借用，而且它本身也没有 `push_str` 这种扩容方法。

需要修改原来的字符串，就接收 `&mut String`：

```rust
fn modify(s: &mut String) {
    s.push_str(" world");
}

let mut text = String::from("Hello");

modify(&mut text);

println!("{text}"); // Hello world
```

所以可以这样区分：

```rust
fn inspect(s: &str) {
    // 读取、查找、切分、分析
}

fn modify(s: &mut String) {
    // 修改、追加、删除
}

fn consume(s: String) {
    // 取得整个 String 的所有权
}
```

还有一种情况：函数不修改原值，而是根据它生成新字符串：

```rust
fn shout(s: &str) -> String {
    s.to_uppercase()
}

let text = String::from("hello");
let result = shout(&text);

println!("{text}");   // hello
println!("{result}"); // HELLO
```

这里函数借用原字符串进行处理，然后返回一个新的 `String`。

## usize 是干什么的

`usize` 是 Rust 里专门用来表示：

> **内存大小、元素数量和索引位置的无符号整数类型。**

例如：

```rust
let arr = [10, 20, 30];

let i: usize = 1;
println!("{}", arr[i]);
```

数组索引要求是 `usize`。

### 为什么不用 `u32` 或 `u64`

因为 `usize` 的位数会跟当前计算机的寻址宽度一致：

```text
32 位系统：usize 是 32 位
64 位系统：usize 是 64 位
```

你的工具链是：

```text
x86_64-pc-windows-msvc
```

因此你电脑上的：

```rust
std::mem::size_of::<usize>()
```

结果是 `8` 字节，也就是 64 位。

它能表示当前机器理论上可以寻址的内存位置范围，所以很适合表示：

- 数组和切片索引
- 容器长度
- 内存大小
- 指针偏移量

### `.len()` 返回的就是 `usize`

```rust
let s = String::from("hello");
let length = s.len();
```

这里：

```rust
length: usize
```

数组、字符串、`Vec` 的长度也通常都是 `usize`：

```rust
let values = vec![1, 2, 3];
let n: usize = values.len();
```

因为长度不可能为负数，而且需要能够覆盖当前机器可寻址的最大容量。

### `usize` 是无符号整数

所以它只能表示：

```text
0, 1, 2, 3, ...
```

不能表示负数：

```rust
let x: usize = -1; // 报错
```

对应的有符号版本叫：

```rust
isize
```

```text
usize → 无符号、机器字长整数
isize → 有符号、机器字长整数
```

不过 `isize` 的使用频率一般明显低于 `usize`。

### 常见的类型不匹配

例如：

```rust
let i: i32 = 1;
let arr = [10, 20, 30];

// arr[i] // 报错
```

因为数组索引需要 `usize`，可以转换：

```rust
println!("{}", arr[i as usize]);
```

但实际代码中，索引变量通常从一开始就写成 `usize`：

```rust
let i: usize = 1;
println!("{}", arr[i]);
```

循环范围也经常自然推断成 `usize`：

```rust
for i in 0..arr.len() {
    println!("{}", arr[i]);
}
```

这里 `arr.len()` 是 `usize`，所以 `i` 也会被推断为 `usize`。

可以先记成一句：

> **`usize` 就是 Rust 用来数“有多少个”和表示“第几个”的机器原生无符号整数。**

## 只有 usize, isize 吗？

对，**Rust 的“跟机器指针宽度走”的整数类型只有两个**：

```rust
usize // 无符号，主要用于长度、索引、内存大小
isize // 有符号，主要用于指针偏移等场景
```

其他整数类型都是固定宽度：

```rust
i8   i16   i32   i64   i128
u8   u16   u32   u64   u128
```

浮点数也只有固定宽度：

```rust
f32
f64
```

所以整个原生数值类型可以看成：

```text
有符号整数：i8 i16 i32 i64 i128 isize
无符号整数：u8 u16 u32 u64 u128 usize
浮点数：f32 f64
```

在你的 64 位电脑上：

```rust
std::mem::size_of::<usize>() // 8 字节
std::mem::size_of::<isize>() // 8 字节
```

但在 32 位系统上，它们会变成 4 字节。

`usize` 很常见；`isize` 相对少见。普通业务整数通常还是用 `i32`、`i64`，索引和长度才用 `usize`。

## 格式化的时候为什么自动把 for 循环大括号后面的分号去掉？

因为 `for` 在 Rust 里属于**块表达式**，放在语句位置时，右大括号 `}` 本身已经足以表示它结束了，所以后面的分号是多余的。

你写：

```rust
for (i, item) in s.as_bytes().iter().enumerate() {
    dbg!(i, item);
};
```

`rustfmt` 会整理成惯用写法：

```rust
for (i, item) in s.as_bytes().iter().enumerate() {
    dbg!(i, item);
}
```

两者执行效果相同。

普通表达式作为语句时一般需要分号：

```rust
dbg!(item);
do_something();
1 + 2;
```

但这些带大括号的控制流表达式，作为独立语句时通常不需要：

```rust
if condition {
    // ...
}

while condition {
    // ...
}

for item in items {
    // ...
}

loop {
    break;
}

match value {
    _ => {}
}
```

而且 `for` 循环的结果固定是单元值 `()`，不像普通 `loop` 可以通过 `break value` 得到值，所以加分号也没有什么值需要丢弃：

```rust
let result = for x in 0..3 {
    dbg!(x);
};

// result 是 ()
```

对比可以返回值的 `loop`：

```rust
let result = loop {
    break 10;
};
```

这里整个 `loop` 的值是 `10`。若单独写：

```rust
loop {
    break 10;
};
```

分号会表示丢弃这个 `10`。

所以 `rustfmt` 去掉 `for` 后的分号，是因为它判断这是**合法但冗余、不符合惯用格式**的分号。
