# 5

## 为什么在定义函数的后面不能加分号？

因为 Rust 里的**函数定义属于“项（item）”**，它不是一条需要用分号结束的普通语句。

```rust
fn build_user(username: String, email: String) -> User {
    User {
        username: username,
        email: email,
    }
}
```

函数定义的结构已经由最外层的 `{ ... }` 完整结束了：

```rust
fn 函数名(...) -> 返回类型 {
    函数体
}
```

所以后面不需要再写 `;`。

可以把它和变量语句对比：

```rust
let x = 10;
```

`let` 是语句，需要分号表示结束。

而这些都是“项”，一般不加分号：

```rust
struct User {
    username: String,
    email: String,
}

fn build_user(...) -> User {
    ...
}

enum Status {
    Active,
    Disabled,
}
```

### 函数体最后为什么也不能加分号？

这里还有另一层原因：

```rust
User {
    username: username,
    email: email,
}
```

这是函数的**尾表达式**，它产生一个 `User`，作为函数返回值。

相当于：

```rust
return User {
    username: username,
    email: email,
};
```

如果给它加分号：

```rust
fn build_user(username: String, email: String) -> User {
    User {
        username: username,
        email: email,
    };
}
```

分号会把 `User { ... }` 的值丢弃，于是函数实际返回 `()`，但函数签名要求返回 `User`，因此报错：

```text
expected `User`, found `()`
```

所以这里有两个不同的位置：

```rust
fn build_user(...) -> User {
    // 函数定义本身不需要分号
    User {
        // 尾表达式不加分号，作为返回值
        username,
        email,
    }
}
```

顺便说，你的字段名和变量名相同时，可以简写：

```rust
fn build_user(username: String, email: String) -> User {
    User { username, email }
}
```

最核心的区别是：

> 函数外面的 `}` 表示函数定义结束；函数里面最后一个表达式不加分号，表示把它的值返回。

## 我是不知道还有哪些值，想知道还有哪些值，这样我知道怎么处理么

啊，你要的不是“查看某个变量”，而是：

> **列出 Rust REPL 当前还保存着哪些变量。**

你用的 Zed Rust REPL 如果是 **Evcxr**，直接运行：

```rust
:vars
```

它会列出当前上下文中保存的变量名和类型：

```text
user: User
email: String
x: i32
```

Evcxr 官方就是用 `:vars` 查看当前上下文中仍然存在的变量。

然后再单独查看某个值：

```rust
dbg!(&user);
```

或者在 REPL 中直接运行变量名：

```rust
user
```

例如：

```rust
let username = String::from("ziru");
let age = 18;
```

运行：

```rust
:vars
```

得到：

```text
age: i32
username: String
```

再运行：

```rust
username
```

就能看到：

```text
"ziru"
```

因此你的使用流程可以是：

```rust
:vars
```

先确认“我现在手里还有什么变量”，再根据类型决定：

```rust
dbg!(&变量名);
```

查看具体内容。

需要全部清空重新开始时运行：

```rust
:clear
```

它会清除当前 REPL 的所有状态。

不过 `:vars` 只列出**变量名和类型**，不会一次打印所有变量的具体值。Rust 的不同类型未必都实现了 `Debug`，因此 Evcxr 不能像 Python 的 `locals()` 那样安全地把所有值统一打印出来。

对。**`:vars` 不是 Rust 代码，而是 Evcxr 的 REPL 命令**，最好放在一个单独运行的代码框里，而且后面不要加分号。官方示例也是把它作为一次独立输入执行。

这样会报错：

```rust
let x = 10;
let y = 20;

:vars;
```

因为整个代码框被当成 Rust 代码编译，Rust 编译器看到 `:` 就会报：

```text
expected expression, found `:`
```

正确写法是分成两个代码框。

第一个代码框：

```rust
let x = 10;
let y = 20;
```

运行后，再单独写一个：

```rust
:vars
```

注意：

```rust
:vars
```

不是：

```rust
:vars;
```

Evcxr 会列出当前持久化变量：

```text
x: i32
y: i32
```

同理，这些命令也建议单独放：

```rust
:clear
```

```rust
:help
```

```rust
:type x
```

原因可以理解成：

- `let x = 10;` 是交给 **Rust 编译器**的。
- `:vars` 是交给 **Evcxr REPL 控制器**的。
- 两者虽然都写在 Rust REPL 中，但不是同一种语言语法。

所以在 Zed Markdown 里，可以专门留一个小代码框：

````markdown
```rust
:vars
```
````

需要查看当前环境时就单独运行它。

## 结构体更新语法

`..` **没有一个统一叫“扩展运算符”的官方名称**，要看它出现在哪里。

你刚才这个场景：

```rust
let user2 = User {
    email: String::from("new@example.com"),
    ..user1
};
```

官方《Rust 程序设计语言》叫：

> **struct update syntax**，结构体更新语法。

Rust Reference 里更正式的名称是：

> **functional update syntax**，函数式更新语法。

意思是：

```rust
..user1
```

用 `user1` 补齐前面没有明确填写的字段。

而在不同位置，名称不同：

```rust
0..10
```

叫 **range expression**，范围表达式。

```rust
let User { username, .. } = user;
```

这里的 `..` 通常叫 **rest pattern**，剩余模式，表示忽略其他字段。

所以你可以这样记：

| 写法                      | 官方名称             |
| ------------------------- | -------------------- |
| `Struct { field, ..old }` | struct update syntax |
| `0..10`                   | range expression     |
| `Struct { field, .. }`    | rest pattern         |

它不像 JavaScript 的 `...` 那样统一称作 **spread syntax**。Rust 里的 `..` 是一个会根据上下文改变含义的语法符号。

## 那 copy 如何清理？还是说它只占用栈一般不需要清理？

对，基本可以这么理解：**`Copy` 类型通常不拥有需要释放的资源，所以不需要“清理”。**

```rust
let x = 10;
dbg!(x);
println!("{x}");
```

`dbg!(x)` 拿到的是一份复制，原来的 `x` 还在。

像这些常见 `Copy` 类型：

```rust
i32
f64
bool
char
&T
```

它们通常只是固定大小的数据，可能放在栈上，也可能直接放在 CPU 寄存器里。作用域结束后：

- 栈空间整体被回收；
- 寄存器以后会被别的值覆盖；
- 不需要运行析构函数；
- 也没有堆内存需要释放。

```rust
{
    let x = 10;
    let y = x;
} // x 和 y 在这里结束，不需要做特殊清理
```

### `Copy` 类型不能拥有需要析构的东西

Rust 有一个重要规则：

> 实现了 `Copy` 的类型不能同时实现 `Drop`。

所以这种类型不能是 `Copy`：

```rust
struct User {
    name: String,
}
```

因为 `String` 拥有堆内存，作用域结束时需要执行 `drop` 释放内存。

而这种可以是 `Copy`：

```rust
#[derive(Copy, Clone)]
struct Point {
    x: i32,
    y: i32,
}
```

它里面只有两个整数，直接丢掉即可。

### 能不能主动 `drop(x)`？

可以写：

```rust
let x = 10;
drop(x);
```

但对 `Copy` 类型基本没有意义，因为传给 `drop` 的只是副本，原来的 `x` 仍能继续使用：

```rust
let x = 10;

drop(x);

println!("{x}"); // 仍然可以
```

编译器通常还会提醒：

```text
calls to std::mem::drop with a value that implements Copy does nothing
```

如果只是想让变量名从这里以后不可用，最自然的是控制作用域：

```rust
{
    let x = 10;
    dbg!(x);
} // 从这里开始就没有 x 了
```

或者用遮蔽让旧名字不可访问：

```rust
let x = 10;
dbg!(x);

let x = (); // 新的 x 遮蔽旧的 x
```

不过对于整数、布尔值这类 `Copy` 数据，通常没必要刻意清理。它们不会像 `String`、`Vec`、文件句柄那样占着外部资源。

所以可以记成：

> 非 `Copy` 值需要明确谁负责释放；`Copy` 值只是普通小数据，离开作用域自然结束，不需要操心清理。

## 实例化结构体使用{}花括号？

对，Rust 里普通结构体实例化就是用花括号：

```rust
struct User {
    username: String,
    email: String,
}
```

创建实例：

```rust
let user = User {
    username: String::from("hi"),
    email: String::from("34028312@qq.com"),
};
```

这里：

```rust
User { ... }
```

本身是一个**结构体表达式**，会产生一个 `User` 值。

字段名和变量名相同时，可以简写：

```rust
let username = String::from("hi");
let email = String::from("34028312@qq.com");

let user = User {
    username,
    email,
};
```

也可以用结构体更新语法：

```rust
let user2 = User {
    email: String::from("new@example.com"),
    ..user
};
```

不过并非所有结构体都用 `{}`。元组结构体用圆括号：

```rust
struct Point(i32, i32);

let point = Point(10, 20);
```

单元结构体什么都不用：

```rust
struct Marker;

let marker = Marker;
```

所以：

- 普通结构体：`User { ... }`
- 元组结构体：`Point(...)`
- 单元结构体：`Marker`

Rust 没有统一的 `new` 关键字；`User::new(...)` 通常只是程序员自己定义的关联函数。

## `#[derive(Debug)]` 是什么

`#[derive(Debug)]` 是 Rust 的一个**派生属性**，意思是：

> 让编译器自动为这个类型实现 `Debug` trait。

例如：

```rust
#[derive(Debug)]
struct User {
    username: String,
    email: String,
}
```

这样你就可以调试打印它：

```rust
let user = User {
    username: String::from("hi"),
    email: String::from("34028312@qq.com"),
};

dbg!(&user);
println!("{:?}", user);
```

输出类似：

```text
User { username: "hi", email: "34028312@qq.com" }
```

格式化打印还可以写：

```rust
println!("{:#?}", user);
```

会更好看：

```text
User {
    username: "hi",
    email: "34028312@qq.com",
}
```

### `#[]` 是什么

Rust 中：

```rust
#[...]
```

叫 **attribute，属性**。

它是写给编译器或宏看的附加信息。

而：

```rust
derive(Debug)
```

表示自动派生 `Debug` 的实现。

大致相当于编译器替你写了：

```rust
impl std::fmt::Debug for User {
    // 自动生成具体实现
}
```

你不需要自己手写。

### 为什么不默认所有类型都能 `Debug`

Rust 不会默认假设所有结构体都应该能打印。

没有加：

```rust
struct User {
    username: String,
}
```

直接这样：

```rust
println!("{:?}", user);
```

会报错，因为 `User` 没有实现 `Debug`。

加上：

```rust
#[derive(Debug)]
```

才可以。

### 可以同时派生多个 trait

```rust
#[derive(Debug, Clone, PartialEq)]
struct User {
    username: String,
    email: String,
}
```

分别表示：

- `Debug`：可以调试打印
- `Clone`：可以显式复制
- `PartialEq`：可以使用 `==`、`!=`

例如：

```rust
let user2 = user.clone();

println!("{}", user == user2);
```

### `dbg!` 也依赖 `Debug`

```rust
dbg!(&user);
```

之所以能打印，是因为 `User` 实现了 `Debug`。

因此学习阶段自己定义结构体时，经常可以直接写：

```rust
#[derive(Debug)]
struct User {
    username: String,
    email: String,
}
```

可以把它理解为：

> 请编译器自动给这个结构体生成一套调试打印能力。

### 有点像装饰器

对，**长得很像 Python 装饰器**，都写在定义上方，用来给下面的对象增加额外行为：

```python
@dataclass
class User:
    ...
```

```rust
#[derive(Debug)]
struct User {
    ...
}
```

但底层机制不太一样。

Python 装饰器通常是在运行时把类或函数传给另一个函数：

```python
User = dataclass(User)
```

Rust 的 attribute 是在**编译阶段**交给编译器或宏处理。`derive(Debug)` 会自动生成类似这样的代码：

```rust
impl Debug for User {
    // 编译器生成具体实现
}
```

所以更准确地说：

> Rust 的 `#[...]` 外观像装饰器，但本质更像“编译期注解 + 代码生成”。

Rust 还有很多类似写法：

```rust
#[derive(Debug, Clone)]
struct User {
    name: String,
}
```

```rust
#[test]
fn test_add() {
    assert_eq!(1 + 1, 2);
}
```

```rust
#[allow(dead_code)]
fn unused_function() {}
```

```rust
#[cfg(target_os = "windows")]
fn windows_only() {}
```

其中有些只是告诉编译器信息，有些会真正生成代码。因此不能把所有 `#[...]` 都理解成“给函数包一层”，但把它先类比成装饰器，作为入门理解是很合适的。

## 定义方法时 self 要传借用啊

对，**方法里最常见的是传 `&self`**，因为通常只是“查看这个实例”，不需要把实例用掉。

```rust
#[derive(Debug)]
struct User {
    username: String,
    email: String,
}

impl User {
    fn show(&self) {
        println!("{}", self.username);
    }
}
```

调用：

```rust
let user = User {
    username: String::from("hi"),
    email: String::from("34028312@qq.com"),
};

user.show();
user.show(); // 还能继续调用
```

这里的：

```rust
fn show(&self)
```

其实是简写，近似于：

```rust
fn show(self: &User)
```

也就是把当前 `User` 借给方法。

Rust 方法里的 `self` 主要有三种。

#### `&self`：只读借用

```rust
fn username(&self) -> &str {
    &self.username
}
```

实例不会被用掉，也不能修改字段。

#### `&mut self`：可变借用

```rust
fn change_email(&mut self, email: String) {
    self.email = email;
}
```

调用者也必须是可变变量：

```rust
let mut user = User {
    username: String::from("hi"),
    email: String::from("old@example.com"),
};

user.change_email(String::from("new@example.com"));
```

实例仍然存在，只是暂时借给方法修改。

#### `self`：取得所有权

```rust
fn delete(self) {
    println!("删除用户 {}", self.username);
}
```

调用后实例就不能再用了：

```rust
user.delete();

// println!("{:?}", user); // 报错：user 已被移动
```

这和你刚才对 `dbg!` 的理解一样：

```rust
fn method(self)
```

表示：

> 这个方法把实例整个拿走，调用结束后实例通常也就到此为止。

所以最简单的判断是：

- 只看实例：`&self`
- 修改实例：`&mut self`
- 消费实例：`self`

还有一种没有 `self` 的：

```rust
impl User {
    fn new(username: String, email: String) -> User {
        User {
            username,
            email,
        }
    }
}
```

调用：

```rust
let user = User::new(
    String::from("hi"),
    String::from("34028312@qq.com"),
);
```

它不依赖已有实例，所以叫**关联函数**，而不是实例方法。

因此，定义普通方法时确实可以默认先想 `&self`。只有明确需要修改，才用 `&mut self`；明确要把实例用掉，才用 `self`。

## 关联函数是不是对标类方法的

可以类比成“类上的方法”，但**更准确地说，Rust 的关联函数更接近 Python 的 `@staticmethod`，而不是 `@classmethod`**。

Rust：

```rust
impl User {
    fn new(username: String) -> Self {
        Self { username }
    }
}
```

调用：

```rust
let user = User::new(String::from("hi"));
```

它属于 `User` 的命名空间，但不会自动接收实例：

```rust
fn new(username: String) -> Self
```

没有：

```rust
&self
&mut self
self
```

所以它是**关联函数**，不是方法。

对应 Python，大致更像：

```python
class User:
    @staticmethod
    def new(username: str) -> "User":
        return User(username)
```

而 Python 的类方法是：

```python
class User:
    @classmethod
    def new(cls, username: str) -> "User":
        return cls(username)
```

这里会自动传入 `cls`，因此子类调用时可以创建子类实例。Rust 没有一个完全相同的“运行时类对象 `cls`”。

不过 Rust 关联函数里经常使用 `Self`：

```rust
impl User {
    fn new(username: String) -> Self {
        Self { username }
    }
}
```

这个 `Self` 指的是当前 `impl` 对应的类型，写起来确实很有类方法的感觉，但它不是作为参数传进来的。

可以这样对应：

| Rust                   | Python 类比                         |
| ---------------------- | ----------------------------------- |
| `fn show(&self)`       | 实例方法 `def show(self)`           |
| `fn change(&mut self)` | 修改实例的实例方法                  |
| `fn consume(self)`     | 消费实例的方法，Python 没有直接对应 |
| `fn new(...) -> Self`  | 更接近 `@staticmethod` 工厂函数     |
| `User::new(...)`       | `User.new(...)`                     |

因此你的理解可以稍微调整为：

> 关联函数就是“挂在类型上的函数”；使用方式像类方法，但机制更接近静态方法。

## 消耗和清理的只是实例的东西，抽象的东西一直是全局可用的？

对，你可以先这么理解：

> `self` 被消费时，消耗的是这个具体实例所拥有的数据；`struct`、`impl`、方法和关联函数这些“类型层面的定义”不会被消耗。

比如：

```rust
struct User {
    username: String,
}

impl User {
    fn new(username: String) -> Self {
        Self { username }
    }

    fn delete(self) {
        println!("删除 {}", self.username);
    }
}
```

创建两个实例：

```rust
let user1 = User::new(String::from("Alice"));
let user2 = User::new(String::from("Bob"));
```

执行：

```rust
user1.delete();
```

被消费的是 `user1`：

- `user1.username` 里的 `String` 最终被释放；
- `user1` 这个变量不能再使用；
- `user2` 完全不受影响；
- `User::new` 仍然可以继续调用；
- `User` 这个类型仍然存在。

```rust
let user3 = User::new(String::from("Carol"));
```

所以可以分成两层。

### 实例层

```rust
user1
user2
user3
```

它们是在程序运行时创建的具体值，各自拥有自己的字段数据。

方法接收：

```rust
self
```

意味着拿走某一个具体实例。

```rust
fn delete(self)
```

不是删除 `User` 这个类型，而是消费调用它的那个实例：

```rust
user1.delete();
```

相当于把 `user1` 交给方法。

### 类型层

```rust
struct User { ... }

impl User { ... }

User::new(...)
```

这些不是某个实例拥有的东西。

它们更像程序中已经定义好的规则：

- `User` 的数据长什么样；
- `User` 有哪些方法；
- 如何创建 `User`；
- 方法执行什么代码。

这些定义不会因为某个实例被销毁而消失。

不过“全局可用”需要稍微修正一下。它们不是无条件在所有地方都可见，而是受模块和可见性控制：

```rust
mod users {
    pub struct User {
        pub username: String,
    }

    impl User {
        pub fn new(username: String) -> Self {
            Self { username }
        }
    }
}
```

其他模块要先能访问到 `User`，才能调用：

```rust
let user = users::User::new(String::from("Alice"));
```

所以更准确地说：

> 类型和方法不会被消费，它们在程序代码中长期存在；但是否能使用，要看作用域、模块和 `pub` 可见性。

你可以把它类比成：

- `struct User`：模具；
- `impl User`：模具附带的使用说明；
- `user1`：用模具造出来的一件具体产品；
- `user1.delete()`：销毁这一件产品；
- 模具和说明书仍然还在，可以继续生产 `user2`、`user3`。

## 一个 impl 块能定义多个方法吗？还是得分开定义？

可以，**一个 `impl` 块里可以定义多个方法和关联函数**，而且通常就会写在一起：

```rust
struct Rectangle {
    width: i32,
    height: i32,
}

impl Rectangle {
    fn area(&self) -> i32 {
        self.width * self.height
    }

    fn perimeter(&self) -> i32 {
        2 * (self.width + self.height)
    }

    fn is_square(&self) -> bool {
        self.width == self.height
    }

    fn new(width: i32, height: i32) -> Self {
        Self { width, height }
    }
}
```

调用：

```rust
let rect = Rectangle::new(10, 20);

dbg!(rect.area());
dbg!(rect.perimeter());
dbg!(rect.is_square());
```

也可以拆成多个 `impl`：

```rust
impl Rectangle {
    fn area(&self) -> i32 {
        self.width * self.height
    }
}

impl Rectangle {
    fn perimeter(&self) -> i32 {
        2 * (self.width + self.height)
    }
}
```

这也是合法的。只要不要在多个 `impl` 中重复定义同名方法：

```rust
impl Rectangle {
    fn area(&self) -> i32 {
        self.width * self.height
    }
}

impl Rectangle {
    fn area(&self) -> i32 { // 报错：重复定义
        0
    }
}
```

一般习惯是：

- 相关方法放在同一个 `impl` 块里；
- 文件很大或实现不同功能时，可以拆成多个 `impl`；
- trait 实现必须单独写：

```rust
impl std::fmt::Display for Rectangle {
    // trait 方法
}
```

你在 Evcxr 里学习时，把结构体和所有固有方法放在同一个代码框里最省心，不过这个代码框反复运行仍可能需要先 `:clear`。

## item 是不是可以理解成它不是顺序执行的，它只是先放到内存里，谁需要的时候我再执行，所以不加分号；分号好像在定义一种执行顺序

这个理解**方向很对**，尤其是你说的：

> item 不是按照书写顺序立即执行的；它是在声明“程序里有什么”。

但“先放到内存里”需要改一下。更准确地说：

> **item 主要是交给编译器建立定义，不一定在运行时占用内存。**

例如：

```rust
fn main() {
    greet();

    fn greet() {
        println!("hello");
    }
}
```

即使 `greet` 写在调用之后，也可以调用。因为 `fn greet` 是 item，编译器会先理解整个作用域中有哪些定义，而不是运行到这一行才创建函数。

类型定义也是如此：

```rust
struct User {
    name: String,
}
```

`struct User` 只是描述 `User` 的内存布局。只有创建实例时，才真正产生运行时数据：

```rust
let user = User {
    name: String::from("Ziru"),
};
```

### 分号确实和“执行流程”关系更大

方法体里的这些代码是顺序执行的：

```rust
fn run() {
    let x = 10;
    println!("{x}");
    do_something();
}
```

它们属于语句或表达式，运行时通常按照顺序发生：

```text
创建 x
→ 打印 x
→ 调用 do_something
```

分号在这里主要有两个作用：

1. 标记某些语句的结束；
2. 丢弃表达式产生的值。

例如：

```rust
let x = {
    1 + 2
};
```

代码块返回 `3`。

而：

```rust
let x = {
    1 + 2;
};
```

加了分号，`3` 被丢弃，代码块返回 `()`。

所以分号并不单纯等于“定义执行顺序”，但你的直觉抓到了关键区别：

```text
item：声明程序里有什么
statement / expression：描述程序运行时做什么
```

### 为什么方法之间不加分号

```rust
impl Rectangle {
    fn area(&self) -> i32 {
        self.width * self.height
    }

    fn perimeter(&self) -> i32 {
        2 * (self.width + self.height)
    }
}
```

这里两个方法都是 **associated item（关联项）**。

它们不是说：

```text
先执行 area
再执行 perimeter
```

而是在声明：

```text
Rectangle 拥有 area 方法
Rectangle 还拥有 perimeter 方法
```

至于哪个方法执行、什么时候执行，由调用决定：

```rust
rect.perimeter();
rect.area();
```

这里才形成运行顺序。

而且方法本身已经由 `{ ... }` 明确结束，解析器不需要再靠分号辨认边界。

所以你可以形成一个很不错的整体认识：

> **花括号包住的函数、结构体、`impl` 等 item 是“定义”；分号更多出现在函数体内部，用于组织会实际求值和执行的语句。**

只是不要把“有分号”绝对等同于“会执行”，因为还有只有声明、没有方法体的 trait 签名：

```rust
trait Shape {
    fn area(&self) -> i32;
}
```

这里分号表示“签名到此结束”，它本身也不会执行。
